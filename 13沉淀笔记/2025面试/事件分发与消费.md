
# 📖 Android 事件分发机制与核心源码深度解析 (完整版)

## 一、 核心概念与三大中枢

Android 事件分发本质上是**责任链模式**的变种，围绕三个核心方法展开：

- **`dispatchTouchEvent` (分发枢纽)：** 事件流转的绝对入口。负责路由，决定事件是向下级分发，还是交由自身处理。
    
- **`onInterceptTouchEvent` (拦截决策)：** `ViewGroup` 独有。作为 `dispatchTouchEvent` 的辅助判断条件，决定是否将事件拦截在当前层级（向下阻断）。
    
- **`onTouchEvent` (事件消费)：** 真正处理触摸逻辑的末端。决定是否“消费”该事件（向上阻断）。
    

> **💡 核心专业名词：多态与动态绑定 (Polymorphism & Dynamic Binding)**
> 
> Java 面向对象核心机制。在事件分发中，即使通过 `super.xxx()` 调用了父类的逻辑框架，但框架内部执行时，当前内存实例对象（`this`）并未改变。因此，框架内调用的子方法依然会动态绑定到当前对象重写的实现上。

## 二、 经典 U 型分发模型

事件的传递遵循“由外向内分发（寻找处理者），由内向外回传（报告处理结果）”的 U 型链路。

Plaintext

```
【向下分发阶段：寻找目标】
Activity.dispatchTouchEvent() 
  └──> ViewGroup.dispatchTouchEvent() 
         ├── (内部调用 ViewGroup.onInterceptTouchEvent() -> 若返回 false)
         └──> View.dispatchTouchEvent()
                ├── (内部调用 View.onTouchEvent() -> 若返回 false)
                ▼ 
【向上回传阶段：兜底消费】
         ┌──────┘ 
         └──> ViewGroup.onTouchEvent() -> (若返回 false)
  ┌──────┘
  └──> Activity.onTouchEvent() -> (最终兜底)
```

## 三、 核心源码链路拆解

### 1. View 层的消费逻辑 (优先级的秘密)

在叶子节点 `View` 的 `dispatchTouchEvent` 中，决定了事件到底被谁消费。

**核心知识点：** 外部监听器（`OnTouchListener`）的优先级绝对高于内部方法（`onTouchEvent`）。

Java

```
// View.java - dispatchTouchEvent() 核心精简版
public boolean dispatchTouchEvent(MotionEvent event) {
    boolean result = false;
    // 1. 优先检查外部设置的 OnTouchListener
    if (mOnTouchListener != null && mOnTouchListener.onTouch(this, event)) {
        result = true; // 外部消费了事件！
    }
    // 2. 如果外部没有消费，才轮到自身原本的 onTouchEvent()
    if (!result && onTouchEvent(event)) {
        result = true; // 内部消费了事件！
    }
    return result;
}
```

### 2. ViewGroup 层的分发与拦截逻辑 (源码骨架)

`ViewGroup` 的 `dispatchTouchEvent` 是整个事件分发最复杂的枢纽，包含拦截判断与目标记录。

**核心名词解释：`mFirstTouchTarget`**

它是一个单向链表节点，充当**“消费目标备忘录”**。如果在 `ACTION_DOWN` 时某个子 View 消费了事件，`ViewGroup` 就会把该子 View 包装成 Target 赋值给它。后续的 MOVE/UP 事件将直接通过它精准投递，不再遍历其他子 View。

Java

```
// ViewGroup.java - dispatchTouchEvent() 核心精简版
@Override
public boolean dispatchTouchEvent(MotionEvent ev) {
    final int actionMasked = ev.getActionMasked();
    
    // 【阶段一：重置状态 (DOWN 事件陷阱)】
    if (actionMasked == MotionEvent.ACTION_DOWN) {
        cancelAndClearTouchTargets(ev);
        resetTouchState(); // 无情清除子 View 申请的 FLAG_DISALLOW_INTERCEPT 标志位
    }

    // 【阶段二：拦截决策】
    final boolean intercepted;
    if (actionMasked == MotionEvent.ACTION_DOWN || mFirstTouchTarget != null) {
        final boolean disallowIntercept = (mGroupFlags & FLAG_DISALLOW_INTERCEPT) != 0;
        if (!disallowIntercept) {
            intercepted = onInterceptTouchEvent(ev); // 执行当前 ViewGroup 的拦截逻辑
        } else {
            intercepted = false; // 子控件强行要求不拦截，直接放行
        }
    } else {
        intercepted = true; // 没有目标且不是 DOWN 事件，自己拦截
    }

    // 【阶段三：寻找接盘侠 (Z 轴空间竞争规则)】
    if (!intercepted && actionMasked == MotionEvent.ACTION_DOWN) {
        // 倒序遍历子 View：从 Z 轴最顶层 (最晚添加) 的控件开始问，而不是排队
        for (int i = childrenCount - 1; i >= 0; i--) {
            final View child = children[i];
            
            // 尝试把事件分发给子 View
            if (child.dispatchTouchEvent(ev)) {
                // 【划重点】：只要顶层子 View 返回 true 消费了 DOWN 事件，立刻赋值！
                mFirstTouchTarget = addTouchTarget(child);
                break; // 立刻终止遍历，底层的子 View 没有机会获取事件
            }
        }
    }

    // 【阶段四：兜底与多态派发】
    if (mFirstTouchTarget == null) {
        // 没有任何子 View 消费，调用父类 View.java 的逻辑框架
        handled = super.dispatchTouchEvent(ev);
    } 
    // ... 后续逻辑省略 ...
}
```

### 3. 多态的奥秘：`super.dispatchTouchEvent` 的真相

当 `ViewGroup` 找不到子 View 处理事件时（`mFirstTouchTarget == null`），会执行 `handled = super.dispatchTouchEvent(ev)`。

- **非对象切换：** 这里的 `super` 并不是创建了父类对象，而是借用了父类 `View.java` 中写好的标准分发框架（即“查监听器 -> 调 `onTouchEvent`” 的流程）。
    
- **动态绑定：** 内存中执行这段逻辑的实例对象（`this`）依然是当前的 `ViewGroup`。因此，当父类框架走到 `onTouchEvent(ev)` 时，虚拟机会动态调用当前 `ViewGroup` 重写的 `onTouchEvent`，从而实现**容器自己消费事件**。
    

## 四、 内部拦截法与滑动冲突破局

在处理嵌套滑动冲突（如 ScrollView 嵌套 ViewPager）时，常使用内部拦截法。

**核心武器：** `requestDisallowInterceptTouchEvent(boolean disallowIntercept)`

1. **工作原理：** 子控件调用此方法，向 `ViewGroup` 注入 `FLAG_DISALLOW_INTERCEPT` 标志位。
    
2. **跳过拦截：** 标志位存在时，`ViewGroup` 强行跳过 `onInterceptTouchEvent`，子控件得以独占 `ACTION_MOVE` 和 `ACTION_UP`。
    
3. **DOWN 事件绝对权力：** 由于 `ACTION_DOWN` 必然触发 `resetTouchState()`，子控件**绝对无法**阻止父控件对 `ACTION_DOWN` 的拦截。若父控件在 `DOWN` 时拦截，子控件将彻底失去事件流。
    

## 五、 高频面试极简总结

1. **权责划分：** `dispatchTouchEvent` 管路由与决策，`onInterceptTouchEvent` 管向下劫杀，`onTouchEvent` 管向上兜底。
    
2. **消费优先级：** `OnTouchListener.onTouch` > `View.onTouchEvent`。
    
3. **握手协议与记录：** 消费必须从 `ACTION_DOWN` 开始。`mFirstTouchTarget` 是在子控件确实返回 `true` 后才写入的备忘录。错过 `DOWN`，后续免谈。
    
4. **空间竞争规则：** 多个子 View 竞争事件，遵循从 Z 轴顶层到底层的倒序遍历，先到先得（返回 `true`），拿到即 `break`，不存在排队队列。
    
5. **兜底的本质：** `super.dispatchTouchEvent` 是 OOP 的极致体现。借用父类流程框架，通过多态动态绑定触发自身 `onTouchEvent`。
    
6. **标志位盲区：** `requestDisallowInterceptTouchEvent` 保不住 `ACTION_DOWN` 事件。
