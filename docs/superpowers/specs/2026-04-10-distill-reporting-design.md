# Distill 报告与验收文档设计

## 1. 背景

Distill 的工程开发已经不再是“改完代码就结束”的轻量活动。一次真实任务往往同时涉及：

1. Skill / 脚本 / 文档的联动修改
2. 真实 Obsidian vault 中的状态变化
3. fetch artifact、review bundle、hook gate、queue / bookmark / 本地资料归档等证据
4. 多 Agent 协同、模型参与、人工审计与受控重试

现状的问题不是“没有信息”，而是**信息分散**：

- 过程状态散落在 session `plan.md`、SQL todos、命令输出和对话里
- 最终结论散落在临时总结和零散说明里
- 真实证据存在，但没有一个稳定的交付面把它们组织起来

结果是：任务刚做完时看似清楚，过几天或换一个 Agent 接手后，就需要重新翻历史，审计成本很高。

Distill 需要一套明确的报告范式，把“执行过程”和“最终验收”区分开来。

## 2. 目标

本设计要实现以下目标：

1. 为 Distill 的中大型开发、E2E、迁移和生产放行提供统一报告口径
2. 让最终结论可以被**快速阅读**
3. 让关键判断可以被**证据回溯**
4. 让跨 Agent 交接不再依赖完整对话上下文
5. 明确记录 bug、异常、残余风险和放行结论，而不是只汇报“成功部分”

## 3. 非目标

本设计不追求以下内容：

1. 不用报告替代 SQL todo、git history、artifact bundle 或 review queue
2. 不要求把所有证据正文复制进报告正文
3. 不把报告做成按时间顺序的完整流水账
4. 不要求所有小改动都生成正式 completion report
5. 不在本设计中引入自动生成器；首版先定义人工可执行的标准

## 4. 受众优先级

默认受众排序如下：

1. **生产放行 / 验收**
2. **跨 Agent 交接 / 审计**
3. **开发复盘 / 知识沉淀**

这意味着首版报告必须优先回答下面四个问题：

1. 这次任务到底改了什么？
2. 现在是否可以放行或继续使用？
3. 这个结论基于哪些验证和证据？
4. 还有哪些已知风险、未完成项或外部阻塞？

## 5. 方案比较

### 5.1 方案 A：单一活文档，从开始写到结束

做法：用一份 Markdown 从需求、设计、执行、阻塞一直记到结项。

优点：

- 最简单
- 所有过程都在一个地方

缺点：

- 过程噪音会污染最终结论
- 文档会持续被覆盖，难以形成冻结验收件
- 对审计者不友好，阅读成本高
- 任务越长，越容易变成难以维护的流水账

### 5.2 方案 B：只写最终完成报告，不保留过程文档

做法：开发时只依赖对话、SQL todo、命令输出，任务结束后再补一份总结报告。

优点：

- 最终文档最干净
- 写作成本相对低

缺点：

- 过程决策和异常上下文容易丢失
- 复盘与审计时经常需要重新翻历史
- 对多 Agent 接力不友好

### 5.3 方案 C：双文档体系（推荐）

做法：

1. 用**活文档**持续跟踪任务状态
2. 用**冻结 completion report** 输出最终结论

优点：

- 过程与结论分离，信息密度更高
- 更适合生产放行、审计和交接
- 能显式绑定验证结果与证据路径
- 与现有 `plan.md`、SQL todo、artifact 资产天然兼容

缺点：

- 需要维护两层文档
- 对作者的结构化表达要求更高

**结论：采用方案 C。**

## 6. 推荐设计

### 6.1 文档角色划分

Distill 的报告体系分为两类文档：

1. **实施跟踪单**（活文档）
2. **Distill Completion Report，简称 DCR**（冻结文档）

二者不是互相替代，而是职责分离。

### 6.2 实施跟踪单

实施跟踪单用于记录任务执行态，应满足“可持续更新、可中断恢复、可压缩交接”。

首版规则：

1. 默认承载体为 session workspace 中的 `plan.md`
2. 允许配合 SQL todos 使用，但 SQL 只负责结构化状态，不负责完整叙述
3. 不要求提交到仓库
4. 可以记录：
   - 当前目标
   - 已完成里程碑
   - 阻塞与假设
   - 关键文件
   - 下一步
5. 不要求具备对外可读性，但要求对后续 Agent 可接力

这一步与当前工作流一致，不引入新的 repo 过程文档，避免把临时执行痕迹污染代码仓库。

### 6.3 DCR（冻结 completion report）

DCR 是正式交付件，用于在任务完成、终止或被阻断时冻结状态。

默认落点：

- `docs/superpowers/reports/YYYY-MM-DD-<topic>-completion-report.md`

命名规则：

- `YYYY-MM-DD` 使用冻结日期
- `<topic>` 使用任务主题的英文 kebab-case 简述
- 同一主题若同日多次产出，可追加 `-v2`、`-v3`

DCR 一旦写出，其职责是代表“这次任务的最终对外说法”，除非发现事实错误，否则不应持续滚动改写。

### 6.4 DCR 的适用范围

以下工作应默认产出 DCR：

1. 改动 Distill skill 或运行时契约
2. 执行真实 E2E、真实回放、真实迁移、真实生产 gate
3. 引入或修复会影响生产使用边界的框架级 bug
4. 涉及多 Agent 协同、人工审计或高成本外部依赖

以下工作可不强制产出 DCR：

1. 纯文案修正
2. 很小的无行为变化文档更新
3. 已包含在更大任务 DCR 中的零碎修补

### 6.5 DCR 必备章节

首版 DCR 至少包含以下章节：

1. **任务摘要**
   - 任务名称
   - 日期
   - 主要受众
   - 最终状态：`放行` / `条件放行` / `阻断` / `终止`

2. **背景与目标**
   - 任务为什么开始
   - 成功标准是什么

3. **范围**
   - 本次包含什么
   - 明确不包含什么

4. **实现与变更摘要**
   - 关键改动点
   - 关键文件
   - 行为层面的变化

5. **验证与门禁结果**
   - 运行过哪些验证
   - 哪些通过，哪些失败
   - 哪些 gate 是在真实环境完成的

6. **产物与状态变化**
   - 新增 / 更新的关键产物
   - 真实系统状态的前后变化
   - 回滚、归档、清理是否完成

7. **发现的问题与修复**
   - 本轮暴露的 bug / 设计缺口
   - 修复方式
   - 若未修复，为什么未修复

8. **残余风险与未完成项**
   - 外部阻塞
   - 明确遗留
   - 条件放行的约束

9. **Agent / 模型参与矩阵**
   - 哪些 Agent 参与
   - 各自负责什么
   - 哪些结论来自独立审计

10. **最终结论**
    - 是否建议生产使用
    - 对下一步的明确建议

11. **附录：证据索引**
    - 不是复制证据正文，而是给出可追溯路径

### 6.6 证据索引的最小要求

证据索引至少应包含以下几类路径或引用：

1. 关键命令
2. 关键文件路径
3. artifact / review bundle / telemetry / hook 结果路径
4. git diff / commit 引用（若有）
5. SQL todo 状态或等价任务状态
6. 若触及真实数据，给出 before / after 的状态描述

证据索引的原则是：

- **只引用真实证据**
- **只引用本次结论真正依赖的证据**
- **不给“看起来很多但无助判断”的噪音路径**

### 6.7 最终状态枚举

DCR 的最终状态固定为以下四种之一：

1. **放行**
   - 目标已完成
   - 关键 gate 已通过
   - 无阻断性残余风险

2. **条件放行**
   - 主体目标已完成
   - 允许进入使用或下一阶段
   - 但存在已知限制，必须显式写明条件

3. **阻断**
   - 关键目标未达到
   - 或验证发现当前不应放行

4. **终止**
   - 工作被主动停止、范围改变或被更大任务吸收
   - 不等同于失败，但必须说明终止原因

禁止使用“差不多完成”“基本可用”“大概没问题”这类模糊结论。

### 6.8 Agent / 模型透明性要求

AI 参与不是 DCR 的附属信息，而是结论可信度的一部分。

因此 DCR 需要至少说明：

1. 主执行 Agent / 模型
2. 是否有独立 reviewer Agent
3. 哪些结论是自动判断，哪些是人工决定
4. 是否存在外部工具或站点限制

如果某个关键结论只由单 Agent 单轮判断得出，也必须在风险部分明确标注。

### 6.9 失败、阻断与回滚记录要求

DCR 不能只报告成功路径，还必须报告：

1. 哪些失败在本轮被修复
2. 哪些失败被识别但延期处理
3. 哪些副作用已回滚
4. 哪些真实环境状态已恢复一致

如果任务触碰真实 backlog、真实收藏夹、真实 queue 或真实知识库状态，必须写明：

- 改前状态
- 改后状态
- 是否已完成收尾

### 6.10 建议模板结构

推荐 DCR 使用如下结构：

```markdown
# <任务名> Completion Report

## 1. 任务摘要
## 2. 背景与目标
## 3. 范围
## 4. 实现与变更摘要
## 5. 验证与门禁结果
## 6. 产物与状态变化
## 7. 发现的问题与修复
## 8. 残余风险与未完成项
## 9. Agent / 模型参与矩阵
## 10. 最终结论
## 11. 附录：证据索引
```

模板目标是保证不同作者写出的报告具备稳定骨架，而不是限制表达风格。

## 7. 工作流设计

推荐工作流如下：

1. 任务开始后，持续维护 session `plan.md` 与 SQL todos
2. 中间里程碑写入实施跟踪单，而不是直接写最终报告
3. 接近完成时，按冻结事实起草 DCR
4. 对照证据索引补齐路径与 gate 结果
5. 独立 reviewer 审视 DCR 是否存在：
   - 结论先行但证据缺失
   - 失败被弱化
   - 残余风险未披露
6. 完成后提交 DCR，并把实施跟踪单保留为内部过程资产

## 8. 验收标准

如果一份 DCR 合格，则应满足：

1. 不看完整对话，也能理解任务目标、变化、结论和风险
2. 另一个 Agent 能据此继续下一步，而无需重新爬历史
3. 生产放行者能快速判断是否可放行
4. 审计者能顺着附录路径定位到关键证据
5. 报告不隐瞒失败，也不夸大完成度

## 9. 首版落地策略

首版不做自动生成器，先建立人工写作标准：

1. 实施跟踪单继续使用 session `plan.md`
2. 正式 completion report 落到 `docs/superpowers/reports/`
3. 后续若使用频率稳定，再补：
   - DCR 模板文件
   - 报告生成脚本
   - reviewer 检查清单

这能避免在标准尚未稳定前，过早固化自动化工具。

## 10. 预期结果

本设计落地后，Distill 的任务输出将形成稳定分层：

1. **执行层**：session `plan.md` + SQL todos
2. **证据层**：artifacts / review bundles / telemetry / git / hook 输出
3. **交付层**：DCR

这样可以同时满足：

- 执行时可持续推进
- 完成后可正式验收
- 过后可低成本审计和交接

## 11. 推荐的下一步实现项

本设计通过后，下一步实现应分两段：

1. 创建 DCR 模板与落点目录
2. 基于本轮真实 mixed-source E2E 产出第一份正式 DCR，作为样板
