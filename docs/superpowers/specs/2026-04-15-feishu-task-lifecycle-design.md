# 全局长任务飞书生命周期通知设计

## 1. 背景

当前这套飞书能力已经具备两个基础面：

1. Distill `on_batch_complete` 已能自动发送飞书完成通知，并在发送失败时 fail-closed
2. `MyObsidian/AGENTS.md` 已写明：非 trivial 任务完成后应发送结构化飞书摘要

但这两层还没有真正收敛成**统一的任务生命周期出口**。

现状的真实问题不是“不会发飞书”，而是：

1. **只对 Distill 批次收尾有硬门禁**
2. **其它普通工程任务仍靠 agent 记得手动发**
3. **运行中的长任务缺少稳定的进度通知**
4. **当用户离开电脑时，无法持续感知任务仍在推进、卡住还是已经结束**

最近一次漏发已经证明：只把“发飞书”写进 repo 约定，不足以达到“铁律级”保障。

因此，这次设计的目标不是继续优化单条完成通知文案，而是把飞书通知升级为**任务生命周期能力**。

## 2. 目标

本设计要实现以下目标：

1. 让 **MyObsidian 中的非 trivial 长任务** 在飞书侧具备可感知的生命周期
2. 让用户离开电脑时，仍能知道任务已经开始、推进到哪一阶段、是否在等人、是否完成或失败
3. 复用现有 Distill 飞书脚本，不重复造第二套通知模板
4. 把“完成前必须通知”的硬门禁，从 Distill `on_batch_complete` 扩展到更一般的任务收尾流程
5. 避免把每个微小动作都推到飞书，控制噪音
6. 对通知失败给出明确处理规则，而不是静默丢失

## 3. 非目标

本设计**不追求**：

1. 不把每一句普通对话、寒暄或一次性短回复都发到飞书
2. 不做 token 级、字符级实时流式转发
3. 不声称能修改 Copilot CLI 平台内部、对所有仓库全局生效的原生 `task_complete` runtime
4. 不在本轮同时实现“飞书回复直接驱动当前 live session”的双向控制
5. 不把进度通知做成高频心跳刷屏系统

换句话说，这套设计面向的是**非 trivial 任务的生命周期可见性**，不是“所有消息都镜像到飞书”。

## 4. 成功标准

当用户离开电脑时，至少应能在飞书中看到以下关键状态中的大部分：

1. 任务已开始
2. 当前正在做哪个阶段
3. 是否已进入等待用户输入 / 外部阻塞
4. 最终是完成、失败还是被终止

并且：

1. 关键状态不能静默漏发
2. 完成通知仍需保持结构化摘要能力
3. Distill 批处理和普通工程任务的收尾口径尽量统一

## 5. 方案比较

### 5.1 方案 A：只把“完成通知”做成统一硬门禁

做法：

1. 保持现在的完成通知模板
2. 继续强化“结束前必须发飞书”
3. 不增加开始 / 进度 / 等待中的通知

优点：

- 最简单
- 对现有脚本改动最小
- fail-closed 语义最容易统一

缺点：

- 不能解决“长任务执行中我离开电脑后如何感知进度”的核心诉求
- 用户只能知道“最后结束了没有”，不知道中间是否卡住
- 对等待用户输入的场景不够友好

### 5.2 方案 B：引入任务生命周期通知层（推荐）

做法：

1. 在现有完成通知之上新增统一的 lifecycle wrapper
2. 定义 `started / progress / waiting_user / completed / failed / cancelled`
3. 完成类事件继续走结构化摘要模板
4. 运行中只在**阶段切换**或**节流后的长阶段**发进度，不做高频刷屏

优点：

- 直接覆盖“离开电脑也能知道进度”的核心场景
- 统一 Distill 与普通工程任务的通知语义
- 能清楚地区分“正在跑”“需要你介入”“已经结束”
- 可以保留 critical event 的 fail-closed，同时让 progress 保持可控

缺点：

- 需要引入任务状态与去重逻辑
- 需要对现有任务执行流增加明确的开始 / 进度 / 收尾动作
- 比单纯完成通知更复杂

### 5.3 方案 C：做外部 supervisor，包裹整个 CLI 会话

做法：

1. 用一个外部进程托管 CLI 生命周期
2. supervisor 直接观察任务状态并统一发飞书
3. 尝试把“通知保障”提升到 repo 约定之外

优点：

- 理论上的保障最强
- 能减少“agent 忘记发”的依赖

缺点：

- 实现复杂度显著更高
- 当前环境没有现成的全局 completion hook 可直接复用
- 容易演变成对 CLI runtime 的旁路劫持，不适合当前这轮快速收口

**结论：采用方案 B。**

它能在当前环境内覆盖核心诉求，同时不把问题升级成“重写整个宿主 runtime”。

## 6. 铁律定义

为了避免“写了约定，但运行中还是漏掉”的情况，需要把本次设计中的保障边界明确写死。

### 6.1 保障范围

本轮“100% 触发保障”的准确含义定义为：

> **在 MyObsidian 仓库内，由 agent 发起的非 trivial 任务，只要进入正式执行，就必须先进入飞书任务生命周期流程；关键生命周期事件不得静默跳过。**

这一定义**不包括**：

1. 普通聊天
2. 一句即可结束的轻量回答
3. 仓库外、无本地约定约束的其它会话
4. 宿主平台内部无法被本地脚本拦截的原生系统事件

### 6.2 关键铁律

1. **任务开始铁律**：非 trivial 长任务进入执行前，必须先发 `started`
2. **等待铁律**：任务若进入等待用户输入 / 外部阻塞，必须发 `waiting_user`
3. **完成铁律**：任务在对外宣称完成前，必须发 `completed`
4. **失败铁律**：任务在对外宣称失败或停止前，必须发 `failed` / `cancelled`
5. **去噪铁律**：`progress` 只在阶段变化或达到节流阈值时发送，不得按微步骤刷屏

## 7. 推荐设计

### 7.1 统一角色划分

建议把现有能力收敛为三层：

| 层级 | 职责 | 说明 |
| --- | --- | --- |
| `send-feishu-notify.sh` | 最底层实际发送 | 继续负责真正把文本发到飞书，并补 `发送者：Distill` |
| `send-feishu-task-summary.sh` | 终态摘要渲染 | 继续负责 `completed / failed / waiting_user` 这类结构化摘要 |
| `task-lifecycle` wrapper | 生命周期编排 | 新增统一入口，负责 `start/progress/wait/complete/fail/cancel` 和状态持久化 |

Distill `agent-hook-handler.sh` 不再单独扮演“特殊通知系统”，而是作为其中一个 producer，把批处理完成事件也接到同一条 lifecycle 终态出口。

### 7.2 生命周期事件模型

建议定义以下事件：

| 事件 | 含义 | 默认是否发飞书 |
| --- | --- | --- |
| `started` | 任务正式开始执行 | 是 |
| `progress` | 阶段推进或长阶段心跳 | 是，节流 |
| `waiting_user` | 任务正在等待用户或外部输入 | 是 |
| `completed` | 任务完成 | 是 |
| `failed` | 任务失败 | 是 |
| `cancelled` | 用户中断或系统终止 | 是 |

### 7.3 事件严重级别

并不是所有事件都应该用同一种失败策略。

建议分为两类：

#### A. Critical events

- `started`
- `waiting_user`
- `completed`
- `failed`
- `cancelled`

规则：

1. 发送失败不能静默吞掉
2. `completed / failed / cancelled` 必须 fail-closed
3. `started` 发送失败时，不应进入“无人值守长任务”状态
4. `waiting_user` 发送失败时，不应假装用户已被提醒

#### B. Advisory events

- `progress`

规则：

1. 允许重试
2. 不因为一次进度通知失败而中止正在运行的 build / test / E2E
3. 若重试仍失败，应在最终终态通知中披露“期间有 N 条进度通知未成功送达”

这样可以兼顾“关键节点不丢”与“运行中不过度脆弱”。

### 7.4 任务状态模型

建议由 lifecycle wrapper 在本地维护一份持久化状态，至少包括：

- `task_id`
- `repo_path`
- `title`
- `status`
- `started_at`
- `updated_at`
- `current_phase`
- `last_progress_at`
- `help_needed`
- `artifact_paths`
- `unsent_progress_count`
- `outbound_events`

建议的最小状态值：

- `running`
- `waiting_user`
- `completed`
- `failed`
- `cancelled`

### 7.5 状态落点

不建议把 lifecycle runtime 状态写回仓库正文。

推荐落点：

- `~/.claude/skills/distill/runtime/task-lifecycle/`

原因：

1. 这是 skill 自己拥有的通知能力
2. 运行态文件不应污染 repo
3. 后续 Distill 批处理与普通工程任务都可复用同一状态目录

### 7.6 触发矩阵

#### 7.6.1 任务开始

触发时机：

1. agent 已完成初步理解
2. 明确将进入多步骤执行、长命令执行、E2E、批处理、代码改动或多阶段验证

动作：

1. 创建 `task_id`
2. 写本地状态
3. 发送 `started`

飞书文本建议包含：

- 任务标题
- 当前阶段
- 预计接下来要做什么
- 当前是否已进入无人值守执行

#### 7.6.2 阶段进度

触发时机：

1. 从“分析”进入“实现”
2. 从“实现”进入“测试”
3. 从“测试”进入“修复”
4. 开始一个明显会持续较久的 shell 命令
5. 同一阶段持续超过节流阈值（建议 10 分钟）时可发心跳

动作：

1. 更新 `current_phase`
2. 写状态
3. 若符合节流条件则发送 `progress`

飞书文本建议包含：

- 当前阶段
- 最近已完成的 1-3 项
- 正在等待的长命令或验证动作

#### 7.6.3 等待用户

触发时机：

1. 需要用户做设计决策
2. 需要用户提供外部信息、凭证、路径、确认
3. 需要用户 review spec / 报告 / 产物后才能继续

动作：

1. 更新状态为 `waiting_user`
2. 使用结构化摘要模板发送 `waiting_user`
3. 明确写出“需要你协助”

#### 7.6.4 任务完成

触发时机：

1. 在对外宣称任务完成之前
2. 在调用 `task_complete` 之前
3. Distill 批处理则在 `on_batch_complete` 收尾中完成

动作：

1. 发送结构化 `completed`
2. 成功后才允许进入最终收尾

#### 7.6.5 任务失败 / 终止

触发时机：

1. 无法继续推进
2. 外部依赖失败且当前无法自愈
3. 用户要求停止

动作：

1. 发送 `failed` 或 `cancelled`
2. 明确失败点、已完成部分、需要的外部协助

## 8. 消息模板建议

### 8.1 `started` / `progress`

这两类消息不需要和完成摘要一样长，但也不能只发一句“还在跑”。

建议最小结构：

1. 任务标题
2. 当前状态
3. 当前阶段
4. 最近已完成
5. 下一步 / 当前长命令

### 8.2 `waiting_user`

建议直接复用 `send-feishu-task-summary.sh`，因为它天然适合表达：

1. 当前结论
2. 已完成
3. 关键路径 / 产物
4. 需要你协助

### 8.3 `completed` / `failed`

继续使用现有结构化摘要模板，保持这些字段：

1. `任务标题`
2. `最终状态`
3. `结论`
4. `已完成`
5. `关键路径/产物`
6. `需要你协助`

## 9. 去重与恢复

### 9.1 去重

为了避免重复发送，建议每条 outbound event 至少记录：

- `task_id`
- `event_kind`
- `phase`
- `attempt`
- `sent_at`

同一 `task_id + event_kind + phase` 若已成功发送，不应重复发送同内容。

### 9.2 崩溃恢复

如果 agent 或 shell 在 `running` 状态下异常退出，下一次进入任务生命周期入口时应检查是否存在“未终结旧任务”。

恢复策略建议：

1. 发现旧任务仍处于 `running`
2. 发送一条恢复/异常说明，告知“上一次任务未正常收尾”
3. 将其标记为 `cancelled` 或 `failed_recovered`
4. 再开启新任务

这样即使用户离开电脑，也不会只看到“开始了”却永远没有后文。

## 10. 与现有 Distill 能力的关系

### 10.1 保留的部分

保留：

1. `send-feishu-notify.sh` 作为唯一真实发送器
2. `send-feishu-task-summary.sh` 作为统一摘要模板
3. `agent-hook-handler.sh` 中 Distill `on_batch_complete` 的 fail-closed 语义

### 10.2 需要收敛的部分

后续实现时应收敛为：

1. Distill 批处理完成通知走统一 lifecycle 终态出口
2. 普通工程任务的完成通知也走同一出口
3. AGENTS 不再只是写“记得发飞书”，而是明确写“非 trivial 任务必须先进入 lifecycle wrapper”

## 11. 测试与验收建议

首轮实现至少应覆盖：

1. `started` 真发成功
2. `progress` 真发成功
3. `waiting_user` 真发成功
4. `completed` 真发成功
5. `failed` 真发成功
6. `completed` 发送失败时 fail-closed
7. `failed` 发送失败时 fail-closed
8. `progress` 发送失败时不误报完成，但会被记录
9. 重复同一事件时不会双发
10. 崩溃恢复能识别未收尾任务

## 12. 分阶段落地建议

### Phase 1

1. 先定义 lifecycle wrapper
2. 只支持 `started / waiting_user / completed / failed`
3. Distill 和普通任务先共用同一终态出口

### Phase 2

1. 增加 `progress`
2. 加入节流和去重
3. 对长命令开始前补发阶段进度

### Phase 3

1. 增加崩溃恢复
2. 增加超长阶段心跳
3. 进一步收紧 AGENTS 与收尾流程

## 13. 最终结论

要满足“我离开电脑时仍能感知任务进度”的诉求，单纯强化完成通知是不够的。

推荐方案是：

1. **把飞书能力提升为任务生命周期能力**
2. **对关键节点实行硬门禁**
3. **对运行中进度实行节流后的阶段通知**
4. **继续复用 Distill 已有的发送器与摘要模板**

这能在不改造宿主平台 runtime 的前提下，把当前“只在 Distill 批处理里 100%”推进到“在 MyObsidian 非 trivial 任务里形成统一、可执行、可审计的生命周期通知约束”。
