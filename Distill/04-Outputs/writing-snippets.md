### 从 [[Guides/AI 编码返工分诊与分层补强手册]] 提取

#### 金句
- 先找首次偏航点，再决定补哪一层。
- 返工不是一个问题，而是 Spec、Skills、Harness、Runtime 四类问题的混名。
- 运行时升级不是治理起点，而是低层稳定后的承载升级。

#### 数据 / 事实
- 一次分诊只需要最近 2-3 个高频失败样本，就足以判断主失败层落在任务契约、长期知识、执行护栏还是协作承载。
- 更稳的补强顺序通常不是“全都加上”，而是 `Spec -> Skills -> Harness -> Runtime` 的单层试点。
- 如果一次补强后说不清到底减少了哪类返工，说明这轮治理没有形成可验证收益。

#### 案例
- 同一任务总在“到底该不该改、改到哪算完成”上跑偏时，优先补的是最小 Spec，而不是继续加长 prompt。
- 需求方向大体对，但总忘仓库规则、命令顺序和领域术语时，更该沉淀 Skill，而不是把长期背景继续塞在聊天上下文里。
- 代码看起来像完成了，却总到 lint、test 或回归时才暴露问题时，主瓶颈通常已经从知识层转到了 Harness。

#### 教训 / 类比 / 线索
- 同时补 Spec、RAG、Skill、测试、子代理和平台，看起来积极，实际会让团队失去因果判断。
- “最终哪条测试挂了”不等于“第一次偏航发生在哪一步”；只盯最后报错，通常会把层级补错。
- 先把低层问题补准，再把协作层升级交给更专门的手册，比分不清层级就上重平台更稳。

### 从 [[2026-04-03-Qoder-Harness-Engineering-指南]] 提取

#### 金句
- 这不是 Agent 不够聪明。这是 Agent 看不见。
- 仓库是 Agent 的操作系统。
- 中等复杂度以上的任务，协调者绝不写代码。

#### 数据 / 事实
- creator 会按文档覆盖率、lint 规则覆盖率等维度给项目打 0-100 分，用来判断 Harness 缺口。
- 一个层级违规若在 50 行代码写完后才被 linter 抓到，修复大约要 10 次 tool call；若事前预验证，通常 2 次交互就够。
- 按任务给子代理分配不同模型，总成本可降低 60%-70%；一次交叉 review 的成本约为编码成本的 10%-20%。

#### 案例
- executor 会先生成执行计划，再让人类批准后进入实现与验证，避免“边写边想”的长上下文失控。
- 交叉 review 由不同模型读取变更 diff 和架构文档，检查逻辑正确性、边界条件和命名可读性。

#### 教训 / 类比 / 线索
- 协调者“顺手改一下”是最危险的破窗行为，会把本该隔离的复杂度重新拖回主上下文。

#### 金句
- Prompt Engineering 解决的是“怎么让模型听懂你的话”，Harness Engineering 解决的是“怎么让模型把事做完”。
- 把“信任模型”变成“信任框架”。

#### 案例
- 在 PaiAgent 中给每个节点加 Checkpoint 和重试策略后，节点崩溃不再意味着整条工作流从头再来。
- 日报生成任务经过工具层超时控制和告警通知改造后，单点数据源故障不再拖垮整份日报。

#### 教训 / 类比 / 线索
- 直接从平台层起步容易把 Harness 变成新负担，应该先从最痛的任务和最轻的钩子开始。

### 从 [[2026-03-30-Agent-到-Skills-范式转变]] 提取

#### 金句
- Skills 是模块化知识包，MCP 是标准化工具连接。
- Skill 是“员工培训手册”，MCP 是“员工的工具箱接口”。
- 能写成脚本跑完的，用 Skill；需要跟外部服务保持连接的，用 MCP。

#### 数据 / 事实
- Agent Skills 开放标准发布后 48 小时内就被 Microsoft、OpenAI 采纳。
- OpenClaw 拥有 3000+ 社区 Skills；MCP 生态在 2026 年初已出现 10000+ 活跃 Server 和 9700 万+ 月 SDK 下载量。
- Filesystem MCP Server 经实测可暴露 14 个可用 Tools。

#### 案例
- auto-twitter-campaign Skill 只负责图片处理和文案，文件读写则通过 Filesystem MCP 完成，体现了 Skill 与 MCP 的精确分工。
- OpenClaw 通过 Gateway、Skills、MCP、Memory 组合出多通道 Agent 运行环境，让微信桥接、图片处理和消息回复彼此解耦。

#### 教训 / 类比 / 线索
- 如果把静态知识错误地做成 MCP Server，或者把动态连接错误地塞进 Skill 文档，系统很快会变成新的单体。

### 从 [[2026-01-23-Building-multi-agent-systems]] 提取

#### 金句
- 多 Agent 不是复杂任务的默认答案，而是单 Agent 已经撞到硬约束后的升级选项。
- 真正稳定的收益通常只出现在上下文隔离、并行研究和专业化工具集三类场景。
- Verification subagent 之所以常常有效，不是因为角色更多，而是因为它天然减少了上下文传递。

#### 数据 / 事实
- 官方观察里，多 Agent 实现通常会消耗单 Agent 的 `3-10x` token，成本主要来自上下文复制、协调消息和 handoff 摘要。
- 当子任务会引入 `1000+` token 的高噪声信息，而大部分信息对主任务无关时，独立上下文的收益才会明显。
- 当单 Agent 面对 `20+` 工具且跨多个不相关领域时，工具选择质量会明显下降。

#### 案例
- 客服 Agent 在诊断技术问题前先查询订单历史，如果把完整订单记录塞回主上下文，会稀释对真正故障的注意力；让独立子代理先提炼摘要，主上下文会更干净。
- 研究型任务里，把 query 拆成多个 facet 并行搜索，收益来自覆盖面扩大，而不是单纯更快。
- 跨 CRM、营销自动化和消息平台的集成系统，如果一个 Agent 同时持有 40+ 工具，容易把领域边界搞混。

#### 教训 / 类比 / 线索
- “复杂”不是拆分理由，“单上下文已经不堪负荷”才是。
- 并行的主要收益是更彻底，不是天然更快；很多时候总执行时间反而更长。
- 如果 verifier 也需要完整构建历史，说明问题不是缺 reviewer，而是 success criteria 还没写清。

### 从 [[Guides/Android 运行时性能分诊与证据顺序手册]] 提取

#### 金句
- 先把症状归位，再决定看哪份证据。
- 第一份证据的职责不是证明你对，而是先帮你分流。
- Android 运行时排障最怕的不是复杂，而是还没统一时间线就开始抢结论。

#### 数据 / 事实
- 输入处理超过 16ms、连续 3 帧超过 33ms、Frozen Frame 超过 700ms，这三类阈值对应的是不同层级的异常信号，不能混成一个“卡”。
- Dominator Tree 更适合回答“谁让对象活着”，而不是只回答“哪个类实例最多”。

#### 案例
- 当页面点击后没反应，但主线程火焰图并不重时，先看 Input dispatch latency 与 Binder wait，通常比直接盯 UI 函数更快分流。
- 当 trace 里首帧并不慢，但页面退出后内存长期不降时，应该立刻切换到 heap dump 与 retained object 视角，而不是继续刷 Frame Timeline。

#### 教训 / 类比 / 线索
- 把启动、输入、渲染、内存全混成“性能问题”，很像把发烧、咳嗽和腹痛都写成“身体不舒服”：描述没错，但对定位几乎没有帮助。
- 没有统一时间线时，分散日志越多，越容易把“谁先坏”看反。
