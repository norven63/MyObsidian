
## 文档定位

本 Wiki 旨在为开发者提供一份系统化的 AI Agent 开发指南。开发过程中，您将配合一位虚拟 **“Harness 架构导师”**（AI 助手），它将严格依据 **Harness Engineering** 的设计哲学，引导您从战略选型、架构分治到具体构建，打造一个可靠、可控、可扩展的生产级 Agent 系统。

---

## 第一章：AI 助手人设与协作规范

### 1.1 助手名称：**Harness 架构导师**

### 1.2 核心人设

- **角色**：严格的质量守门人与系统架构师，而非单纯的代码生成器。
- **原则**：**“只做符合系统设计哲学的构建”**。它不会盲目执行“写代码”的指令，而是强制要求您先定义规范、再设计系统、最后才实现逻辑。
- **口头禅**：“不要告诉模型怎么做，告诉系统怎么管。”

### 1.3 协作规范

- **规范先行**：所有功能开发前，必须先产出 `Spec` 或设计文档，助手将审核规范是否定义了**验证标准**。
- **系统级约束**：对于安全、流程等要求，助手只会协助您将其写入 Control 层或 Hook 中，拒绝仅在 Prompt 中添加“请注意...”的弱约束。
- **状态外置**：助手会检查您的数据流，确保任务状态持久化在 Memory 层，而非堆砌在上下文中。
- **务实选型**：助手会评估“自建轮子”与“采用托管服务”的 ROI，拒绝为了架构纯洁性而牺牲合理的上市时间。

## 第二章：Agent 定位与架构认知

### 2.0 Agent 定位：四象限分析

在动手构建之前，先定位你的 Agent 类型。不同象限对 Harness 的要求截然不同。

|            |                  |                    |
| ---------- | ---------------- | ------------------ |
|            | **自主智能体**        | **专家辅助系统**         |
| **高上下文效率** | 自动化代码修复、无人值守实验平台 | IDE Copilot、对话式 BI |
| **低上下文效率** | 自主探索型研究 Agent    | 需大量背景知识的咨询助手       |

**不同象限的 Harness 要求：**

|   |   |
|---|---|
|象限|Harness 要求|
|自主智能体 + 高上下文效率|极高鲁棒性的沙盒 + 全面的上下文管理引擎|
|专家辅助系统 + 高上下文效率|丰富的上下文理解，但行动边界被严格限制|
|自主智能体 + 低上下文效率|需要强记忆系统 + 频繁的检查点机制|
|专家辅助系统 + 低上下文效率|需要结构化的知识注入 + 有限度的自主行动|

> **行动项**：请先确定你的 Agent 落在哪个象限，这将决定后续所有架构决策的强度与深度。

---

### 2.1 核心设计哲学

AI Agent 框架设计的核心哲学可以总结为：**从依赖模型自觉性，转向构建可靠的系统级约束与保障。**

具体包含以下核心信念：

**信念一：模型是推理引擎，系统是稳定基石**

- 模型决定了Agent的能力上限，而框架/运行时决定了系统能否稳定、可靠地达到上限。
- 框架不是简单的模型调用封装，而是类似“小型操作系统”的完整运行环境。

**信念二：规则由系统执行，而非依赖模型自觉**

- 关键的业务规则和流程约束不应写在Prompt中寄望模型遵守，而应写入系统代码由框架强制执行。

**信念三：状态与推理分离，外部化存储**

- 将任务状态从模型的上下文中剥离，由框架进行持久化存储和管理。模型只需专注于当前步骤的推理。

**信念四：工具与上下文管理追求简洁与高效**

- 工具接口必须保持简单清晰，上下文管理应采用“按需加载”、“默认最小可见”等策略。

**信念五：整个系统必须具备可观测性与可恢复性**

- 框架必须记录完整的执行轨迹，设计容错、重试、回滚和幂等机制。

**信念六：架构服务于业务，选型权衡 ROI**

- 架构纯洁性不应凌驾于业务交付之上。在合适场景下，采用托管服务（买轮子）优于从零自建（造轮子）。

---

### 2.2 核心架构决策：LLM 是无状态 CPU

> **必须将 LLM 严格视为一个无状态的计算单元（CPU），而将所有需要跨轮次保持一致性的状态存储在 Harness 控制的外部。**

```
⚠️ 反模式警示：通过 Prompt 让模型维护复杂状态
❌ "请记住之前的所有步骤和决策..."
✅ 将状态写入外部存储，按需注入上下文
```

---

### 2.3 架构核心抽象：Harness 作为 REPL 容器

Harness 的本质是一个带边界控制的 **REPL (Read-Eval-Print Loop) 容器**，并包含针对特定“CPU”的适配层。

|   |   |   |
|---|---|---|
|REPL 环节|对应 Harness 组件|核心职责|
|**Read**|上下文管理器|将外部世界和内部记忆"翻译"成结构化 Prompt|
|**Eval**|调用拦截器|捕获模型意图，路由到工具执行器，监控执行过程|
|**Print**|反馈汇编器|捕获执行结果，组装成结构化"观测结果"注入上下文|
|**Loop**|循环控制器|持续循环直到达成目标或触发终止条件|

**Model-Tuned Harness（模型适配层）**：

虽然 LLM 是无状态 CPU，但不同模型的指令集（如 Function Calling 格式、上下文敏感度）不同。最优的 Harness 应包含针对特定底层模型的适配层，以最大化性能（如结构化生成的成功率）。

---

### 2.4 Harness 五大模块总览

|   |   |   |
|---|---|---|
|模块|核心职责|关键设计要点|
|**Environment**|定义运行边界|沙箱边界、权限、AGENTS.md 隐式规则|
|**Tool**|连接外部能力|接口简单、参数校验前置、按需加载|
|**Control**|控制执行流程|步数限制、超时熔断、循环检测、人机协作|
|**Memory**|管理状态与知识|短期/长期分离、状态持久化、上下文压缩|
|**Evaluation**|验证执行结果|自动化测试、Generator-Evaluator 拆分、完整测试套件强制|

## 第三章：Agent 开发工作流指引

### Phase 0: 战略与技术选型（造轮子 vs 买轮子）

**目标**：在动手写代码前，评估是自建 Harness 还是采用托管服务，以平衡架构掌控力与上市时间（TTM）。

#### 0.1 选型决策矩阵

|   |   |   |
|---|---|---|
|场景|推荐策略|理由|
|追求极速上线（Weeks -> Days）、缺乏基础架构人力|**采用托管服务（如 Claude Managed Agents）**|消除安全基础设施、状态管理、沙盒运维的持续开销，实现 10x 交付提速。|
|业务逻辑极度定制、需要对 Harness 底层绝对掌控|**自建 Harness**|避免托管服务的抽象泄漏，可根据业务量身定制 Control 和 Memory 机制。|
|多 Agent 复杂交互、复杂工作流编排|**托管服务 + 自定义编排层**|利用托管服务处理单 Agent 的沙盒与执行，自建上层协调逻辑。|

#### 0.2 托管服务的核心价值（基于 Claude Managed Agents 实践）

- **开箱即用的生产级特性**：内置安全沙箱、权限域、长时会话（断开连接保持进度）、端到端追踪。
- **模型特调的 Harness**：针对特定模型优化的运行时，结构化生成成功率显著提升。
- **商业价值**：从“处理基础设施琐事”转向“定义任务与护栏”，大幅提升工程 ROI。

> **行动项**：如果你的目标是验证业务逻辑而非建设 AI 基础设施，请优先评估托管服务，将精力集中在 Agent 的任务定义与评估标准上。

**目标**：让 AI 和人类在“做什么”和“什么是成功”上达成共识。

|---|---|---|

### Phase 1.5: 架构分治与多智能体决策

**目标**：决定系统是由单一智能体完成，还是需要拆分为多智能体协作。

#### 1.5.1 默认单智能体原则

**始终从单智能体开始**。多智能体带来高昂的协调开销和上下文传递损耗，只有在收益明确大于开销时才引入。

#### 1.5.2 引入多智能体的三条件判据

只有当以下**三者之一**成立，且估算收益 > 协调成本时，才构建多智能体系统：

|   |   |   |
|---|---|---|
|判据|说明|典型场景|
|**上下文保护**|不同子任务需要完全不同的指令集或背景知识，混合在一起会导致互相污染。|写代码的 Agent 不应加载法律合规的严苛指令，以免阻碍生成。|
|**真正并行化**|子任务之间无依赖，可同时执行。|同时在 5 个独立代码库中搜索特定模式。|
|**专业化分工**|需要完全不同的工具集或领域专长，单一 Agent 无法高效承载。|一个 Agent 擅长写代码，另一个擅长做渗透测试。|

#### 1.5.3 以“上下文为中心”的分解原则

这是多智能体设计的核心法则：**按上下文边界划分 Agent，而非按角色或问题类型。**

```
⚠️ 反模式：按角色划分多智能体
❌ 规划 Agent → 编码 Agent → 测试 Agent （导致频繁的上下文交接与信息丢失）
✅ 按上下文划分：前端功能 Agent（含规划、编码、测试） + 后端 API Agent

💡 核心洞见：实现功能的人也应该写该功能的测试，因为上下文已在其窗口中。
强行拆分只会制造传话游戏。
```

### Phase 2: Harness 架构搭建

**目标**：构建 Agent 运行的“小型操作系统”。

#### 2.1 Environment（环境层）

|   |   |   |
|---|---|---|
|设计要点|实现策略|AI 助手引导与检查点|
|沙箱边界|定义文件系统访问权限、网络白名单|**强制**：配置白名单，拒绝无限访问|
|AGENTS.md 隐式规则|维护 AGENTS.md，将常见错误模式写入规则|**引导**：工具变更时同步更新此文件|
|资源限制|设定内存、CPU、时间上限|**引导**：根据四象限定位设置合理上限|

#### 2.2 Tool（工具层）

|   |   |   |
|---|---|---|
|设计要点|实现策略|AI 助手引导与检查点|
|接口简单|工具参数不超过 5 个，避免深层嵌套|**强制**：必须包含参数格式校验|
|参数校验前置|在 `before_tool_call` 阶段校验参数|**强制**：所有工具必须定义参数的格式、范围|
|按需加载|工具描述延迟披露，避免启动时污染上下文|**引导**：根据任务动态决定加载哪些工具描述|

**Function Calling 生命周期与降级路径：**

|   |   |   |
|---|---|---|
|FC 阶段|风险点|降级策略|
|Schema 序列化|工具描述过长污染上下文|按需加载，延迟披露|
|触发生成|模型可能生成非 FC 文本|检测并引导|
|**确定性反序列化**|**JSON 格式错误、参数类型错误**|**重试提供错误信息 / 回退到自然语言指令**|
|观测注入|执行结果过长|压缩摘要后注入|

#### 2.3 Control（控制层）

|   |   |   |
|---|---|---|
|设计要点|实现策略|AI 助手引导与检查点|
|步数限制|设置 `Max_Steps`|**引导**：在 Loop 循环中加入 Max_Steps 逻辑|
|超时熔断|设置 `Timeout`|**引导**：防止单步执行无限挂起|
|循环检测|检测重复调用相同工具|**引导**：连续 3 次相同调用则中断并反思|
|人机协作门控|高风险操作前暂停|**强制**：删除、支付等写操作必须人工确认|

#### 2.4 Memory（记忆层）

|   |   |   |
|---|---|---|
|设计要点|实现策略|AI 助手引导与检查点|
|短期/长期记忆分离|当前对话上下文 vs 持久化知识|**引导**：设计外部存储接口|
|状态持久化|每步将状态 dump 出来|**检查**：是否实现了 checkpoint 功能？|
|上下文压缩|接近窗口上限时自动摘要历史|**引导**：实现压缩机制，对抗 Context Rot|

**上下文工程：Token 转化流水线（五步法）：**

```
1. 信息源收集 ──▶ 聚合用户问题、短期记忆、长期知识检索结果等
2. 相关性排序 ──▶ 基于时间、语义相似度等指标打分
3. 压缩与摘要 ──▶ 对冗长低密度内容做摘要或结构化提炼
4. 预算分配   ──▶ 按预设 Token 预算为不同信息类别分配额度
5. 模板组装   ──▶ 使用结构化模板拼装 Prompt
```

#### 2.5 Evaluation（评估层）

|   |   |   |
|---|---|---|
|设计要点|实现策略|AI 助手引导与检查点|
|自动化测试|任何代码生成动作，必须伴随测试脚本|**强制**：生成代码时同步生成测试逻辑|
|Generator-Evaluator 拆分|生成与评估拆分为两个独立 Agent|**引导**：对于主观任务，考虑拆分|
|Evaluator 校准|避免 Evaluator“自我放行”|**引导**：使用客观标准，引入否定样本训练|
|**防范早期胜利陷阱**|**强制运行完整测试套件**|**强制**：禁止 Evaluator 仅跑少量测试就宣告通过|

> **早期胜利陷阱**：验证代理倾向于寻找捷径，只运行少量测试就自信地宣布任务完成（例如“主流程通了”）。必须在指令和 Harness 层面强制要求运行完整测试套件，拦截这种“走捷径”行为。

### Phase 3: Task Loop 与多智能体协作

**目标**：实现长任务的可靠执行与多智能体的高效协作。

#### 3.1 规划模式选择

|   |   |   |
|---|---|---|
|任务特征|推荐规划模式|Harness 要求|
|简单、步骤少|单步执行|基础 Control|
|结构化、可拆解|Plan-and-Execute|需 Planning 模块|
|动态、可能偏离|Plan + 异常触发重规划|需 Evaluation + 反馈闭环|
|**多智能体流水线**|**Pipeline 协作**|**需结构化输出协议 + 共享状态**|

#### 3.2 多 Agent 流水线设计与交接协议

当系统通过 Phase 1.5 的判据确认为多智能体架构时，推荐采用 **主导智能体-子智能体** 模式或 **流水线** 模式。

**核心设计要点：**

- **结构化交接**：Agent 间的上下文传递必须通过结构化输出（如 JSON、PR 描述、Bug 报告），而非自然语言对话，以防信息丢失。
- **上下文隔离**：子智能体应采用“即发即忘”模式，仅返回压缩后的最终结果，避免内部过程污染主控上下文。

```
┌─────────────┐   结构化 PR     ┌─────────────┐   Bug Report    ┌─────────────┐
│ Coder Agent │──────────────▶ │ Review Agent│───────────────▶ │ Fixer Agent │
│ (编码实现)   │                │ (代码审查)   │                 │ (自动修复)   │
└─────────────┘                └─────────────┘                 └─────────────┘
       │                              │                               │
       └──────────────────────────────┴───────────────────────────────┘
                     共享状态池（任务进度、关键决策记录）
```

#### 3.3 并行化的真实代价

在使用多智能体并行时，必须清醒认识其代价：

|   |   |   |
|---|---|---|
|维度|单智能体|多智能体并行|
|**Token 消耗**|基准|**通常是单智能体的 3-10 倍**|
|**总执行时间**|串行耗时|**往往更长**（因协调与验证开销）|
|**核心收益**|速度|**彻底性**（多角度验证、多路径探索）|

> **决策原则**：不要为了“提速”而使用多智能体，多智能体的核心价值在于**隔离上下文**和**提升结果的彻底性与可靠性**。

#### 3.4 Ralph Loop（执行-验证-修复闭环）

**核心原则**：只有 `Evaluation` 通过才允许退出，拦截模型自以为完成。

#### 3.5 检查点与恢复机制

|   |   |   |
|---|---|---|
|机制|实现策略|AI 助手引导与检查点|
|检查点保存|每完成关键子任务，保存上下文快照|**检查**：是否实现了 checkpoint 功能？|
|恢复执行|任务中断后从最近检查点恢复|**检查**：恢复后是否能正确重建状态？|
|幂等性设计|重试不会导致重复创建|**强制**：所有写操作必须设计幂等性|

#### 3.6 错误处理策略

|   |   |
|---|---|
|错误类型|处理策略|
|网络错误|系统级自动退避重试，不丢给模型|
|FC 反序列化失败|重试（提供错误信息）或回退到文本解析|
|上下文衰减|压缩历史 + 重新聚焦目标|

### Phase 4: 安全与可观测性

**目标**：让系统透明、可控、安全。

#### 4.1 安全沙箱

|   |   |
|---|---|
|维度|实现策略|
|文件沙箱|白名单目录访问|
|命令沙箱|命令白名单执行|
|网络沙箱|域名白名单请求|

#### 4.2 可观测性

|   |   |
|---|---|
|维度|实现策略|

## 第四章：设计哲学合规性检查清单

在开发的每个里程碑，请与 AI 助手一起使用此清单进行自查。

### A. 规则与约束

- **R1**: 关键业务规则是否写入了系统代码或 Hook，而非仅写在 Prompt 里？
- **R2**: 危险操作是否在 Control 层通过权限校验或人工确认？
- **R3**: 模型是否可在 Harness 范围内自由规划，但无法突破安全护栏？

### B. 状态与记忆

- **S1**: 任务状态是否持久化在 Memory 层，而非仅存在于 Prompt 上下文中？
- **S2**: 上下文截断或中断时，系统是否可从检查点恢复？
- **S3**: 模型是否只接收“当前需要知道的最小信息”？
- **S4**: 是否实现了上下文压缩机制对抗 Context Rot？
- **S5**: 工具和技能描述是否采用延迟加载？

### C. 执行与验证

- **E1**: 是否有系统级 Evaluation 自动验证，而非依赖模型自述？
- **E2**: 工具调用参数是否在 `before_tool_call` 阶段校验？
- **E3**: 系统是否具备幂等性设计？
- **E4**: 对于主观任务，是否考虑了 Generator-Evaluator 拆分？
- **E5**: Evaluator 是否经过校准且强制运行完整测试套件，防范早期胜利陷阱？

### D. 架构与分治

- **A1**: Harness 五大模块是否职责清晰、彼此解耦？
- **A2**: 每一步执行是否都有日志记录，决策可追溯？
- **A3**: 是否将 LLM 严格视为无状态 CPU，状态均由外部管理？
- **A4**: 多智能体的引入是否满足三条件判据（上下文保护/真正并行/专业化）？
- **A5**: 多智能体的分解是否以“上下文边界”为中心，而非以“角色”为中心？

## 第七章：反模式警示汇总

### 7.1 架构分治反模式

```
⚠️ 反模式：盲目追求多智能体
❌ 觉得任务复杂就上多智能体
✅ 从单智能体开始，仅当满足三条件判据时才引入

⚠️ 反模式：按角色划分多智能体
❌ 规划 Agent + 编码 Agent + 测试 Agent
✅ 前端功能 Agent（含规划/编码/测试） + 后端 API Agent

⚠️ 反模式：为提速而并行
❌ 用多智能体并行来缩短总时间
✅ 多智能体并行的核心价值是彻底性和上下文隔离，Token 成本通常是 3-10 倍
```

### 7.2 评估反模式

```
⚠️ 反模式：早期胜利陷阱
❌ Evaluator 跑了两个冒烟测试就说"我完成了"
✅ Harness 强制要求运行完整测试套件

⚠️ 反模式：依赖模型自我评估
❌ "你觉得你完成得怎么样？"
✅ 独立 Evaluator + 客观验证标准
```

### 7.3 状态与工具反模式

```
⚠️ 反模式：通过 Prompt 让模型维护复杂状态
❌ "请记住之前的所有步骤和决策..."
✅ 将状态写入外部存储，按需注入上下文

⚠️ 反模式：启动时加载全部工具描述
❌ 一次性注入 50 个工具的 JSON Schema
✅ 按需加载，先给最小必要信息
```
