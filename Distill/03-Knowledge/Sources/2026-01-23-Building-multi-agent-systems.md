# Building multi-agent systems: When and how to use them

## 1. 为什么值得保存 (Why Save This)
- **它在解决什么问题**：它直接回答了一个团队最容易做错的工程判断: 什么时候多 Agent 真能解决问题，什么时候只是在给系统额外加税。
- **它对现有知识图谱补了什么**：现有 Topic 已经能讲清“先补 Spec / Skills / Harness，再谈框架升级”的顺序；这篇文章补的是来自一线平台实践的更硬判据，把“别急着上多 Agent”从经验判断补成可执行的决策门。
- **如果不记住会漏掉什么**：你很容易把多 Agent 当成复杂任务的默认答案，忽略真正的收益常常只出现在上下文隔离、并行搜索和专业化工具集三类场景里。
- **它在学习链路中的角色**：它不是最终 Guide 本身，而是给 Guide 和 Topic 提供证据层支撑的 Source，尤其适合拿来约束“何时升级 Sub-Agent 编排”的门槛。

## 2. 核心机制与论证链 (Core Logic)
### 2.1 主机制 / 完整主线
- **背景约束**：团队一看到任务变复杂，就倾向于拆 planner / worker / reviewer，默认把“多角色”理解成“更高级”。但多 Agent 会引入额外提示维护、上下文传递和协调失败点。
- **关键判断**：多 Agent 不该为“复杂感”买单，而该为“单 Agent 无法跨过去的具体约束”买单。文章把这个约束收敛成三类: 上下文污染、可并行搜索空间、以及工具 / prompt / 专长的明显分域。
- **主机制 / 方案**：先把单 Agent 做到位，再检查是否出现以下硬信号：一是单 Agent 被高噪声上下文拖垮，二是任务能被拆成相互独立的并行研究面，三是单 Agent 的工具集合与行为模式已经明显互相打架。只有命中这些条件，多 Agent 的收益才有可能超过协调成本。
- **为什么这比常见做法更成立**：因为它把“多 Agent 值不值得”从架构偏好，改成了成本收益判断。文章明确指出，在其实单 Agent 就能解决的问题上，多 Agent 往往会多花 3-10 倍 token，却不一定带来更好结果。

### 2.2 关键分支 / 条件 / 例外
- **并行主线 / 次主线**：上下文隔离带来的收益，和并行研究带来的收益不是一回事。前者解决主上下文被脏信息稀释，后者解决搜索面不够宽。
- **关键条件 / Guard**：如果子任务之间共享状态很重、输出边界说不清，或只是因为“任务听起来复杂”才拆分，那么拆成多 Agent 大概率只会制造更多传话成本。
- **例外 / 易错分叉**：验证子代理是一个相对稳定有效的特殊模式，因为 verifier 不需要理解完整构建历史，只需要按 success criteria 做黑盒检查；但这也不是绝对铁律，更强 orchestrator 有时可以直接承担评估职责。

## 3. 关键证据 / 案例 / 事实 (Evidence Anchors)
| 证据锚点 | 核心信息 | 来源/条件 | 对结论的支撑 |
|----------|----------|----------|--------------|
| 单 Agent 基线 | 文章先明确“多数企业工作流先从单 Agent 开始” | Claude 官方博客，2026-01-23 | 说明多 Agent 不是默认基线，而是升级选项 |
| 协调成本 | 多 Agent 实现通常消耗单 Agent 的 3-10 倍 token | 官方测试观察，主要来自上下文复制、协调消息和 handoff 摘要 | 给“别为了复杂感而拆分”的成本证据 |
| 上下文隔离门槛 | 当子任务会引入 1000+ token 高噪声且大部分对主任务无关时，隔离才明显有效 | 订单查询 + 技术支持案例 | 说明“上下文污染”必须是可观察现象，不是抽象担心 |
| 工具分域信号 | 当一个 Agent 面临 20+ 工具、跨多个不相关领域时，工具选择质量会明显下滑 | CRM / marketing / messaging 集成示例 | 说明专业化拆分要建立在工具域冲突上 |
| 验证子代理例外 | verification subagent 在多领域都稳定有效，因为它天生减少 context transfer | 文中 verifier pattern | 说明“执行者不自证”是多 Agent 中少数高回报模式 |

## 4. 适用边界与反模式 (Boundaries)
- **适用边界**：适合已经出现上下文爆炸、并行研究需求或工具域冲突的 Agent 系统，尤其适合把“是否要引入 Sub-Agent”变成架构审查问题的团队。
- **常见误判**：把“任务复杂”误判成“必须多 Agent”，或者按角色名拆分，却不先定义输入、输出和局部验收。
- **代价 / 失效信号**：如果拆分后 token 成本明显上涨、协调时间拉长、主协调者不得不反复重写 worker 产物，说明收益没有覆盖协调税。
- **取舍 / 阈值**：文章给出的真实取舍不是“能不能做”，而是“是否值得为更宽搜索面和更干净上下文支付更高 token 与更长总执行时间”。
- **替代路径**：当问题主要来自任务契约不清、背景供给不足或执行验证缺失时，应先回到 (relates) [[Concepts/Agent 框架选型]]、(relates) [[Topics/AI 编码工程化与 Agent 体系选型]] 对应的更低层补救，而不是直接上多 Agent。

## 6. 我可以怎么复用 (Reuse Actions)
| 动作 | 适用场景 | 验收标准 | 失败信号 |
|------|----------|---------|---------|
| 先为当前工作流做一次“单 Agent 是否已到边界”审查 | 团队准备引入 planner / worker / reviewer 前 | 能说清是上下文污染、并行研究还是工具域冲突在驱动升级 | 只能笼统说“任务很复杂，所以想拆” |
| 只在一个高噪声子任务上试点 Sub-Agent | 例如长日志排查、资料检索、订单 / 用户历史过滤 | 主上下文 token 压力下降，主 Agent 输入更短更聚焦 | worker 回来仍携带大段原始上下文，主 Agent 没有真正减负 |
| 将 verifier 独立成 reviewer 子代理 | 已有清晰 success criteria 的编码 / 文档 / 测试任务 | verifier 可以只凭产物和标准完成判定，不需要完整构建历史 | verifier 也被迫重复理解全部构建过程，说明验收标准不清晰 |
