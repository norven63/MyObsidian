## 2. 核心机制与论证链 (Core Logic)
### 方案推导链条 (AI分析)
- **初始痛点**：直接和 AI 聊天写代码会把不同复杂度的需求都拖进同一套高摩擦流程，导致简单需求负担过重，复杂需求又缺少硬门禁。
- **演进尝试**：作者调研多种 Spec Coding 框架后，发现“Spec 全量强制”本身也是偶然复杂度。
- **底层逻辑**：规则始终生效，Spec 按复杂度按需加载；简单需求不强迫走完整文档流程，中高复杂度需求则用 Spec、tasks、review、archive 建立强门禁和可追溯上下文。

### 反事实推演 (Counterfactual Timeline)
- **第 1 周**：团队把所有需求都交给 AI 直接编码，快但不可追责。
- **第 4 周**：复杂需求开始频繁跑偏，讨论、规划、执行、审查互相打架。
- **第 12 周**：没有 knowledge/ 和 Reverse Sync 的团队，所有改进都停留在聊天记录里，下一轮还要从零再讲一遍。

## 4. 适用边界与反模式 (Boundaries)
- **反模式**：把 Spec 当成“自动生成代码的唯一真理”，要求 AI 一步从规范映射到完整实现。
- **适用边界**：≤5 人日的小需求不该承担完整 spec 流程；但核心业务迁移和跨模块变更必须有 HARD-GATE。
- **利益冲突/偏见**：文章默认团队接受较强流程纪律，包括显式确认、逐 task 执行、自动 commit 与两阶段 review。
- **替代方案对比**：Vibe Coding 更快但不可控；固定重型 Spec 框架更稳但拖慢简单任务；渐进式 Spec 把流程重量和任务复杂度对齐。
- **最大陷阱**：只有 rules 没有 knowledge/，AI 只会“守格式”，不会真正理解业务与架构的来龙去脉。

## 5. 我可以怎么复用 (Reuse Actions)
### 最小可运行原型 (PoC)
```yaml
rules:
  always_on:
    - project-context.md
    - coding-style.md
    - security.md
workflow:
  simple: [research, apply]
  medium: [research, propose, apply, review]
  complex: [research, propose, apply, fix, review, archive]
gate:
  no_spec_no_code: true
  reverse_sync: true
```
