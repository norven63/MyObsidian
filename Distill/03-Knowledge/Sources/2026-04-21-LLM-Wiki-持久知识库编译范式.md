# LLM Wiki：让知识库从一次次检索变成持久编译

## 1. 为什么值得保存 (Why Save This)
- **它在解决什么问题**：它回答的是“为什么很多知识库产品明明能检索文档，却还是让 LLM 在每次提问时从零重新拼装知识”，并把问题从“检索够不够强”转成“有没有一个会持续累积和修缮的中间知识层”。
- **它对现有知识图谱补了什么**：现有图谱里已经有 `Agent Skills`、`Harness Engineering`、`独立评估器` 这些运行时与知识沉淀概念，但还缺一篇把它们投射到“知识库本身如何被 LLM 持续维护”上的证据层材料；这篇正好把 raw sources、wiki、schema、ingest/query/lint 组回同一条主线。
- **如果不记住会漏掉什么**：你容易把知识库建设继续理解成“把文件上传给 RAG”，却看不到“让知识在文档层先被编译、交叉引用、标注冲突、积累 synthesis”这条更高杠杆的路径。
- **它在学习链路中的角色**：它不是一篇工具教程，而是 `LLM 持久 Wiki 与知识库编译范式` 这条 Topic 的底层证据，尤其适合支撑“何时该从 query-time retrieval 升级到持久 wiki”的判断。

## 2. 核心机制与论证链 (Core Logic)
### 2.1 主机制 / 完整主线
- **背景约束**：当资料会持续累积、问题会反复出现、而且很多问题都需要跨 5 篇以上材料做综合判断时，单纯的 query-time retrieval 会让 LLM 每次都重新找片段、重新拼知识、重新形成临时结论。
- **关键判断**：真正有复利的知识系统，不该只在提问时“临时找材料”，而应该先把知识编译进一个持久、可互链、可修缮的 wiki 中间层。
- **主机制 / 方案**：文章把系统拆成三层：`Raw sources` 负责存原始证据且保持不可变，`Wiki` 负责由 LLM 增量维护的结构化知识页，`Schema` 负责告诉 LLM 应该如何 ingest、query 和 lint。于是每次新 source 到来时，LLM 不只是生成一次回答，而是更新多个 wiki 页面、补 cross-link、记录冲突、维护 index 与 log，让后续问题优先建立在已经编译好的知识层上。
- **为什么这比常见做法更成立**：因为它把“每次问答都要从原料重建知识”改成“知识先被编译成持久工件，再供问答调用”。这样真正会累积的是 wiki 本身，而不是一段随会话消失的回答。

### 2.2 关键分支 / 条件 / 例外
- **并行主线 / 次主线**：除了 ingest 之外，文章还明确给出 query 与 lint 两条并行主线：query 产生的新对比、新分析可以回写成 wiki 页面，lint 则负责持续发现冲突、陈旧说法、孤立页面和可继续补的空白。
- **关键条件 / Guard**：这个模式只有在资料会持续增长、问题会反复复用，而且愿意维护 schema / index / log 时才真正值回票价。
- **例外 / 易错分叉**：如果你的资料集很小、问题低频、或者知识需要强实时刷新，继续用简单检索、数据库查询或一次性上传文件，通常更便宜；强行上“持久 wiki”只会多出维护成本。

## 4. 适用边界与反模式 (Boundaries)
- **适用边界**：适合研究型、长期学习型、团队内知识沉淀型场景，尤其当资料持续进来、问题重复出现、而且你希望分析结果不要只留在聊天记录里时。
- **常见误判**：把这套模式理解成“更高级的 RAG”或“把 Obsidian 当数据库”。其实它真正强调的是持久 synthesis、cross-link 维护和健康巡检，而不是换一个检索前端。
- **代价 / 失效信号**：如果 raw source 和 published wiki 没分层、schema 只是一篇没人维护的说明、query 结果从不回写、lint 也从未执行，wiki 很快会退化成一堆静态摘要和孤岛页面。
- **取舍 / 阈值**：当问题越来越常需要跨文档综合、跨时间修订、把旧结论与新证据对照时，维护 wiki 的成本才开始低于“每次重新问一遍”的成本；在此之前，简单检索仍可能更划算。
- **替代路径**：小而静态的资料集更适合简易检索；强实时结构化事实更适合数据库 / API 查询；需要制度级权威口径时，持久 wiki 仍应作为解释层而不是唯一事实源。

## 6. 我可以怎么复用 (Reuse Actions)
| 动作 | 适用场景 | 验收标准 | 失败信号 |
|------|----------|---------|---------|
| 先把仓库拆成 raw sources / published wiki / schema 三层 | 你正准备让 LLM 长期维护一个知识库 | 每个文件都能说清自己属于“原始证据”“编译结果”还是“维护规则” | 原始资料被持续改写，或者 LLM 输出与原料混在同一目录 |
| 用单篇 source 跑一次 ingest，并要求它同时更新 index、log 和至少一个已有主题页 | 你想验证知识是否真的会被编译进图谱，而不只是多出一篇摘要 | 一次 ingest 不止生成一份新页，而是能解释“哪些既有页面被补强了” | 新 source 只生成孤立摘要，既有主题与导航完全不动 |
| 把高价值 query 结果回写成 page，而不是留在对话里 | 你发现自己在重复问同类综合问题 | 回答能变成新的 comparison / synthesis / guide 页面，并进入 index | 重要分析永远停留在聊天记录，下次还得重新问 |
| 定期跑 lint / health-check，主动找陈旧说法、断链和孤立页 | 图谱开始持续增长 | 系统能给出明确 gap list、冲突点和下一批补料方向 | wiki 页数增加，但没人知道哪些页已过时或无人可达 |
