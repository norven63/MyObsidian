## 2. 核心机制与论证链 (Core Logic)
### 方案推导链条 (AI分析)
- **初始痛点**：很多团队已经知道 LLM 会“回答问题”，却不知道何时该引入一个真正能规划、记忆、调用工具的 Agent，更不知道该按什么标准选框架。
- **演进尝试**：行业把 Agent 的复杂度逐渐拆成规划、记忆、工具、状态流转、多角色协作、数据连接等层，再由不同框架在各自最强的层上提供抽象。
- **底层逻辑**：先判断任务到底是“简单工具调用”“有状态工作流”“多 Agent 协作”还是“数据密集型 RAG”，再选 LangChain、LangGraph、CrewAI、AutoGen、LlamaIndex 等底座，而不是反过来按热度倒推需求。

### 反事实推演 (Counterfactual Timeline)
- **第 1 周**：团队先把 Agent 理解成“能调用工具的 LLM”，能做演示，但能力边界还不清楚。
- **第 4 周**：随着需求增多，状态流转、多角色协作和数据访问被混成同一类问题，框架差异开始变重要。
- **第 12 周**：如果始终不按任务形态区分框架，选型讨论会反复回到“这个框架到底适不适合现在的问题”。

## 4. 适用边界与反模式 (Boundaries)
- **反模式**：把 AI Agent 理解成“会聊天 + 会调 API”的统一外壳，以为所有框架都只是语法不同的同一件事。
- **适用边界**：这篇更像入门与选型地图，强项是帮助读者建立框架坐标，而不是给出统一实验条件下的深度对比。
- **利益冲突/偏见**：文章唯一完整实战使用 LangChain，因此读者会更容易先从 LangChain 理解 Agent 开发流程。
- **替代方案对比**：文中把框架按问题形态分层：LangChain 偏通用工具链，LangGraph 偏状态工作流，CrewAI/AutoGen 偏多角色协作，LlamaIndex 偏数据连接与 RAG。
- **最大陷阱**：把不同框架负责的复杂度层级混为一谈，只看名字，不看任务结构。

## 5. 我可以怎么复用 (Reuse Actions)
### 最小可运行原型 (PoC)
```python
def choose_framework(requirements: dict[str, bool]) -> str:
    if requirements["rag_or_knowledge_base"]:
        return "LlamaIndex"
    if requirements["multi_agent_collaboration"]:
        return "CrewAI or AutoGen"
    if requirements["stateful_workflow"]:
        return "LangGraph"
    return "LangChain"
```
