# Agent 框架选型

## 先把Agent 框架选型放进脑子里
### 一眼认知骨架
- **对象**：Agent 框架选型是根据任务的知识密度、状态复杂度、协作方式与运行时要求，为问题选择合适承载层——有时甚至是“暂时不选框架”——的决策过程。
- **目的**：它解决的是“当前难点到底来自哪里”的工程判断：是知识供给不足、状态流转混乱、多角色协作复杂，还是工具接入和恢复审计不够稳定。
- **组成**：把它拆开看，真正稳定决定结果的是：先识别主复杂度来自哪里：知识检索、状态流转、并行协作、工具生态，还是运行时恢复与治理
- **主线**：先识别主复杂度来自哪里：知识检索、状态流转、并行协作… → 再判断问题是否真的需要升级框架 → 只有当复杂度已经超出轻量方案承载范围，才把需求映射到… → 最后用真实任务试跑，而不是只看 stars、教程和…
- **变体**：边界、分支和失败条件会改写这条主线；后文重点看它最容易被忽略的前提、误判点和不适用场景。
- **用法**：当单次 prompt、脚本或轻工作流已经无法稳定承载任务，你需要判断是补状态图、补多角色系统、补知识底座，还是根本不该升级时，就该想到选型。

### 展开理解
- 选型本质上是在判断“复杂度应该住在哪里”：住在知识检索层、状态机层、协作层，还是干脆不该离开轻量方案。
- 很多表面上的“框架问题”，其实是需求边界、验收契约或运行时护栏还没讲清楚；真正成熟的选型，先诊断问题，再决定是否换底座。

## 为什么不能只把Agent 框架选型当成一个名词定义
### 它解决的判断 / 工程问题
- 它解决的是“当前难点到底来自哪里”的工程判断：是知识供给不足、状态流转混乱、多角色协作复杂，还是工具接入和恢复审计不够稳定。
- 它还能避免两类常见浪费：轻问题上重框架，导致系统复杂度先爆炸；重问题还靠脚本硬撑，导致逻辑偷偷藏进 prompt 和人工补丁里。

### 如果忽略它会怎样
- 团队会把“最热的那个”误当成“最合适的那个”，最后不是买来了根本用不上的能力，就是把核心问题继续埋在不适合的抽象下面。
- 更糟的是，框架一旦成了信仰，后续所有设计讨论都容易退化成品牌争论，而不再围绕任务状态、验收边界和运行成本本身展开。

### 为什么系统宁可这样设计
框架选型之所以要独立出来，是因为团队真正要买单的不是“功能名词”，而是主复杂度的承载方式：知识检索、显式状态、多人协作、恢复 / 审计，各自对应不同底座。这样换来的是你能把成本花在最匹配的抽象上；代价是前面必须先做复杂度诊断，并接受“很多问题其实还不到换框架的时候”。

## 真正决定Agent 框架选型结果的，是哪条机制链
### 机制链
1. 先识别主复杂度来自哪里：知识检索、状态流转、并行协作、工具生态，还是运行时恢复与治理；没有这个诊断，选型只会变成拍脑袋。
2. 再判断问题是否真的需要升级框架；很多时候补一个更清晰的 Spec、一个更可靠的 Harness，收益比换底座更直接。
3. 只有当复杂度已经超出轻量方案承载范围，才把需求映射到对应类型：知识密集型偏向 RAG / LlamaIndex，显式状态与重试更像 LangGraph，多角色协作更像 CrewAI / AutoGen。
4. 最后用真实任务试跑，而不是只看 stars、教程和 demo；真正该比较的是状态是否更清晰、失败是否更容易定位、团队是否更能维护。

### 关键条件 / 分支 / 例外
- 如果主问题仍是 Spec 不清、Harness 缺失或知识沉淀不足，先补前置层，不要用框架替代 diagnosis。
- RAG-heavy、state-machine-heavy、multi-agent-heavy 的任务，最优底座通常不同，不能用一个流行标签兜底。
- 真正的选型结果必须在真实任务、失败恢复与团队维护成本上验证，而不是只看 demo、教程或 stars。

### 最低决定层 / 关键锚点
- 最低锚点是 complexity matrix：先定位复杂度源头，再映射到 RAG / state graph / orchestration / runtime。
- 其次是底座对 state visibility、retry / recovery、evaluator integration 和 tool ecosystem 的承载能力。
- 最后要看迁移与运维成本：如果接入后大量逻辑仍躲在 prompt 和人工补位里，说明框架并没有接住主复杂度。
- 如果讨论始终停在 stars、demo 和品牌名，就会误判“选型已经完成”，并看丢 complexity matrix 与 state / recovery 承载能力这层真正决定成败的最低锚点。

### 排查 / 应用抓手
- **排查时先问**：现有方案到底卡在知识供给、状态可见性、协作分工，还是运行时稳定性？如果问题都还没定性，选型一定会跑偏。
- **设计时先问**：如果今天不许换框架，最先该补的是什么——Spec、Harness、RAG 还是状态机？这个问题能帮助你识别“换框架”是不是伪需求。
- **常见观察信号**：讨论只剩“谁更火、谁教程多、谁是一站式平台”，或者框架上了以后团队仍说不清状态边界与验收口径，通常说明选型没有对准真实复杂度。

## 用一个最小场景看清Agent 框架选型是怎么落地的
### 最小例子
- **场景**：团队要做“内部支持助手”。如果当前需求主要是读知识库并回答问题，核心复杂度在知识供给；如果随后又要管理工单状态、升级路径与人工接管，复杂度才转向状态机和流程编排。
- **为什么这里会想到它**：表面上都叫“做 Agent”，但背后的困难根本不是同一类；不先拆清复杂度来源，就会把多 Agent、RAG 和工作流混成同一个问题。
- **结果**：正确的选型不会只给出一个品牌名，而会先回答“我们究竟在为哪种复杂度买单”：如果主要是知识回答，RAG / LlamaIndex 往往已足够；如果后来新增工单状态、升级路径和人工接管，再考虑 LangGraph 这类显式状态承载层；若只是三四个固定工具调用，甚至不必升级框架。

## 把Agent 框架选型学歪后，最容易出现哪些误判
### 常见误解
- 把框架选型放在第一步。真实顺序应该是先识别任务形态，再判断要不要升级承载层。
- 以为“支持多 Agent / 支持图编排 / 支持 RAG”这些标签本身就能说明适配度；标签只是功能清单，不是问题匹配证明。

### 失效 / 反噬信号
- 如果框架引入后，团队仍然说不清任务状态、失败边界和验收标准，那么你只是把混乱搬到了更重的抽象里。
- 如果实现过程中大量逻辑仍躲在 prompt、手工脚本或人工补位里，说明所选框架并没有真正接住核心复杂度。

### 不适用场景
- 对低频、低风险、低复杂度任务，最好的选型往往就是不选框架，而是继续用脚本、局部工作流和清晰规则解决问题。

## Agent 框架选型总要和这些概念一起看
- [[Concepts/AI Agent|AI Agent]] `(depends[中])`：先知道一个 Agent 至少要承担什么能力，选型时才有比较基准。
- [[Concepts/Sub-Agent 编排|Sub-Agent 编排]] `(extends[中])`：当复杂度明确来自多角色协作时，选型自然会偏向支持任务拆解与结果汇总的底座。
- [[Concepts/Harness Engineering|Harness Engineering]] `(complements[中])`：框架决定“谁承载复杂度”，Harness 决定“承载以后怎样更稳定地跑”。

## 先记住Agent 框架选型的核心判断，再用三个问题检查自己
### 记忆锚点
- **一句话记住**：先判复杂度，再判底座；选型不是“谁更强”，而是“谁更对题”。
- **看到这个信号就想到它**：团队开始争论框架品牌，却还说不清任务的主复杂度到底来自哪里。

### 自测问题
1. 什么情况下你该先补 Spec 或 Harness，而不是急着做框架选型？
2. 如果一个任务的主要复杂度来自知识供给，为什么不该默认上多 Agent 框架？
3. 说出一个“最热框架”反而会成为错误选择的场景，并解释错在哪。
