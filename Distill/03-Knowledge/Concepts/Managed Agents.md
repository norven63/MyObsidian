# Managed Agents

## 1. 它是什么 (What It Is)
### 快速判断 / Quick Scan
| 维度 | 内容 |
| --- | --- |
| 一句话定义 | Managed Agents 是把 session、harness、sandbox 等运行时对象抽成稳定接口并托管化交付的做法，目的是让长任务执行在模型和底层基础设施变化时仍然可靠。 |
| 不要和什么混为一谈 | 它不等于“把 Agent 搬到云上”；真正关键的是接口稳定、状态恢复和安全边界，而不只是部署位置改变。 |
| 什么时候想到它 | 当 Agent 需要跨会话恢复、长时间运行、多租户隔离、统一审计和受控工具访问时，就应该想到 Managed Agents。 |

### 展开理解
- Managed Agents 关注的不是“某个 Agent 会不会更聪明”，而是“这套运行时能不能在模型、容器、网络和权限边界都变化时还保持连续可用”。它把平台级稳定性问题从单次任务里剥离出来。

## 2. 为什么它重要 (Why It Matters)
### 它解决的判断 / 工程问题
- 它解决的是“哪些东西应该比单次容器、单代模型活得更久”的问题：一旦把任务状态、执行循环、凭证和运行环境都塞进同一个短命容器，模型升级、容器故障、网络接入和权限隔离就会一起变成平台瓶颈。
- 它也把“Agent 能否恢复”“凭证能否隔离”“不同模型能否继续接棒”这些问题前置成运行时契约，而不是等线上事故发生后再靠补丁挽救。

### 如果忽略它会怎样
- 你会得到一个看起来能跑、但非常脆的系统：任务一旦跨会话、跨环境或跨模型版本，就容易从“偶尔不稳定”升级成“整个流程不可恢复、不可审计、不可安全扩展”。

## 3. 它是怎么工作的 (How It Works)
### 机制链 / Mechanism Chain
1. 先把“会话状态”“执行外壳”“沙箱环境”等长期存在的问题抽象成独立对象，而不是让它们附着在某次临时进程里。
2. 为这些对象定义比某一代模型或某个具体容器更长寿的接口：session 负责可恢复历史，harness 负责任务推进与上下文装配，sandbox 负责受控执行与环境隔离。
3. 再由平台接管对象生命周期、故障恢复、权限边界和审计记录，让任务可以在不同模型、不同执行环境之间继续接棒，而不必重写上层工作流。
4. 这样即使模型、调度器或沙箱实现发生替换，任务编排与历史状态仍能保持连续性；变化被压在接口后面，而不是渗透到业务逻辑里。

### 排查 / 应用抓手 (Diagnostic / Application Hooks)
- **排查运行时脆弱性时先问**：任务状态到底存在 prompt 里、容器里，还是独立 session 里？如果进程一死就全丢，说明还没进入 Managed Agents 范式。
- **设计平台边界时先问**：哪些能力必须比模型版本更稳定？哪些凭证绝不能落进 sandbox？这些问题通常决定 session / harness / sandbox 的接口切分方式。
- **常见观察信号**：每次模型升级都要大面积重写平台逻辑、调试必须 SSH 进“宠物容器”、凭证和执行代码混在同一环境里，都是所谓“托管”仍停留在表面的信号。

## 4. 一个最小例子 / 对比 (Minimal Example / Contrast)
### 最小例子
- **场景**：一条“分析仓库并修复缺陷”的长任务已经运行了 40 分钟，过程中容器崩溃或底层模型需要切换版本。
- **为什么这里会想到它**：如果状态只存在单个容器内，重启后就只能从零开始；如果 session、执行历史和权限边界都被托管，任务可以在新的执行环境里继续接棒。
- **结果**：真正被用户感知到的差别不是“跑在云上”，而是“任务没有因为底层实现替换而失忆”。

### 对比
| 易混对象 | 真正差异 | 这里为什么不是它 |
| --- | --- | --- |
| 纯托管部署 / 把 Agent 放进云容器 | 主要解决部署位置与运维托管 | 它不自动提供稳定接口、恢复语义或安全边界 |
| [[Concepts/Harness Engineering]] | 更像运行时设计原则与方法论 | Managed Agents 是把这些原则产品化、平台化后的交付形态 |

## 5. 常见误解与边界 (Mistakes & Boundaries)
### 常见误解
- 只要把 Agent 托管出去就会自动稳定。实际上，如果没有版本治理、状态存储、权限分层和接口演进策略，托管只会让复杂度集中爆炸。

### 失效 / 反噬信号
- 如果每次模型升级或工具替换都要大面积重写平台逻辑，或者运行时一出事就只能去翻容器内部状态，说明所谓“稳定接口”并没有真正建立起来。

### 不适用场景
- 低频、短链路、失败成本可接受的任务，往往不值得一开始就建设完整托管平台；此时更轻量的 Harness 或任务脚本通常更划算。

## 6. 与哪些概念容易一起出现 (Nearby Concepts)

- [[Concepts/Sub-Agent 编排|Sub-Agent 编排]] `(implements[中])`：托管运行时常会把高噪声执行分配给多个独立执行单元，从而支持更复杂任务网络；没有受控运行时，子代理越多越容易失控。
