# Sub-Agent 编排

## 先把Sub-Agent 编排放进脑子里
### 一眼认知骨架
- **对象**：Sub-Agent 编排是一种把复杂任务拆给多个独立上下文执行单元的运行时策略，由协调者负责任务规划、委派与结果汇总，执行者在隔离上下文中完成具体动作。
- **目的**：它解决的是“什么时候一个超长上下文已经比一个任务网络更难维护”的问题。单个 Agent 一旦同时背负长日志、多分支探索和失败历史，决策质量很容易被噪声拖垮。

- **变体**：边界、分支和失败条件会改写这条主线；后文重点看它最容易被忽略的前提、误判点和不适用场景。
- **用法**：当任务能被拆成若干相对独立的子问题、不同步骤需要不同专长，或者主上下文已经被长日志、大 diff 和多分支探索拖垮时，就该想到子代理编排。

### 展开理解
- Sub-Agent 编排最核心的收益，往往不是并行本身，而是上下文垃圾回收：把高噪声执行、分支探索和局部细节从主上下文里隔离出去。
- 真正可扩展的多 Agent，不靠“角色名字起得多漂亮”，而靠每个子代理都拿到清晰输入、明确产物和有限职责，协调者只处理汇总与决策。

## 为什么不能只把Sub-Agent 编排当成一个名词定义
### 它解决的判断 / 工程问题
- 它解决的是“什么时候一个超长上下文已经比一个任务网络更难维护”的问题。单个 Agent 一旦同时背负长日志、多分支探索和失败历史，决策质量很容易被噪声拖垮。
- 它还帮助团队把“谁负责拆分、谁负责执行、谁负责验收”区分开，避免一个主代理既要想战略，又要处理所有局部脏活，最终把自己压成瓶颈。

### 如果忽略它会怎样
- 复杂任务会继续堆在一个大上下文里，结果不是推理更强，而是切换成本更高、记忆污染更重、失败分支更难回收。
- 反过来，如果不加边界就盲目上多 Agent，又会得到另一种混乱：多个执行单元互相覆盖状态，协调者反而花更多时间救火和重写别人产物。

### 为什么系统宁可这样设计
子代理编排独立出来，不是为了凑角色，而是为了把长任务里彼此独立的噪声拆进不同上下文，让协调者只看结构化产物和下一轮分派。这样换来的是主上下文更干净、局部探索可并行；代价则是必须设计清楚子任务 I/O、共享状态和最终汇总规则，否则多 Agent 只会把耦合放大。

## 真正决定Sub-Agent 编排结果的，是哪条机制链
### 机制链
1. 先由协调者把总体目标拆成若干可独立理解、可局部验收的子任务，并为每个子任务定义输入、输出和成功条件。
2. 每个子任务进入独立上下文，由对应执行代理只读取和自己有关的背景、工具与约束，从而避免把整个任务历史一起背在身上。
3. 协调者收集子代理返回的 diff、报告、测试结果或研究结论，判断是否有冲突、缺口或新的分支需要继续下发。
4. 最终再由评估器或主流程对合并结果做统一验收，避免“每个子任务都看起来对，但整体仍然错”的局面。

### 关键条件 / 分支 / 例外
- 只有子任务输入输出清楚、部分独立时才值得拆；没有边界的复杂任务换成多 Agent 只会更乱。
- 如果多个 worker 高频改同一状态或互相等待，说明拆分边界没有立住，编排层并未真正减负。
- 最终必须有统一 evaluator / coordinator 收口，否则“局部都对、整体仍错”会被放大。

### 最低决定层 / 关键锚点
- 最低锚点是 task contract：每个 sub-agent 拿什么输入、交什么产物、由谁验收。
- 其次是 context isolation：worker 只读与本任务相关的背景、工具和约束，而不是把全任务历史一起背在身上。
- 最后是 coordinator merge + evaluator gate；这一步决定并行产物如何变成整体结论。

### 排查 / 应用抓手
- **排查时先问**：每个子代理到底要交付什么产物——代码、报告、测试结果还是审计结论？如果产物说不清，拆分大概率也不清楚。
- **设计时先问**：哪些状态必须共享，哪些必须隔离？好的编排先设计工件边界，再设计角色数量。
- **常见观察信号**：协调者频繁重写子代理产出、多个子代理反复改同一文件、并行执行没有明显减轻主上下文负担，通常都说明拆分边界失效了。

## 用一个最小场景看清Sub-Agent 编排是怎么落地的
### 最小例子
- **场景**：一次跨多个模块的重构，需要同时完成接口分析、迁移脚本编写和测试补齐；每一部分的上下文和噪声类型都不一样。
- **为什么这里会想到它**：如果让一个主代理从头到尾同时处理这三件事，它很快会在大量局部细节里丢失总体判断；而这三类子任务又恰好可以各自独立推进并回传结果。
- **结果**：协调者可以让接口分析子代理回传依赖与风险清单、让迁移子代理回传脚本与变更说明、让测试子代理回传验证结果，再由统一评估环检查接口兼容性和测试通过情况；主流程因此拿到的是结构化产物，而不是一坨混在一起的执行噪声。

### 对比
| 易混概念 / 做法 | 真正差异 | 这里为什么不是它 |
| --- | --- | --- |
| 单 Agent 执行 | 上下文统一、协调成本低 | 这里的主要问题是噪声过高、分支过多，一个上下文已经很难同时承接所有局部细节 |
| [[Concepts/Agent Skills]] | Skill 负责封装知识与步骤模板 | 这里要解决的是“谁来做、怎样隔离上下文”，不是“知道什么长期可复用” |

## 把Sub-Agent 编排学歪后，最容易出现哪些误判
### 常见误解
- 只要任务复杂就该上多 Agent。实际上，无法拆清接口的任务，换成多 Agent 只会增加汇总成本。
- 以为“并行”就是主要收益。对子代理系统来说，更重要的往往是把长任务的上下文噪声隔离开，而不是单纯堆高同时执行数。

### 失效 / 反噬信号
- 如果协调者不断亲自下场修补所有产出，说明它已经从编排者退化回执行者，拆分没有真正减负。
- 如果多个子代理频繁覆盖同一状态文件、反复等待别人先完成，说明任务之间并不独立，编排层只是把耦合暴露得更明显了。

### 不适用场景
- 强依赖共享上下文、结果高度耦合、局部拆分收益很小的任务，往往仍更适合单体执行；把它们拆成多 Agent 只会增加协调噪声。

## 先记住Sub-Agent 编排的核心判断，再用三个问题检查自己
### 记忆锚点
- **一句话记住**：子代理编排拆的是上下文噪声和职责边界，不是为了凑更多角色名字。
- **看到这个信号就想到它**：一个主代理开始同时背大日志、多分支探索和大量局部细节，已经明显被上下文污染拖慢。

### 自测问题
1. 什么症状说明你该考虑子代理，而不是继续把所有执行塞进主上下文？
2. 协调者最不该亲自做的事情是什么，为什么？
3. 如果一个任务拆分后仍频繁冲突，你会优先怀疑拆分中的哪一类边界问题？
