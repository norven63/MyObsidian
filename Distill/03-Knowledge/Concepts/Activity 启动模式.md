# Activity 启动模式

## 1. 它是什么 (What It Is)
### 快速判断 / Quick Scan
| 维度 | 内容 |
| --- | --- |
| 一句话定义 | Activity 启动模式是 Android 在执行 `startActivity` 时，用来决定“新建还是复用实例、任务栈如何重排、返回路径如何保持一致”的规则集合。 |
| 不要和什么混为一谈 | 它不是单纯的 API 背诵题，也不只是 Manifest 参数；它真正约束的是导航语义如何映射到任务栈秩序。 |
| 什么时候想到它 | 多入口页面、通知/深链接唤起、返回路径异常、同一页面是否允许多实例共存时，都应该先想到启动模式。 |

### 展开理解
- 启动模式的关键不是“页面能不能打开”，而是“这次打开会怎样改写已有历史”。当你把它看成实例管理与任务栈治理规则，就能理解为什么它会同时影响生命周期、返回键行为和用户对导航稳定性的预期。

## 2. 为什么它重要 (Why It Matters)
### 它解决的判断 / 工程问题
- 它帮助你判断某次跳转到底该“自然入栈”还是“复用旧实例并重整历史”，从而避免多个入口把同一个页面推成互相冲突的返回路径。
- 它也帮助团队把“页面导航设计”和“实例生命周期管理”放在同一张图里思考，而不是 UI 设计一套、运行时行为另一套。

### 如果忽略它会怎样
- 你会把大量“返回栈怎么乱了”“页面怎么又开了一个”“通知回到哪都不对”的问题误判成 UI Bug；但真正根因常常是导航语义与任务栈规则不匹配。

## 3. 它是怎么工作的 (How It Works)
### 机制链 / Mechanism Chain
1. 当 `startActivity` 发生时，系统会同时看调用上下文、`Intent` flags、目标 Activity 的 `launchMode`，以及当前任务栈里是否已经存在可复用实例。
2. 系统据此决定是创建新实例、把 `Intent` 送给已有实例，还是把目标实例连同其上的历史一起搬到前台 / 清理重组。
3. 这个决定会直接改变生命周期回调路径：新建实例更可能走 `onCreate`，复用实例更可能走 `onNewIntent`，同时影响用户返回时看到的栈顺序。
4. 所以启动模式本质上是在把“入口语义”翻译成“实例复用 + 任务栈整理”的运行时动作；最终结果不是某个 API 调用，而是用户从哪里来、按返回会回哪里去。

### 排查 / 应用抓手 (Diagnostic / Application Hooks)
- **排查返回栈异常时先问**：这次进入是从列表、通知还是深链接来的？业务预期是保留旧历史，还是把当前页面当作新的入口？
- **设计导航时先问**：这个页面允许多实例共存吗？如果用户二次进入同一页面，应该看到一个新实例，还是回到旧实例并刷新内容？
- **常见观察信号**：重复详情页、返回键把人带回奇怪入口、`onNewIntent` 触发与否和预期不一致、通知点击后把原任务栈意外清空，都是启动模式配置与导航语义错位的典型信号。

## 4. 一个最小例子 / 对比 (Minimal Example / Contrast)
### 最小例子
- **场景**：用户先从列表页进入某个商品详情页，稍后又从通知点击进入同一商品详情。
- **为什么这里会想到它**：列表入口通常希望“自然入栈”，而通知入口更常希望“复用已有详情页并重整返回路径”，否则用户会得到两个几乎一模一样的详情页。
- **结果**：一旦实例复用策略设计正确，用户感知到的就不是“又开了一页”，而是“我回到了那个本来就在做的任务上下文”。

### 对比
| 易混对象 | 真正差异 | 这里为什么不是它 |
| --- | --- | --- |
| 普通跳转写法 / Nav API | 负责表达“要去哪个页面” | 它能发起跳转，但不单独决定旧实例是否复用、任务栈是否重排 |
| 启动耗时排查 | 负责解释“为什么慢” | 启动模式解释的是“栈为什么这样组织”，不是“为什么首帧这么晚” |

## 5. 常见误解与边界 (Mistakes & Boundaries)
### 常见误解
- 只要返回路径怪，就一把梭改成 `singleTask` 或 `singleTop`。启动模式不是修补按钮，而是导航语义的编码方式；脱离业务入口设计去乱改，通常只是把问题从“重复实例”换成“历史被莫名清空”。

### 失效 / 反噬信号
- 页面仍在重复创建、通知二次进入后上下文丢失、返回键直接弹回桌面或错误入口，说明你配置的模式没有匹配真正的进入语义，甚至和 `Intent` flags 互相打架。

### 不适用场景
- 如果问题本质是冷启动慢、主线程阻塞、系统服务卡顿或 Fragment 内部导航混乱，启动模式最多解释实例与任务栈，不会替代性能分析和具体容器内导航排错。

## 6. 与哪些概念容易一起出现 (Nearby Concepts)
- [[Concepts/Binder IPC|Binder IPC]] `(supports[弱])`：Activity 启动会穿过系统服务与跨进程调度链，但 Binder 解释的是“消息如何跨进程到达”，不是“栈为什么这样组织”。
- [[Concepts/Perfetto|Perfetto]] `(observes[中])`：当你怀疑某次跳转既有返回栈异常又有启动时延问题时，Perfetto 负责给时间线证据，帮助区分“语义错了”还是“执行太慢”。
- [[Concepts/InputManagerService|InputManagerService]] `(contrasts[弱])`：IMS 处理用户输入分发，启动模式处理页面实例和任务栈；两者都会被用户感知为“页面行为不对”，但排查抓手完全不同。

## 7. 来源对照 (Source Cross-check)
- **来源 1** ([[2026-02-22-Android-启动模式详细解析]])：强调启动模式的核心不是参数表，而是“系统到底创建新实例还是复用旧实例”，把注意力拉回任务栈本质。
- **我的整合结论**：真正应该记住的是“入口语义 -> 实例复用决策 -> 任务栈结果”这条链。只要问题表现为多入口下的返回路径不稳定，优先把它当成启动模式问题，而不是先怪 UI。

## 8. 自测问题 (Self-Test)
### 记忆锚点 / Memory Anchor
- **一句话记住**：启动模式不是“怎么打开页面”，而是“这次打开会怎样改写任务栈历史”。
- **看到这个信号就想到它**：同一页面被多个入口反复打开、返回路径不稳定、通知/深链接进入后上下文错乱。

### 自测问题
1. 为什么说启动模式本质上管理的是“任务栈语义”，而不只是“页面实例个数”？
2. 当列表入口和通知入口同时指向同一详情页时，你会优先检查哪类复用 / 回栈预期？
3. 如果页面打开得很慢但返回栈完全正确，为什么不该先把问题归到启动模式上？
