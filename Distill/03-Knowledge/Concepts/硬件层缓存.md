# 硬件层缓存

## 1. 它是什么 (What It Is)
### 快速判断 / Quick Scan
| 维度 | 内容 |
| --- | --- |
| 一句话定义 | 硬件层缓存是把某个 View / 子树的绘制结果预先缓存成 GPU 图层或纹理，让后续多帧只做复用和变换，而不是每帧重新录制与重绘。 |
| 不要和什么混为一谈 | 它不是“打开必快”的万能优化，也不是普通图片内存缓存；它优化的是渲染阶段重复工作，不是业务数据加载。 |
| 什么时候想到它 | 复杂 View 在短时间内只做平移、缩放、旋转、alpha 变化，内容几乎不变时，就该考虑硬件层缓存。 |

### 展开理解
- 它本质是一笔“先花钱、后摊销”的缓存交易：先付出建层、显存占用、纹理上传的代价，再用多帧复用去回本。
- 所以它只适合“内容稳定 + 复用次数足够多”的场景。只要每帧都 invalidate，缓存就会从加速器变成负担。

## 2. 为什么它重要 (Why It Matters)
### 它解决的判断 / 工程问题
- 它解决“复杂内容明明不变，为什么还要每帧重画”的问题。对于 transform / alpha 类动画，真正变化的常常只是矩阵和透明度，不是内容本身。
- 它也帮助你把优化思路从“继续压榨每帧绘制”转成“能否用一次建层换多帧复用”，这是渲染优化里非常典型的缓存思维。

### 如果忽略它会怎样
- 你可能让一个复杂 View 在动画期每帧都重新录制和重绘，白白消耗 UI 线程与 RenderThread 预算。
- 但如果误用它，你又会因为频繁建层、显存占用和纹理上传，把原本能忍的动画做得更差；所以忽略“适用边界”同样危险。

## 3. 它是怎么工作的 (How It Works)
### 机制链 / Mechanism Chain
1. 系统或开发者先把目标 View 提升为 hardware layer，把当前子树的绘制结果录入离屏 layer / texture。
2. 后续帧如果只变化位置、缩放或 alpha，RenderThread 可以直接复用这层缓存，跳过大量子树重录和重绘。
3. 一旦子 View 内容变化、属性触发 invalidate，layer 就需要重建，建层成本与纹理上传会重新发生。
4. layer 越大、设备显存越紧、内容变化越频繁，重建与回收的代价就越高。
5. 所以它真正的收益，取决于“稳定复用帧数能否覆盖建层成本”；这本质上是缓存命中率思维，而不是开关思维。

### 排查 / 应用抓手 (Diagnostic / Application Hooks)
- **排查时先问**：这个 layer 在动画期间真的稳定吗，还是每隔一两帧就被 invalidate 重建？大小是否已经大到纹理上传本身就很贵？
- **设计时先问**：这个元素的变化是内容变化，还是仅仅变换矩阵变化？如果内容常变，是否应该取消 layer、改用更轻的绘制策略？
- **常见观察信号**：`buildLayer` 频繁出现、RenderThread 上传脉冲、低端机显存压力升高、动画期仍持续 redraw / invalidate，都是硬件层缓存失效的信号。

## 4. 一个最小例子 / 对比 (Minimal Example / Contrast)
### 最小例子
- **场景**：一个复杂营销卡片只做平移和透明度动画，卡片内部的图片、文案和阴影在 300ms 内基本不变。
- **为什么这里会想到它**：这类场景最怕每帧都重走整棵 View 树；如果先建一层缓存，后续帧往往只需复用纹理并更新变换。
- **结果**：当内容稳定时，硬件层缓存能明显减轻绘制与提交压力；但如果卡片里有倒计时文本每帧更新，缓存就会频繁失效，收益很快被吃掉。

### 对比
| 易混概念 / 做法 | 真正差异 | 这里为什么不是它 |
| --- | --- | --- |
| 普通重绘 | 每一帧都重新录制和绘制内容 | 这里真正想利用的是“内容没变”，所以应该优先复用而不是重复准备 |
| 图片内存缓存 | 优化的是资源读取与解码 | 硬件层缓存优化的是 View 子树的渲染复用，不等于图片已缓存就自动解决渲染成本 |
| 一切动画都开 layer | 更像粗暴套用技巧 | 如果内容持续变化，layer 会不断重建，反而可能比普通绘制更差 |

## 5. 常见误解与边界 (Mistakes & Boundaries)
### 常见误解
- “硬件层缓存是无脑加速项。”实际上，它是一种严格依赖场景的缓存策略，命中率不够时就是额外负担。
- “只要感觉卡，就先开 layer。”如果不先判断内容稳定性、layer 大小和动画时长，这种优化很容易南辕北辙。

### 失效 / 反噬信号
- `buildLayer` 频繁出现、动画期仍不断 invalidate、显存压力增大或低端机帧时间更差，都是缓存没有回本的典型症状。
- 你本来想减少重绘，结果却引入更多上传与回收开销，这说明问题不在“有没有缓存”，而在“缓存能不能稳定复用”。

### 不适用场景
- 文本、列表内容、动态图片等持续变化的区域，通常不适合长期依赖硬件层缓存。
- 如果卡顿根因是输入根本没到、业务逻辑阻塞或网络等待，硬件层缓存也帮不上忙。

## 8. 自测问题 (Self-Test)
### 记忆锚点 / Memory Anchor
- **一句话记住**：硬件层缓存像“先截图再搬运”；如果这张图每一帧都得重拍，就不如直接现场重画。
- **看到这个信号就想到它**：复杂 View 只做 transform / alpha 动画、`buildLayer` 与 invalidate 频繁、开 layer 后低端机反而更差。
