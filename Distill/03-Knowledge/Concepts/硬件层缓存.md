# 硬件层缓存

## 硬件层缓存真正做的，不是无脑加速，而是用一次建层换多帧复用
### 一眼认知骨架
- **对象**：View / 子树的 GPU 图层缓存，不是普通图片内存缓存。
- **目的**：在内容基本不变的多帧动画里，避免每帧重新录制和重绘整棵子树。
- **组成**：`setLayerType(View.LAYER_TYPE_HARDWARE, null)`、`RenderNode`、离屏 layer / FBO、纹理上传、invalidate / buildLayer 失效链。
- **主线**：提升为硬件层 → 首次建层并缓存纹理 → 后续帧复用做 transform / alpha → 内容变化触发失效 → 重新建层或回收。
- **变体**：手动开层 / 框架自动建层、稳定内容 / 高频变化内容、大 layer / 小 layer、低端机显存紧张都会改写收益。
- **用法**：最适合解释复杂 View 的短时平移、缩放、旋转、alpha 动画，以及 buildLayer / 上传脉冲造成的反效果。
### 快速判断 / Quick Scan
- 如果只用一句话说清，硬件层缓存像“先截图再搬运”：图不变时赚，图每帧都重拍时就亏。
- 你最容易把它和“开启硬件加速”混为一谈；但真正的关键不是有没有 GPU，而是能不能稳定复用那层缓存。
- 一旦你开始追 `buildLayer`、纹理上传、显存压力、动画期 invalidate 或“开 layer 后低端机反而更差”，就已经进入它的适用边界。
### 展开理解
硬件层缓存本质上是一笔“先花钱、后摊销”的缓存交易。它先让系统为某棵 View 子树建一层离屏缓存，之后多帧只复用这张纹理并做矩阵 / 透明度变化，于是大量重绘成本就被省掉了。但只要内容持续变化、layer 太大、上传太贵，缓存命中率就会迅速下降，原本想省下来的成本反而以 buildLayer、纹理上传和显存占用的形式回来报复你。

## 系统宁可为稳定动画付出建层与显存成本，也不愿每帧重录整棵子树
### 它解决的判断 / 工程问题
硬件层缓存要解决的是：**当某个复杂 View 在动画期间真正变化的只是位置、缩放、旋转或 alpha，而内容本身基本不变时，系统能否用一次建层换取后续多帧复用，从而避免每帧重复录制整棵子树。** 这是一类典型的“用缓存换吞吐”的问题。
### 如果忽略它会怎样
如果忽略它，你可能让复杂 View 在 300ms 动画期间每一帧都重新录制、重绘、提交，白白消耗 UI 线程和 RenderThread 预算；但如果误用它，又会把本来可以接受的动画做得更差，因为频繁建层、上传和回收会形成新的脉冲开销。真正危险的，不是“没开 layer”，而是**不知道什么时候开、什么时候关、什么时候该放弃**。
### 为什么系统宁可这样设计 (Design Rationale / Trade-off)
系统没有对所有动画都默认长期缓存，而是把 layer 做成一项显式的场景化策略。这样换来的好处是：对于稳定 transform / alpha 动画，能显著减少重复工作；代价则是首次建层、纹理上传、显存占用和缓存失效管理都要真实支付。也就是说，系统宁可把它做成一项需要判断的 trade-off，也不愿把“缓存一定更快”这种危险假设写死在渲染模型里。

## 真正决定收益的是 LayerType、FBO / 纹理、invalidate 和 rebuild 这条链
### 机制链 / Mechanism Chain
1. 开发者或框架先把目标 View 提升为 hardware layer，典型入口就是 `setLayerType(View.LAYER_TYPE_HARDWARE, null)` 或动画期的自动建层。
2. 首次绘制时，这棵子树会被录进离屏 layer / FBO，并形成一张可由 GPU 复用的纹理；这一步常伴随 `buildLayer` 与上传成本。
3. 后续帧如果只变化矩阵或 alpha，RenderThread / GPU 可以直接复用这张纹理，不必每帧重新录制整棵子树。
4. 一旦子 View 内容变化、文本更新、图片替换、阴影重新计算或 invalidate 触发脏区，layer 就会失效，需要重新建层和重新上传。
5. 动画结束或策略取消时，这层缓存会被释放或回收；如果长期保留却继续频繁失效，就会把“优化项”变成显存与上传负担。
### 关键条件 / 分支 / 例外 (Critical Conditions / Exceptions)
- **内容稳定分支**：只有当动画期间内容大体稳定，transform / alpha 成为主要变化时，layer 才更容易回本。
- **大纹理分支**：layer 越大、设备显存越紧张，首次上传与重复重建的代价就越高；低端机尤其容易被反噬。
- **invalidate 分支**：哪怕只有一小块内容频繁变化，也可能把整层缓存逼回 rebuild 路径；“有缓存”不等于“就不会重画”。
- **问题边界例外**：硬件层缓存只能减少某类渲染工作，不能替你解决输入根本没到、网络等待或业务逻辑阻塞。
### 最低决定层 / 关键锚点 (Decisive Layer Anchors)
- 真正决定收益的，不是“开没开 layer”，而是 `buildLayer → 纹理 / FBO → 复用 → invalidate → rebuild` 这条链。只盯开关，你会把缓存命中和失效的差别全看丢。
- `buildLayer`、纹理上传和显存压力是最关键的第一证据；如果只看 FPS，不看这些脉冲，几乎一定会把问题看歪。
- layer 成本大多落在 RenderThread / GPU 路径上，而不是只体现在 UI 线程；忽略这层，你会把“后半程更重了”误判成“主线程没优化好”。
- layer 的生命周期本身就是 trade-off：动画结束不及时关、长期保留一个经常失效的大 layer，很可能把一次局部优化换成全局内存压力。
### 排查 / 应用抓手 (Diagnostic / Application Hooks)
- 做对比时别只看“开 / 关”两帧的 FPS，至少同时观察 `buildLayer`、上传脉冲、invalidate 频率和动画结束后的资源释放时机。
- 硬件层更适合短时、稳定、transform / alpha 为主的动画；如果内容每帧都变，应优先考虑重新设计动画或回到普通重绘。
- 如果开层后低端机反而更差，先怀疑大 layer、显存紧张和频繁 rebuild，而不是先怀疑“GPU 不够快”。

## 一张复杂卡片做平移和 alpha 动画时，缓存为什么有时赚、有时赔
### 最小例子
- **场景**：一个复杂营销卡片只做平移和透明度动画，卡片内部的图片、文案和阴影在 300ms 内基本不变。
- **为什么这里会想到它**：这类场景最怕每帧都重走整棵 View 树；如果先建一层缓存，后续帧往往只需复用纹理并更新变换。
- **结果**：当内容稳定时，硬件层缓存能明显减轻绘制与提交压力；但如果卡片里有倒计时文本或频繁刷新图片，这层缓存会不断失效，收益很快就被 rebuild 和上传吃掉。
### 对比
- **普通重绘**：每帧都重新录制和绘制内容；优点是简单直接，缺点是复杂子树动画时重复工作太多。
- **图片内存缓存**：优化的是资源读取与解码；硬件层缓存优化的是整个 View 子树的渲染复用，两者不是一回事。
- **“所有动画都开 layer”**：更像粗暴套技巧；一旦内容持续变化，layer 很容易从优化项变成拖累项。

## 把 layer 当成万能开关，最容易在低端机和动态内容上反噬
### 常见误解
- “硬件层缓存是无脑加速项。” 实际上，它是一项严格依赖命中率的缓存策略，命中率不够时就是额外负担。
- “只要感觉卡，就先开 layer。” 如果不先判断内容稳定性、layer 大小和动画时长，这类优化很容易南辕北辙。
### 失效 / 反噬信号
- `buildLayer` 频繁出现、动画期仍不断 invalidate、显存压力增大、低端机帧时间更差，都是缓存没有回本的典型症状。
- 你本来想减少重绘，结果却引入更多上传与回收开销，这说明问题不在“有没有缓存”，而在“缓存能不能稳定复用”。
### 不适用场景
- 文本、列表内容、动态图片等持续变化的区域，通常不适合长期依赖硬件层缓存。
- 如果卡顿根因是输入根本没到、业务逻辑阻塞或网络等待，硬件层缓存也帮不上忙。

## 硬件层缓存通常要和这些概念一起看
- [[Concepts/RenderThread|RenderThread]] `(depends[强])`：硬件层缓存的收益最终要通过 RenderThread 对图层与纹理的复用体现出来；失效成本也会沿着同一条提交链放大。
- [[Concepts/Perfetto|Perfetto]] `(observes[中])`：当你怀疑 buildLayer、上传或 layer 失效造成帧时间抖动时，Perfetto 是最直观的时间线证据。
- [[Concepts/InputManagerService|InputManagerService]] `(contrasts[弱])`：输入及时并不保证画面流畅；用户觉得“不跟手”时，既可能是事件晚到，也可能是缓存策略让渲染晚到了。

## 记住“先截图再搬运，重拍太勤就亏”，再做三个自测
### 记忆锚点 / Memory Anchor
- **一句话记住**：硬件层缓存像“先截图再搬运”；如果这张图每一帧都得重拍，就不如直接现场重画。
- **看到这个信号就想到它**：复杂 View 只做 transform / alpha 动画、`buildLayer` 与 invalidate 频繁、开 layer 后低端机反而更差。
