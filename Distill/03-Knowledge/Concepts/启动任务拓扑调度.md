# 启动任务拓扑调度

## 启动拓扑调度真正管理的，不是顺序表，而是谁此刻已经 ready 可以执行
### 一眼认知骨架
- **对象**：应用初始化任务的 DAG 调度器，不是一次性的排序结果列表。
- **目的**：在依赖复杂但又想并行缩短启动关键路径时，持续判断“谁现在可以执行”。
- **组成**：任务节点、依赖边、入度表、邻接表、`START / END` 虚拟节点、完成事件回调、线程执行策略。
- **主线**：建 DAG → 初始化入度 / 邻接表 → 触发 ready 任务 → 任务完成后解锁后继 → `END` ready 表示整批完成。
- **变体**：主线程任务 / 后台任务、环依赖、漏掉完成回调、理论可并行却实际仍串行，都会改写效果。
- **用法**：最适合解释冷启动初始化很多、依赖交错、关键路径过长、任务理论能并行却实际跑不起来的场景。
### 快速判断 / Quick Scan
- 如果只用一句话说清，拓扑调度不是“先生成一个 A→B→C 列表”，而是“持续维护 ready 集，谁入度归零谁立刻可跑”。
- 你最容易把它和普通线程池并发混为一谈；但线程池只会并发，不懂依赖，DAG 才能保证“并行但不乱序”。
- 一旦你开始追“为什么启动链这么长”“理论上可并行的任务为什么还在排队”“哪个任务卡住了后继一大片”，就该想到它。
### 展开理解
很多启动框架表面写着“拓扑排序”，实际却只是先排一张执行顺序表再照着串行跑。这种做法把真正有价值的那部分——**ready 判断与完成事件驱动**——浪费掉了。拓扑调度的工程价值，不在算法题本身，而在它能把初始化从“人肉维护顺序与回调”升级成“可观测、可并行、可解释关键路径”的运行时系统。

## 框架宁可维护 DAG、入度表和完成事件，也不想把启动逻辑写成一串脆弱回调
### 它解决的判断 / 工程问题
拓扑调度要解决的是：**初始化任务既有依赖关系，又不想全部串行压在主线程时，系统怎样以低成本判断谁现在可并行、谁必须继续等前置完成。** 如果没有显式依赖模型，启动治理很快就会退化成一堆手工 if / else、回调嵌套和隐式时序约定。
### 如果忽略它会怎样
如果忽略这层结构，你会越来越难解释启动为什么慢：到底是某个任务本身变重了、关键路径太长了，还是本来可以并行的任务被错误串行化了。任务一多，所谓“顺序初始化”就会演变成维护噩梦，任何一个依赖修改都可能悄悄把首屏关键路径拉长。
### 为什么系统宁可这样设计 (Design Rationale / Trade-off)
框架没有把初始化做成“谁先谁后写死在代码里”，而是宁可多维护一张 DAG 和两张表。这样换来的好处是：ready 判断可 O(1) 完成、完成事件能即时解锁后继、关键路径可被显式观察和优化；代价则是要管理图结构、循环依赖、任务线程亲和性以及完成回调的可靠性。也就是说，框架宁可承担建模成本，也不愿长期维护一团脆弱的顺序代码泥球。

## 真正决定并行度的是入度、邻接表、完成回调和关键路径，而不是“有没有线程池”
### 机制链 / Mechanism Chain
1. 先把启动任务和依赖关系建成一张 DAG：节点是任务，边表示“from 完成后 to 才能开始”；必要时可加 `START / END` 虚拟节点统一入口和收口。
2. 初始化两份核心数据结构：入度表负责 O(1) 判断任务是否 ready，邻接表负责在某个任务完成后快速找到它解锁了谁。
3. 执行开始时，触发 `START` 或所有入度为 0 的任务；真正的执行可以落到线程池，也可以回到主线程，但“能不能跑”先由 DAG 决定。
4. 某个任务完成后触发 `onTaskCompleted(task)`：遍历它的后继任务、把后继入度减一；谁的入度归零，谁就立刻进入 ready 集。
5. 当 `END` 就绪或所有任务都完成时，整批初始化结束；真正决定启动时长的，通常是最长依赖链，也就是关键路径，而不是任务总数本身。
### 关键条件 / 分支 / 例外 (Critical Conditions / Exceptions)
- **环依赖分支**：只要图里出现环，某些任务就永远不可能 ready；这不是“调度器慢”，而是模型本身已经错误。
- **完成回调分支**：图建得再漂亮，任务若没正确回调 `onTaskCompleted()`，后继任务照样永远解锁不了。
- **线程亲和分支**：DAG 只负责依赖，不自动解决主线程亲和性；某些任务虽然 ready，也只能在主线程上执行。
- **小规模例外**：任务很少、依赖很简单时，强上 DAG 可能只是徒增复杂度；是否值得建模，要看规模与收益。
### 最低决定层 / 关键锚点 (Decisive Layer Anchors)
- 真正决定调度结果的，不是最终排出来的顺序，而是“当前 ready 集”。如果把 ready 集看丢，拓扑调度就会被误学成一张静态列表。
- 入度表和邻接表才是工程实现的核心：前者让 ready 判断 O(1)，后者让“一个任务完成后谁被解锁”变成 O(out-degree)；只有顺序表，没有这些结构，基本就丢掉了并行收益。
- `onTaskCompleted()` 是真正的解锁边界。图纸上写着依赖关系并不等于运行时真的会发生推进；少了完成事件闭环，调度器只剩摆设。
- 线程池不等于调度器：线程池只能负责“有人来执行”，却不知道依赖何时满足；如果把两者混为一谈，就会把乱序和阻塞都看歪。
### 排查 / 应用抓手 (Diagnostic / Application Hooks)
- 启动慢时先画关键路径：别只问“哪个任务最慢”，还要问“哪个依赖链把首屏拖长了”。
- 验证每个任务的完成回调是否真实闭环，尤其是异常 / 取消路径；很多框架卡死，问题不是线程池没跑，而是后继从未被解锁。
- 用 [[Concepts/Perfetto|Perfetto]] 或自定义 trace 验证设计有没有落地：理论上可并行的任务如果在 trace 里仍完全串行，说明 DAG 还停留在图纸上。

## 一组互有依赖的初始化任务，怎么从串行泥球变成可解释的 ready 集
### 最小例子
- **场景**：启动需要完成“日志初始化、配置拉取、AB 实验、网络栈准备、首屏路由准备”五类任务，其中配置又是其中几项任务的前置依赖。
- **为什么这里会想到它**：如果全部串行，主线程会被白白拉长；如果没有依赖建模，又可能让首屏在关键前置没完成时过早运行。
- **结果**：用拓扑调度后，无依赖任务可以先并行，等配置任务完成后再解锁依赖它的后继；系统真正关心的是“当前谁 ready 了”，而不是一张死板的执行顺序表。
### 对比
- **手工串行初始化**：简单但把所有任务压成一条长链，会把本来可并行的任务白白串行化。
- **一次性 Kahn 结果列表**：先求顺序再执行，容易忽略运行时 ready 判断和完成事件驱动，工程收益会被削弱。
- **纯线程池并发**：只会并发，不懂依赖；没有 DAG 约束就很难保证正确顺序，也看不见关键路径。

## 把拓扑排序学成一张静态列表，通常会让框架只剩形式没有收益
### 常见误解
- “拓扑排序就是得到一个执行顺序列表。” 这是最常见的看歪；真正落地时，更重要的是入度表、邻接表和完成事件驱动。
- “只要上了 DAG，就一定更快。” 如果关键路径没变、主线程任务没搬走、依赖建模还错了，框架只会更复杂，不会自动更快。
### 失效 / 反噬信号
- 图里出现环依赖、任务完成后后继始终不 ready、理论可并行但 trace 完全串行，都是拓扑调度没有真正成立的信号。
- 依赖关系越来越多却没有分层治理，DAG 退化成“谁都依赖谁”的大型泥球，也会让调度收益快速消失。
### 不适用场景
- 如果初始化规模很小、几乎没有依赖，手工顺序执行可能更清晰，没必要为了“高级感”硬上 DAG。
- 如果任务依赖高度动态且运行中不断变化，静态拓扑图只能解决部分问题，仍需要状态机、重试和熔断等更高层控制。

## 启动任务拓扑调度要和这些概念连着看
- [[Concepts/ActivityManagerService|ActivityManagerService]] `(context[中])`：AMS 解决系统如何把 app 真正拉起来，拓扑调度解决 app 进程内部如何不把初始化关键路径继续拉长，两条链叠加决定最终启动体验。
- [[Concepts/Handler 消息机制|Handler 消息机制]] `(execution[中])`：很多 ready 任务最后仍要切回主线程执行，真正落地时必须和 Looper / MessageQueue 的节奏协作。
- [[Concepts/Perfetto|Perfetto]] `(observes[强])`：想证明 DAG 真的缩短了关键路径，最可靠的方法不是猜，而是看 trace 里的并行度和等待段。

## 记住“维护 ready 集，不是生成顺序表”，再做三个自测
### 记忆锚点 / Memory Anchor
- **一句话记住**：拓扑调度 = 用 DAG 持续维护 ready 集，而不是先写一张固定顺序表。
- **看到这个信号就想到它**：初始化任务越来越多、依赖交错、理论上可并行却实际没并起来、关键路径说不清为什么这么长。
