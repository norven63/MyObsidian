# 启动任务拓扑调度

## 1. 它是什么 (What It Is)
### 快速判断 / Quick Scan
| 维度 | 内容 |
| --- | --- |
| 一句话定义 | 启动任务拓扑调度是用 DAG + 入度表 + 邻接表来组织应用初始化任务的事件驱动编排方式。 |
| 不要和什么混为一谈 | 它不只是“先算出一个执行列表再照着跑”；更关键的是在运行时用完成事件持续触发后继任务，而不是把拓扑排序学成一次性的静态结果。 |
| 什么时候想到它 | 应用冷启动初始化很多、任务有依赖又想并行、主线程初始化被挤爆、需要回答“关键路径在哪儿”时。 |

### 展开理解
- 这类问题的本质不是“先后顺序写清楚没有”，而是“任务依赖如何被结构化表达，哪些任务现在已经 ready，哪些任务还必须等前置完成”。
- 一旦理解成事件驱动编排，你就不会再执着于生成一张长长的执行序列，而会更关心就绪条件、完成回调和关键路径。

## 2. 为什么它重要 (Why It Matters)
### 它解决的判断 / 工程问题
- 它解决的是：**初始化任务既有依赖关系，又不想全部串行阻塞主线程时，系统应该怎样低成本判断谁能并行、谁必须等待。**
- 用入度表和邻接表后，“一个任务完成后谁会被解锁”可以做到 O(1)~O(out-degree) 地推进，而不是每次重新扫描全部依赖。

### 如果忽略它会怎样
- 你会把启动治理写成一堆手工 if / else、显式顺序调用或脆弱的回调嵌套；任务一多，复杂度和维护成本会迅速失控。
- 你也很难解释为什么某次启动突然变慢：到底是单个任务变慢了，还是关键路径过长，还是本来可并行的任务被错误串行化了。

## 3. 它是怎么工作的 (How It Works)
### 机制链 / Mechanism Chain
1. 先把每个初始化任务和依赖关系建成一张 DAG：节点是任务，边表示“from 完成后 to 才能开始”。
2. 运行时维护两份核心数据结构：入度表用来 O(1) 判断任务是否 ready，邻接表用来在某个任务完成后快速找到它解锁了谁。
3. 常见实现会引入 `START` / `END` 这样的虚拟节点：开始时只需触发 `START`，完成时只要等到 `END` ready，就知道整批任务结束。
4. 某个任务完成后触发 `onTaskCompleted(task)`，系统遍历它的后继任务，把这些后继的入度减一；谁的入度归零，谁就可以立刻进入执行。
5. 真正的收益来自“边执行边排序”：不需要预先生成完整顺序表，而是在每次完成事件到来时动态释放下一个 ready 集合，并配合线程池或主线程策略执行。

### 排查 / 应用抓手 (Diagnostic / Application Hooks)
- **先画关键路径**：启动慢时别只问“哪个任务最慢”，还要问“哪个任务把最长依赖链拖长了”，这通常比单看平均耗时更有价值。
- **检查完成事件是否真实闭环**：很多框架表面上有 DAG，实际上某个任务没有正确回调 `onTaskCompleted`，后继任务永远不会 ready。
- **用 tracing 验证设计是不是落地了**：理论上可并行的任务如果在 trace 里仍完全串行，说明你的拓扑编排只写在图纸上，没有真正进入执行层。

## 4. 一个最小例子 / 对比 (Minimal Example / Contrast)
### 最小例子
- **场景**：应用启动需要完成“日志初始化、配置拉取、AB 实验、网络栈准备、首屏路由准备”五类任务，其中部分任务互相独立，部分任务必须等待配置完成。
- **为什么这里会想到它**：如果全部串行，主线程会被拉长；如果没有依赖建模，又可能让首屏在关键前置没完成时提早运行。
- **结果**：用拓扑调度后，你可以让无依赖任务先并行，等配置任务完成后再释放依赖它的后继；系统真正关注的是“当前 ready 集”而不是一张死板列表。

### 对比
| 易混概念 / 做法 | 真正差异 | 这里为什么不是它 |
| --- | --- | --- |
| 手工串行初始化 | 简单但把所有任务压成一条长链 | 会把本来可并行的任务白白串行化，关键路径越来越长 |
| 一次性 Kahn 结果列表 | 先求序列再执行 | 序列只是中间产物，真正运行时更需要的是 ready 判断和完成事件驱动 |
| 纯线程池并发 | 只会并发，不懂依赖 | 没有 DAG 约束就无法保证正确顺序，也看不见关键路径 |

## 5. 常见误解与边界 (Mistakes & Boundaries)
### 常见误解
- “拓扑排序就是得到一个执行顺序列表。” 真正落地时，更重要的是入度表、邻接表和完成事件驱动。
- “只要上了 DAG，就一定更快。” 如果关键路径没变、主线程任务没搬走、依赖建模还错了，框架只会更复杂，不会自动更快。

### 失效 / 反噬信号
- 任务图里出现环依赖、任务完成后后继始终不 ready、理论可并行但 trace 完全串行，都是拓扑调度没有真正成立的信号。
- 依赖关系越来越多却没有分层治理，也会让 DAG 退化成“谁都依赖谁”的大型泥球。

### 不适用场景
- 如果初始化规模很小、几乎没有依赖，手工顺序执行可能更清晰，没必要为了“高级感”硬上 DAG。
- 如果任务依赖高度动态且运行中不断变化，静态拓扑图只能解决部分问题，仍需要状态机、重试和熔断等更高层控制。
