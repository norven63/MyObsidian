# Agent Skills

## 1. 它是什么 (What It Is)
### 快速判断 / Quick Scan
| 维度 | 内容 |
| --- | --- |
| 一句话定义 | Agent Skills 是以文件系统为载体的模块化能力包，用元数据、文档、模板和脚本把稳定知识与工作流封装成可按需加载的能力。 |
| 不要和什么混为一谈 | Skill 不是动态工具协议，也不是一次性 prompt 片段；它更像长期维护的“培训手册 + 操作骨架”，强调复用与沉淀。 |
| 什么时候想到它 | 某类任务会反复出现、解释成本高、规则相对稳定，而且不需要持续实时连接外部系统时，优先想到 Skill。 |

### 展开理解
- Skill 的价值，不在“多了一个目录”，而在把组织经验从主对话里拆出来，变成可维护、可组合、可复用的能力资产。这样主 Agent 只保留判断与协调，而不必每次重新背整套背景。

## 2. 为什么它重要 (Why It Matters)
### 它解决的判断 / 工程问题
- 它解决的是“高复用知识和固定流程该放在哪里”的问题：如果每次都把业务背景、最佳实践和操作步骤塞进主 Prompt，结果一定是上下文膨胀、能力难维护、更新影响面极大。
- 它还帮助团队把“这类事长期应该怎么做”固定下来，让不同 Agent 在同一套知识和操作骨架上工作，而不是靠临场发挥。

### 如果忽略它会怎样
- 组织经验会继续散落在历史对话和个人记忆里；每次重做相似任务都要重新解释背景，主 Agent 的上下文会越来越胖，而真正稳定的知识却越来越难升级。

## 3. 它是怎么工作的 (How It Works)
### 机制链 / Mechanism Chain
1. 先把稳定的领域知识、步骤约束、模板和脚本收敛成一个独立目录，而不是继续堆在系统 prompt 里。
2. 再用元数据描述 Skill 的适用场景、入口命令和依赖资源，让 Agent 在合适时按需加载，而不是全量常驻上下文。
3. 运行时会把 Skill 提供的说明、模板和脚本，与 MCP / Harness 提供的工具和执行边界拼起来：Skill 负责教“为什么这样做、按什么顺序做”，运行时负责真正执行。
4. 当流程变化时，只更新对应 Skill 即可，主 Agent 不需要重新学习整个领域背景；组织经验也就从“模型偶尔记住了什么”变成“仓库里明确存在什么能力包”。

### 排查 / 应用抓手 (Diagnostic / Application Hooks)
- **排查能力沉淀方式时先问**：这项能力是不是反复出现、规则稳定、解释成本高？如果是，优先考虑 Skill，而不是继续往主 Prompt 里堆说明。
- **设计边界时先问**：这里需要的是静态知识和流程骨架，还是实时连接、会话状态和标准化协议？前者更像 Skill，后者更像 MCP / 服务。
- **常见观察信号**：每做一次相似任务都要重新解释背景、主 Prompt 越来越长、改一个流程会波及大量无关能力，通常说明本该沉淀成 Skill 的东西还在裸奔。

## 4. 一个最小例子 / 对比 (Minimal Example / Contrast)
### 最小例子
- **场景**：把“部署到 Railway 的标准流程”做成 Skill，目录里放好环境变量约束、回滚步骤、排障清单和执行脚本。
- **为什么这里会想到它**：部署问题会反复出现，但规则相对稳定，最浪费的不是执行动作本身，而是每次都要重新解释同一套背景。
- **结果**：Agent 遇到部署任务时，会先加载这套稳定知识和脚本骨架，而不是每次重新向用户追问整套流程。

### 对比
| 易混对象 | 真正差异 | 这里为什么不是它 |
| --- | --- | --- |
| [[Concepts/MCP]] | MCP 解决“怎么标准化连接工具与数据源” | 这里主要难点是复用知识与流程，而不是工具接入协议本身 |
| [[Concepts/渐进式 Spec]] | Spec 解决“这次任务要怎么做” | Skill 解决的是“这类事长期都应该怎么做”，偏可复用能力而非单次契约 |

## 5. 常见误解与边界 (Mistakes & Boundaries)
### 常见误解
- 把所有工具接入、权限管理、长连接状态都塞进 Skill。这样会让 Skill 从知识封装退化成难维护的小平台，结果是边界越来越糊，复用性反而下降。

### 失效 / 反噬信号
- 如果 Skill 一改就牵动大量无关能力、Agent 每次都得全量加载所有 Skill、或者 Skill 内部开始堆凭证与会话状态，说明它已经越界，不再只是知识与流程骨架。

### 不适用场景
- 依赖实时会话状态、频繁双向 I/O、需要标准协议协商或长期在线连接的能力，更适合放进 MCP 或专用服务，而不是继续硬塞给 Skill。

## 6. 与哪些概念容易一起出现 (Nearby Concepts)
- [[Concepts/Harness Engineering|Harness Engineering]] `(depends[中])`：Harness 决定 Skill 如何被发现、加载、执行和验证；没有运行时壳，Skill 很难稳定发挥作用。
- [[Concepts/MCP|MCP]] `(complements[强])`：Skill 负责说明“为什么这样做、什么时候这样做”，MCP 负责把“怎么连工具”标准化；两者刚好把知识层和工具层拆开。
- [[Concepts/渐进式 Spec|渐进式 Spec]] `(contrasts[中])`：Spec 面向单次任务契约，Skill 面向重复能力沉淀；前者帮你把这次任务说清楚，后者帮你把这类任务长期做对。

## 7. 来源对照 (Source Cross-check)
- **来源 1** ([[2026-03-30-Agent-到-Skills-范式转变]])：强调 Skill 更像员工培训手册，用渐进式披露把高复用知识从主上下文里拆出来。
- **来源 2** ([[2026-03-25-DeerFlow-超级AI员工框架]])：补充了在多角色 / 多 Agent 系统里，Skill 如何成为共享的组织记忆，而不是只服务单个对话。
- **我的整合结论**：Skill 的真正价值，是把“模型偶尔会做”变成“团队明确拥有这项能力”。它不是为了多一个目录，而是为了让知识和流程能像代码一样被维护、被替换、被复用。

## 8. 自测问题 (Self-Test)
### 记忆锚点 / Memory Anchor
- **一句话记住**：Skill 是把高复用知识与流程封成能力包；一旦问题开始依赖实时连接或会话状态，就要警惕它已经不是 Skill 了。
- **看到这个信号就想到它**：同类任务反复出现、背景说明总在重复、主 Prompt 越来越长而且难维护。

### 自测问题
1. 什么样的知识最适合沉淀成 Skill，而不是继续写在单次 prompt 里？
2. 为什么说“需要实时连接外部系统”往往是在提醒你去想 MCP，而不是继续扩 Skill？
3. 如果一个 Skill 一改就牵动大量无关能力，这通常暴露了什么边界问题？
