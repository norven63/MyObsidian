# MCP

## 先把MCP放进脑子里
### 一眼认知骨架
- **对象**：MCP 是一个标准化工具连接协议，用统一的 Tools、Resources、Prompts 把 Agent 与外部工具、数据源和服务对接起来，减少每个 Agent 单独硬编码集成的成本。
- **目的**：它解决的是“哪些外部能力应该被标准化成平台接口，而不是继续散落在各个 Agent 私有集成里”的问题。只要一个能力会被多个 Agent 复用，这个判断就变得很重要。
- **组成**：把它拆开看，真正稳定决定结果的是：先把外部能力抽象成协议对象，例如 Tools 负责动作调用、Resources 负责资源…
- **主线**：先把外部能力抽象成协议对象，例如 Tools 负责动… → MCP Server 在协议后面处理认证、参数校验、… → Agent 或 Harness 只面向标准化接口做发… → 底层服务即使独立演进、替换实现或扩展能力，只要协议兼…
- **变体**：边界、分支和失败条件会改写这条主线；后文重点看它最容易被忽略的前提、误判点和不适用场景。
- **用法**：当多个 Agent / 客户端需要稳定连接同一类数据库、文件系统、SaaS API 或内部服务，并且希望调用语义、权限边界和错误模型保持一致时，就该想到 MCP。

### 展开理解
- MCP 的位置更像协议层插座：上层 Agent 面向统一接口发现和调用能力，下层服务把认证、原始 API 细节和资源暴露方式藏在协议后面。
- 它真正减少的不是“写一个函数”的成本，而是随着 Agent 数量增加，重复接入、重复鉴权、重复约定调用语义的系统性成本。

## 为什么不能只把MCP当成一个名词定义
### 它解决的判断 / 工程问题
- 它解决的是“哪些外部能力应该被标准化成平台接口，而不是继续散落在各个 Agent 私有集成里”的问题。只要一个能力会被多个 Agent 复用，这个判断就变得很重要。
- 它也把工具接入从“谁会写胶水代码”升级成“能否被发现、复用、审计和替换”的平台设计问题，让能力面不再绑死在单个运行时实现上。

### 如果忽略它会怎样
- 你会得到一堆看似都能访问外部系统、实际上行为并不一致的 Agent：认证方式不同、错误语义不同、参数格式不同，调试和治理成本会随着接入数线性放大。
- 更常见的后果是，团队每新增一个 Agent 就得重写一遍数据库或 API 适配层，最后真正被复用的不是能力，而是重复劳动。

### 为什么系统宁可这样设计
MCP 被单独做成协议层，是为了让多个 Agent / 客户端共享同一类外部能力时，不必各自重写鉴权、参数格式和错误语义。这样换来的是工具发现、权限边界和能力复用都可以围绕稳定协议治理；代价则是要维护 server 适配层、协议对象和向后兼容，而不是直接把底层 API 暴露给每个 Agent。

## 真正决定MCP结果的，是哪条机制链
### 机制链
1. 先把外部能力抽象成协议对象，例如 Tools 负责动作调用、Resources 负责资源读取、Prompts 负责可复用提示入口，而不是让 Agent 直接碰底层 API。
2. MCP Server 在协议后面处理认证、参数校验、资源发现和错误语义，把原始接口转换成上层更稳定、更可枚举的能力面。
3. Agent 或 Harness 只面向标准化接口做发现与调用，因此不同客户端能在相似的调用心智模型下复用同一能力，而不必各自重学底层系统。
4. 底层服务即使独立演进、替换实现或扩展能力，只要协议兼容，上层 Agent 的接入方式就不必整体重写，能力治理也因此更容易统一。

### 关键条件 / 分支 / 例外
- 只有当能力是 live external capability 且会被多个客户端共享时，协议层收益才显著。
- 静态知识、一次性手工步骤、局部脚本逻辑不该硬做成 MCP，否则只是把简单问题过度协议化。
- 如果 server 仍暴露一堆私有 wrapper 和特殊参数，说明抽象失败，MCP 只是换了个名字的胶水层。

### 最低决定层 / 关键锚点
- 真正的最低层不是“MCP 很统一”这句口号，而是 Tools / Resources / Prompts 三类协议对象如何映射到底层能力。
- 然后看 MCP Server 如何处理 auth、parameter validation、resource discovery 和 error normalization。
- 最后看客户端是否只面向稳定 contract 调用；若还要记住大量私有约定，就不是真正的 MCP 接入。
- 如果把 MCP 误读成“统一工具名”或“再包一层 wrapper”，就会看丢协议对象、server 侧校验和错误归一化这层真正决定接入是否稳定的最低层。

### 排查 / 应用抓手
- **排查时先问**：当前难点到底是“如何接入和治理一个活的外部能力”，还是“如何讲清楚一套知识和流程”？前者更像 MCP，后者通常更像 Skill 或 Spec。
- **设计时先问**：哪些输入输出、权限边界、错误语义和资源发现方式需要对多个 Agent 保持稳定？这些问题正是 MCP 的接口设计重点。
- **常见观察信号**：同一服务被不同 Agent 各写一套 wrapper、鉴权逻辑被拷进脚本和 prompt、同类调用在不同地方参数格式都不一样，通常说明 MCP 层本该存在却还没建立起来。

## 用一个最小场景看清MCP是怎么落地的
### 最小例子
- **场景**：代码分析 Agent 和报表 Agent 都需要查询同一套数据库。直接各写一遍 SQL、认证和错误处理逻辑，短期能跑，长期却会出现行为漂移。
- **为什么这里会想到它**：真正重复的不是“写查询语句”本身，而是围绕数据库访问形成的一整套接入语义：权限、错误、资源暴露方式、调用格式都需要被多个 Agent 共享。
- **结果**：通过 MCP 暴露统一的 `schema` 资源和 `query` 工具后，不同 Agent 面向的是同一能力契约；底层连接池、鉴权方式或错误处理变了，客户端也不必一起推倒重来。

### 对比
| 易混概念 / 做法 | 真正差异 | 这里为什么不是它 |
| --- | --- | --- |
| [[Concepts/Agent Skills]] | Skill 封装知识、步骤和判断提示 | 这里的难点是提供真实可调用的外部能力，而不是解释“为什么这样做” |
| 每个 Agent 自己集成 SDK / API | 一次性接入快，但能力边界和语义容易漂移 | 这里要求多个 Agent 复用同一外部能力，所以更需要稳定协议而不是散装 wrapper |

## 把MCP学歪后，最容易出现哪些误判
### 常见误解
- 为了“统一”，连静态知识、纯说明文档和一次性手工步骤也强行做成 MCP 服务。这样只会把简单问题协议化，徒增维护成本。
- 认为只要叫 MCP Server，就天然具备良好复用性。协议只是外壳，真正决定复用价值的仍是能力抽象是否清晰、权限是否收敛。

### 失效 / 反噬信号
- 如果每个 MCP Server 仍然暴露一堆高度定制、几乎无法复用的接口，那么只是把不一致换了个地方堆放，并没有真正平台化。
- 如果客户端为了调用某个 MCP 能力，仍然必须记住一大堆私有约定、特殊参数和错误分支，说明协议层抽象并没有帮你减负。

### 不适用场景
- 纯静态知识、一次性手工步骤、局部内部函数或不需要动态连接的内容，更适合放进 Skill、模板、脚本或普通代码，而不是硬做协议层。

## MCP总要和这些概念一起看
- [[Concepts/Agent Skills|Agent Skills]] `(complements[强])`：Skill 说明“什么时候、为什么、按什么原则调用”，MCP 提供“实际可以调用什么接口”。
- [[Concepts/Harness Engineering|Harness Engineering]] `(depends[弱])`：Harness 决定 MCP 如何被纳入权限控制、执行日志和恢复流程，从而避免协议层能力失控使用。
- [[Concepts/Sub-Agent 编排|Sub-Agent 编排]] `(contrasts[弱])`：MCP 解决的是工具连接与能力复用，子代理解决的是任务拆解与上下文隔离，两者处在不同层级。

## 先记住MCP的核心判断，再用三个问题检查自己
### 记忆锚点
- **一句话记住**：MCP 管的是外部能力接口，不是知识模板，也不是任务契约。
- **看到这个信号就想到它**：同一服务被多个 Agent 复用，但调用格式、权限和错误语义开始各写各的。

### 自测问题
1. 什么样的能力最适合被做成 MCP，而不是继续写在 Skill、prompt 或私有 SDK 封装里？
2. 为什么把所有静态知识都做成 MCP Server 往往是在错误地协议化问题？
3. 如果一个 MCP 层越来越像胶水代码堆，它最可能暴露了什么抽象失败？
