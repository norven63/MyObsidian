# 渐进式 Spec

## 先把渐进式 Spec放进脑子里
### 一眼认知骨架
- **对象**：渐进式 Spec 是一种按需求复杂度分层暴露流程重量的变更契约：rules 始终生效，Spec / tasks / review / archive 只在复杂度足够高时逐步增强。
- **目的**：它解决的是“这次变更值得多重流程预算”的判断问题。简单需求若被全套流程拖住，效率会迅速流失；复杂需求若继续靠口头对齐，返工和误判就会迅速上升。
- **组成**：把它拆开看，真正稳定决定结果的是：先把所有任务都必须遵守的底线规则和常驻知识固定下来，例如安全边界、仓库约束、命名规范和关…
- **主线**：先把所有任务都必须遵守的底线规则和常驻知识固定下来，… → 再根据变更复杂度、影响面和失败成本判断是否需要升级到… → 复杂度上升后，再逐步引入 Spec、任务拆解、独立评… → 当任务结束后，把新学到的稳定知识反向同步到 rule…
- **变体**：边界、分支和失败条件会改写这条主线；后文重点看它最容易被忽略的前提、误判点和不适用场景。
- **用法**：当任务开始跨模块、验收标准复杂、失败成本上升，或者单靠聊天上下文已经不足以支撑协作与验收时，就该想到更明确的分层契约。

### 展开理解
- 渐进式 Spec 更像一套可伸缩的变更控制系统：底线规则一直在，额外流程只在风险、歧义和协作成本上升时出现。
- 它真正要回答的问题不是“要不要写文档”，而是“为了避免下一次误解，当前最值得增加的那层契约是什么”。

## 为什么不能只把渐进式 Spec当成一个名词定义
### 它解决的判断 / 工程问题
- 它解决的是“这次变更值得多重流程预算”的判断问题。简单需求若被全套流程拖住，效率会迅速流失；复杂需求若继续靠口头对齐，返工和误判就会迅速上升。
- 它还把协作中的关键对象拆开：哪些规则必须常驻，哪些设计取舍需要写进 Spec，哪些高风险任务必须引入独立 review，哪些经验需要回写成长期知识。

### 如果忽略它会怎样
- 团队通常会在两个极端之间摇摆：要么所有任务都被重流程拖慢，要么复杂任务也只靠聊天推进，直到上线前才发现范围、验收和边界根本没对齐。
- 另一个隐性损失是，任务做完后的稳定经验没有被沉淀回规则或知识底座，同类问题下次还要重新花一遍沟通成本。

### 为什么系统宁可这样设计
渐进式 Spec 被单独提出，是因为工程契约的正确重量必须随复杂度、爆炸半径和失败成本上升，而不能所有任务一刀切。这样换来的是简单任务不被重流程拖慢，复杂任务又能在需要时得到更强的范围和验收保护；代价则是团队要维护清晰的升级阈值，并在任务结束后把稳定经验回写到底座。

## 真正决定渐进式 Spec结果的，是哪条机制链
### 机制链
1. 先把所有任务都必须遵守的底线规则和常驻知识固定下来，例如安全边界、仓库约束、命名规范和关键业务事实；这一层不因任务简单就消失。
2. 再根据变更复杂度、影响面和失败成本判断是否需要升级到更重契约：从局部小改，到跨模块变更，再到高风险系统级调整，流程重量逐层上升。
3. 复杂度上升后，再逐步引入 Spec、任务拆解、独立评估、执行记录与归档，而不是一开始就把所有附加流程压给所有任务。
4. 当任务结束后，把新学到的稳定知识反向同步到 rules、skills 或知识库中，让下一次类似任务从更好的起点开始，而不是永远重复临时对齐。

### 关键条件 / 分支 / 例外
- 基础 rules 始终存在，不会因为任务轻就消失；渐进的是额外契约与门禁的重量。
- Spec / review / archive 的重量应该按风险逐级抬升，而不是默认全开或默认全关。
- 如果经验没有回写到 rules / skills / knowledge，渐进体系就只做了前半程，后续任务仍会重复支付同样沟通成本。

### 最低决定层 / 关键锚点
- 最低锚点是 baseline rules 与 task-specific spec 的边界：哪些是常驻底线，哪些是这次任务新增的契约。
- 其次是 escalation triggers：复杂度、影响面、失败成本达到什么阈值才升级流程。
- 最后是 post-task backflow：新稳定经验如何回写成 rules / skills / knowledge artifacts，让下一次任务从更高起点开始。

### 排查 / 应用抓手
- **排查时先问**：如果现在只靠聊天和临场发挥推进，这次变更最可能在哪个环节出错——范围、验收、依赖、评审，还是知识回写？
- **设计时先问**：当前最值得新增的一层契约是什么：更清晰的 Spec、任务拆解、独立 review，还是任务后归档？不要一上来把所有流程全开。
- **常见观察信号**：所有任务一刀切走 full Spec，或者复杂任务仍然只靠聊天推进；同类经验反复重讲而没有进入规则或知识底座，也说明渐进层次没有立住。

## 用一个最小场景看清渐进式 Spec是怎么落地的
### 最小例子
- **场景**：改一个按钮文案时，只要遵守基础规则并跑局部检查即可；而一项涉及权限判断、数据迁移和跨模块重构的变更，就必须明确范围、拆任务、做独立 review，并在结束后沉淀新规则。
- **为什么这里会想到它**：这两个任务都属于“改代码”，但失败成本和协作成本完全不同；如果对它们施加同样重量的流程，要么浪费效率，要么放大风险。
- **结果**：渐进式 Spec 让流程重量随着复杂度逐步抬升，把有限的工程注意力用在真正值得写清楚、值得多一层证明的地方。

## 把渐进式 Spec学歪后，最容易出现哪些误判
### 常见误解
- 为了严谨，所有任务都走最重流程。这样很快就会把流程成本吃掉 AI 带来的效率收益。
- 把 Spec 写成说明文而不是验收契约。真正有效的 Spec 不是更长，而是更能帮助团队判断“什么算完成”。

### 失效 / 反噬信号
- 如果团队写了很多 Spec，但仍然频繁返工、无法判断完成度，说明文档只是形式存在，没有变成真正的门禁与协作接口。
- 如果任务结束后没有任何经验回流到规则、技能或知识库中，说明流程只做了前向约束，没有完成“越做越轻”的反向同步。

### 不适用场景
- 一次性、低风险、局部影响的小改动，没有必要被完整 Spec 系统压住；此时保留底线规则和必要验证，往往更符合成本收益。

## 渐进式 Spec总要和这些概念一起看
- [[Concepts/Harness Engineering|Harness Engineering]] `(depends[中])`：Spec 只有被测试、评审和运行时门禁真正执行，才不会退化成装饰文档。
- [[Concepts/独立评估器|独立评估器]] `(implements[中])`：当任务复杂度足够高，独立评估器就是把 Spec 变成可执行验收的关键角色。
- [[Concepts/Agent Skills|Agent Skills]] `(contrasts[中])`：Skill 沉淀长期能力，Spec 约束单次变更；前者回答“这类事一直怎么做”，后者回答“这次具体要做到什么”。

## 先记住渐进式 Spec的核心判断，再用三个问题检查自己
### 记忆锚点
- **一句话记住**：复杂度升一级，契约重一级；规则常驻，但额外流程只在值得时出现。
- **看到这个信号就想到它**：任务开始跨模块、验收口径变模糊、回滚成本上升，单靠聊天已经撑不住协作。

### 自测问题
1. 为什么“一刀切地要求所有任务都写 full Spec”会伤害效率？
2. 什么时候你该从常驻规则升级到完整 Spec、任务拆解或独立 review？
3. 如果团队写了很多 Spec 仍然频繁返工，最可能缺的是哪一层契约或回写机制？
