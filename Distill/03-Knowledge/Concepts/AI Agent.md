# AI Agent

## 先把AI Agent放进脑子里
### 一眼认知骨架
- **对象**：AI Agent 是把目标转成可追踪任务状态、规划、工具动作与反馈闭环的执行系统，重点不在“会回答”，而在“能持续行动并对完成负责”。
- **目的**：它解决的是“这件事到底需要一个回答、一个脚本，还是一个能持续推进任务的执行单元”的判断问题。很多任务不是一次推理结束，而是要经历观察、行动、再观察的循环。
- **组成**：把它拆开看，真正稳定决定结果的是：先把人类目标翻译成机器可追踪的状态：目标、约束、成功条件、失败边界都要能被系统识别，而不…
- **主线**：先把人类目标翻译成机器可追踪的状态：目标、约束、成功… → 再根据当前状态决定下一步动作类型：继续思考、补信息、… → 动作返回的新观察必须写回状态或记忆，让 Agent… → 每轮循环都要围绕完成条件重新规划
- **变体**：边界、分支和失败条件会改写这条主线；后文重点看它最容易被忽略的前提、误判点和不适用场景。
- **用法**：当任务需要多步执行、外部工具、环境反馈、失败后调整策略，而且“是否完成”不能只靠一句回答判断时，就该想到 Agent。

### 展开理解
- 判断一个系统是不是 Agent，最直接的问题不是“它会不会调工具”，而是“它能不能记住自己做过什么、接下来该做什么、什么时候该停”。
- 所谓自治，不是无限放权，而是系统能在既定目标和边界内自己推进下一步；如果进度仍主要靠人类在聊天窗口里手工追踪，它就还没真正进入 Agent 范式。

## 为什么不能只把AI Agent当成一个名词定义
### 它解决的判断 / 工程问题
- 它解决的是“这件事到底需要一个回答、一个脚本，还是一个能持续推进任务的执行单元”的判断问题。很多任务不是一次推理结束，而是要经历观察、行动、再观察的循环。
- 它还把“知道怎么做”和“真的把事做完”区分开：聊天模型可以给建议，但 Agent 要为过程中的重试、分支选择、终止条件和外部结果负责。

### 如果忽略它会怎样
- 团队很容易把“会解释”误当成“会执行”：模型说得头头是道，但真正落地时要么停在建议层，要么不断重复同类动作，却没有可靠的任务状态把它拉回正轨。
- 反过来，如果所有任务都强行上 Agent，又会把简单问题过度系统化，平白引入状态管理和权限成本。

### 为什么系统宁可这样设计
系统之所以把 Agent 和普通聊天 / 固定脚本分开，不是为了概念时髦，而是因为长任务必须把**目标状态、可执行动作、外部观察和完成边界**持续留在系统里。这样换来的是任务可以根据新观察改写下一步，失败也能定位到具体哪一轮决策；代价则是必须显式维护 state、memory、tool policy 和 termination，而不能再靠一句 prompt 模糊兜底。

## 真正决定AI Agent结果的，是哪条机制链
### 机制链
1. 先把人类目标翻译成机器可追踪的状态：目标、约束、成功条件、失败边界都要能被系统识别，而不是只留在一句自然语言里。
2. 再根据当前状态决定下一步动作类型：继续思考、补信息、调用工具、请求人工确认，还是直接执行某个外部动作。
3. 动作返回的新观察必须写回状态或记忆，让 Agent 知道“刚刚发生了什么”；否则它只会像失忆的对话模型一样重复试错。
4. 每轮循环都要围绕完成条件重新规划；一旦达到成功边界、触发失败边界，或需要外部评估器复核，闭环才算真正结束。

### 关键条件 / 分支 / 例外
- 如果任务路径几乎固定、也没有环境反馈，脚本或工作流通常更合适；这时硬上 Agent 只会增加状态管理和调试成本。
- 如果工具结果没有被写回状态或记忆，Agent 会退化成“每一轮都像第一次执行”的对话体，表面在循环，实则没有真正闭环。
- 只有当成功条件、失败边界和人工接管条件清楚时，Agent 的自治才成立；否则“自主”只是把不确定性交给用户兜底。

### 最低决定层 / 关键锚点
- 真正决定行为的不是“会不会调工具”，而是任务状态结构里是否明确保存了 goal / constraint / success / failure / next-step 这些可机读对象。
- 其次是 observe → update state / memory → choose next action → execute 这条闭环是否真的存在；少任何一环，系统都会退回建议型助手。
- 再往下看，tool result、evaluator result 和 termination signal 是否能进入下一轮决策；没有这层，Agent 只会不断重试，或者提前自称完成。
- 如果把 AI Agent 误判成“带工具调用的聊天模型”，就会看丢 state / feedback / termination 这层真正决定它能否持续把事做完的执行闭环。

### 排查 / 应用抓手
- **排查时先问**：目标、已尝试动作、失败原因和待办状态到底存在什么地方？如果这些信息只活在临时聊天里，系统还不够像 Agent。
- **设计时先问**：这项任务真的需要根据实时观察动态调整下一步吗？如果路径几乎固定，可维护脚本或工作流往往更合适。
- **常见观察信号**：反复问同一类背景、重复同样的查询、输出大量建议却不真正执行，通常说明它缺的不是更强模型，而是状态与闭环。

## 用一个最小场景看清AI Agent是怎么落地的
### 最小例子
- **场景**：目标是“修复一个 failing test”，系统需要读取失败日志、定位代码、修改实现、重新运行测试，再根据新结果决定下一步。
- **为什么这里会想到它**：这不是一次回答能解决的题，而是一个会随着测试输出不断变化的执行循环；每轮动作都依赖上一轮观察。
- **结果**：聊天模型可能停在“你可以先看看这些文件”，而 Agent 会持续推进到“测试通过、确认无新回归”，或者在触发失败边界时明确升级。

### 对比
| 易混概念 / 做法 | 真正差异 | 这里为什么不是它 |
| --- | --- | --- |
| 聊天式 LLM | 擅长解释、建议和局部分析 | 这里要求系统自己持续行动，并对“是否完成”负责，不能停在建议层 |
| 固定脚本 / 工作流 | 路径预先写死、执行确定 | 这里每一步都要根据测试与代码反馈重规划，分支很难在一开始完全写死 |

## 把AI Agent学歪后，最容易出现哪些误判
### 常见误解
- 只要能调用一个工具，就可以叫 Agent。真正关键的不是“有没有工具按钮”，而是有没有持续的任务状态和动作闭环。
- 把“自主”理解成“不用人管”。实际上，好的 Agent 更像带边界的执行者：它会自己推进，但仍受成功条件、权限和评估门禁约束。

### 失效 / 反噬信号
- 如果系统经常忘记目标、重复上一次失败的动作、或者必须靠用户不断提醒“你刚刚已经试过这个了”，说明状态管理没有真正成立。
- 如果系统表面上有规划模块，但完成判断仍全靠主观自述，没有外部证据或明确终止条件，闭环依然是假的。

### 不适用场景
- 一次性、确定性极强、路径几乎固定的小任务，往往直接用脚本或单次 LLM 调用更省；强行上 Agent 只会增加状态、权限和调试成本。

## 先记住AI Agent的核心判断，再用三个问题检查自己
### 记忆锚点
- **一句话记住**：Agent = 有状态地对外行动，并对完成条件负责；不是“会调用工具的聊天框”。
- **看到这个信号就想到它**：任务需要反复观察环境、决定下一步、重试直到完成，而不是一次回答结束。

### 自测问题
1. 为什么“会调用工具”还不足以证明一个系统是成熟 Agent？
2. 在“目标、状态、动作、反馈、终止条件”这五项里，哪一项最容易被团队偷懒省掉，后果是什么？
3. 说出一个脚本比 Agent 更合适的场景，并说明原因。
