# 独立评估器

## 1. 它是什么 (What It Is)
### 快速判断 / Quick Scan
| 维度 | 内容 |
| --- | --- |
| 一句话定义 | 独立评估器是把生成与验收拆给不同角色或不同上下文的实践，要求评估器通过执行式验证、边界探索和规则对照来发现实现者看不到的问题。 |
| 不要和什么混为一谈 | 它不是“再让另一个模型读一遍总结”，也不是作者自己换个口气做自评；真正的独立性至少包括角色分离、上下文分离和证据分离。 |
| 什么时候想到它 | 当任务影响面大、失败代价高、主观判断空间大，或者实现者不具备可靠自证能力时，就该引入独立评估器。 |

### 展开理解
- 独立评估器的职责不是“发表第二个意见”，而是把“看起来完成”改造成“有证据地完成 / 未完成”。它更像验收者，而不是礼貌性旁听者。
- 真正重要的不是一定换模型，而是换视角、换上下文、换证据输入；如果评估器只读作者自己的总结，偏差仍然会跟着一起被复制过去。

## 2. 为什么它重要 (Why It Matters)
### 它解决的判断 / 工程问题
- 它解决的是“谁来证明这件事真的做对了”的问题。实现者天然容易高估自己的完成度，尤其在 UI、全栈流程、安全逻辑和长链路任务上，自证能力往往不足。
- 它还把验收从主观自信变成可反驳的证据过程：不是“我觉得差不多了”，而是“我运行了这些检查、观察到这些结果，因此可以判断通过或失败”。

### 如果忽略它会怎样
- 很多任务会停留在“实现者说做完了”的阶段，隐藏的问题要等到用户、线上环境或下游流程替你发现，返工成本会显著更高。
- 更隐蔽的风险是评审流程形式化：看起来多了一道 review，实际上只是另一段漂亮话，没有任何新的证据进入系统。

## 3. 它是怎么工作的 (How It Works)
### 机制链 / Mechanism Chain
1. 先把验收标准、边界条件和证据来源讲清楚：要看什么输入输出、什么日志、什么实际行为，才能判断通过。
2. 再让评估器在与实现者分离的角色或上下文中读取 diff、产物、日志、运行结果或可交互系统，而不是只读作者总结。
3. 评估器通过测试、真实点击、接口调用、数据库检查、权限探测或规则比对来找问题，把“怀疑”变成“可复现证据”。
4. 如果发现问题，它的职责不是止于评论，而是把缺陷和复现步骤送回修复环；如果判定通过，也应该给出足够支撑结论的检查证据。

### 排查 / 应用抓手 (Diagnostic / Application Hooks)
- **排查时先问**：这项变更最需要什么证据才能被证伪——UI 行为、API 结果、数据库状态、权限边界，还是性能指标？回答这个问题才能设计合格评估器。
- **设计时先问**：评估器必须运行哪些真实操作，而不能只读代码或截图？很多产品级缺陷只有在实际交互里才会暴露。
- **常见观察信号**：评估器只会复述实现者说法、不给复现步骤、没有任何运行证据，或者和实现者共享同一上下文与偏见，这些都说明“独立”只是形式上的。

## 4. 一个最小例子 / 对比 (Minimal Example / Contrast)
### 最小例子
- **场景**：开发 Agent 说“登录功能已完成”。独立评估器不会只看说明，而会实际尝试错误密码、会话过期、权限绕过和数据库审计记录等关键路径。
- **为什么这里会想到它**：登录功能的正确性不只体现在代码 diff 上，还体现在 UI、接口、会话状态与权限边界的一致性上；这些都是实现者最容易遗漏的地方。
- **结果**：真正有价值的评估输出，不是“看起来不错”，而是“在这些输入下观察到这些结果，因此这里通过 / 这里失败且可复现”。

### 对比
| 易混概念 / 做法 | 真正差异 | 这里为什么不是它 |
| --- | --- | --- |
| 自我反思 / 自评 | 仍然是实现者视角，容易带着原有假设看问题 | 当前需要的是角色和证据分离，而不是作者再多说一遍自己的理解 |
| 机械测试 / CI | 擅长验证已编码好的断言 | 这里还需要补充探索式验收、真实交互和边界探测，不能完全指望现有测试覆盖 |

## 5. 常见误解与边界 (Mistakes & Boundaries)
### 常见误解
- 同一个模型先写再批自己，也算独立评估。没有角色分离和证据分离，这种“独立”通常只是形式化。
- 认为评估器的任务只是礼貌指出几点意见。真正的评估器要么给通过证据，要么给失败证据，而不是给模糊印象分。

### 失效 / 反噬信号
- 如果评估器只会重复实现者的总结、不提供复现步骤和证据，它很快就会沦为另一层漂亮话生成器。
- 如果评估器从不接触真实运行结果，只评论代码风格或概念正确性，它对高风险流程的保护作用会非常有限。

### 不适用场景
- 极小、可快速人工验证、失败成本低的任务，没有必要总是上重型独立评估；此时局部检查和轻量 review 往往更划算。

## 8. 自测问题 (Self-Test)
### 记忆锚点 / Memory Anchor
- **一句话记住**：独立不是换个嘴说同样的话，而是换角色、换上下文、换证据来判断完成度。
- **看到这个信号就想到它**：实现者说“已经做完”，但这件事光看代码或口头总结还远远不足以证明它真的做对了。

### 自测问题
1. 为什么“另一个模型读总结”通常还不算合格评估器？
2. 在什么场景下，评估器必须运行真实操作，而不能只看代码或截图？
3. 如果评估器总给不出可复现证据，它最大的缺陷是什么？
