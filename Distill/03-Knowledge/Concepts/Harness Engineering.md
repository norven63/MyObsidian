# Harness Engineering

## 先把Harness Engineering放进脑子里
### 一眼认知骨架
- **对象**：Harness Engineering 是把约束、验证、状态恢复、任务调度和审计放到模型外层运行时的工程实践，目的是让非确定性的模型在确定性的系统里稳定交付。
- **目的**：它解决的是“哪些保证必须由运行时负责，而不能指望模型自觉记住”的问题。架构边界、权限控制、验证顺序、失败恢复，都是典型的系统责任，而不是语言模型的天然强项。
- **组成**：把它拆开看，真正稳定决定结果的是：先把仓库规则、上下文装配、工具权限和执行边界外化成运行时约束，而不是散落在聊天、文档和个…
- **主线**：先把仓库规则、上下文装配、工具权限和执行边界外化成运… → 再把 build、lint、test、规则检查、评估… → 当任务被中断或失败时，由 Harness 记录检查点… → 对于高噪声或高风险步骤，再通过子代理、独立评估器和权…
- **变体**：边界、分支和失败条件会改写这条主线；后文重点看它最容易被忽略的前提、误判点和不适用场景。
- **用法**：当任务开始跨文件、跨工具、跨多轮验证，且失败成本明显上升、重跑代价变高时，就该从“让模型更努力”切换到“让系统更能兜底”。

### 展开理解
- Harness 更像 Agent 的外骨骼：它把仓库规则、工具权限、执行脚本、验证入口、日志与检查点做成工作环境，让模型不必每次靠记忆硬扛所有约束。
- 它的核心价值不是“让模型更聪明”，而是把错误更早暴露、把重试更局部化、把长任务从“全靠运气”变成“可以治理的流程”。

## 为什么不能只把Harness Engineering当成一个名词定义
### 它解决的判断 / 工程问题
- 它解决的是“哪些保证必须由运行时负责，而不能指望模型自觉记住”的问题。架构边界、权限控制、验证顺序、失败恢复，都是典型的系统责任，而不是语言模型的天然强项。
- 它还把“做完再测”改造成“执行过程中持续校验”：很多返工不是因为模型不会写代码，而是因为错误直到很后面才被发现。

### 如果忽略它会怎样
- 你会得到一个看似能跑、实际很脆的系统：长任务靠聊天历史勉强维持，一旦中断就从零开始；规则越写越多，但错误并没有更早暴露。
- 随着任务变复杂，团队会越来越依赖人工盯过程、人工解释失败、人工决定是否完成，AI 的效率优势也会被这些补位成本吃掉。

### 为什么系统宁可这样设计
Harness 要独立存在，是因为权限、验证、恢复、日志和检查点都属于运行时责任，而不是模型“应该自己记住”的责任。这样换来的是错误能在执行过程中暴露、任务能在局部失败后恢复，系统也能留下可审计证据；代价则是必须建设 sandbox、validator、checkpoint 和 run loop，而不是只写一条更长的 prompt。

## 真正决定Harness Engineering结果的，是哪条机制链
### 机制链
1. 先把仓库规则、上下文装配、工具权限和执行边界外化成运行时约束，而不是散落在聊天、文档和个人记忆里。
2. 再把 build、lint、test、规则检查、评估器等可执行验证接入主流程，让系统能在修改前、中、后持续发现偏差。
3. 当任务被中断或失败时，由 Harness 记录检查点、日志和局部状态，使恢复、重试和回滚不必依赖“重新让模型理解一遍全部背景”。
4. 对于高噪声或高风险步骤，再通过子代理、独立评估器和权限分层把执行与验收拆开，让最终结论基于证据而不是基于自述。

### 关键条件 / 分支 / 例外
- 只有把 build / lint / test / evaluator 等检查接进 loop，Harness 才算成立；只有 orchestration 而没有验证并不是真正的 Harness。
- 对低风险短任务可以采用轻量 Harness，没必要一开始就把所有 runtime 部件全部平台化。
- 如果中断后状态无法恢复，或失败只能整段重跑，说明 Harness 缺的不是文案，而是 checkpoint / resume 机制。

### 最低决定层 / 关键锚点
- 关键锚点是 run loop 里的 plan → act → check → repair，以及每一步对应的可执行验证入口。
- 其次是 sandbox / permission boundary：什么能跑、能读、能写，谁来兜底，这些才是“护栏”真正落地的地方。
- 再往下一层是 checkpoint / log / evaluator integration；这些决定失败后是局部修复，还是整段任务只能从零开始。

### 排查 / 应用抓手
- **排查时先问**：哪些保证必须由系统强制执行，而不是靠模型记住？如果答不出来，Harness 边界通常还没立住。
- **设计时先问**：什么检查可以前置、什么状态必须持久化、什么步骤应该允许局部重试？这三个问题决定了 Harness 的护栏质量。
- **常见观察信号**：规则文档越写越长，但错误没有更早暴露；任务一中断就得从零重跑；调试时只能回看整段聊天，而没有日志、检查点或可执行证据。

## 用一个最小场景看清Harness Engineering是怎么落地的
### 最小例子
- **场景**：一项跨多个包的缓存改造，需要修改接口、补测试、确认架构依赖不被破坏，还要在失败时知道该回退哪一步。
- **为什么这里会想到它**：如果只让模型“自己小心点”，它很容易先改出一大串 diff，最后才发现测试或架构规则不通过；这类错误越晚发现，返工越贵。
- **结果**：有 Harness 时，系统会先给出仓库规则和验证入口，在执行中持续跑检查、记录失败输出，必要时把复核交给评估器；任务因此从“猜着改”变成“带证据推进”。

### 对比
| 易混概念 / 做法 | 真正差异 | 这里为什么不是它 |
| --- | --- | --- |
| Prompt / Context Engineering | 主要优化提示表达与上下文呈现 | 它能提升理解质量，但不直接提供可执行验证、恢复与审计能力 |
| [[Concepts/渐进式 Spec]] | 更像任务契约，说明这次要做什么、做到什么程度 | 当前难点是“怎样把目标变成可稳定执行的流程”，不是单纯把目标写清楚 |

## 把Harness Engineering学歪后，最容易出现哪些误判
### 常见误解
- 只要加了 orchestrator 就叫 Harness。若没有验证、恢复、审计、权限与反馈环，它仍然只是更复杂的调用脚本。
- 认为 Harness 只在超大项目才有价值。实际上，只要失败成本开始上升，哪怕是轻量 Harness，也往往比继续裸奔更划算。

### 失效 / 反噬信号
- 如果规则越来越多，但错误没有更早暴露、任务中断仍要从零开始，说明 Harness 只增加了流程重量，没有增加护栏质量。
- 如果协调者仍亲自处理所有高噪声执行，或者每次都得人工解释失败原因，说明运行时并没有真正接住上下文成本和恢复成本。

### 不适用场景
- 对低风险、低复杂度、一次性任务，完整 Harness 往往过重；此时一组明确规则加局部验证，通常比建设整套运行时更符合成本收益。

### 记忆锚点
- **一句话记住**：Harness 是把模型放进护栏、验证与恢复壳里的运行时工程，而不是“更长的 prompt”。
- **看到这个信号就想到它**：任务跨文件、跨工具、跨多轮验证，一旦失败就很贵，而且重跑不能再靠从零解释背景。
