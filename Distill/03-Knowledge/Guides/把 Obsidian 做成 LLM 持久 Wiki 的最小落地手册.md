# 把 Obsidian 做成 LLM 持久 Wiki 的最小落地手册

## 什么时候该用这份手册
### 0.1 适用对象
- 已经在 Obsidian 或 Markdown 仓库里持续积累资料，开始反复问同类综合问题的人。
- 想让 LLM 帮忙维护知识库，但不想把系统做成“每次问答都重新发现知识”的团队或个人。

### 0.2 触发信号
- 你已经发现很多好回答会在聊天结束后消失，下次还得重新问一遍。
- 新资料进来时，你知道它应该改变已有认知，但目前系统只能多存一篇摘要，改不了既有知识页。

### 0.3 非目标 / 不适用
- 不用于强实时业务真值系统；数据库、报表和 API 仍应承担实时事实源角色。
- 不用于资料量很小、问题低频、维护意愿也不高的场景；这类场景继续用简单检索通常更轻。

### 0.4 前置准备
- 关联 Topic：(depends) [[Topics/LLM 持久 Wiki 与知识库编译范式]]
- 先读 Source：(depends) [[Sources/2026-04-21-LLM-Wiki-持久知识库编译范式]]
- 明确三类目录：原始资料区、LLM 维护的 published wiki 区、schema / meta / runtime 区。
- 先选一类资料做试点，不要一上来就全量导入所有历史笔记。

## 先判断你需不需要这套模式
### 1.1 决策矩阵 / Decision Matrix
| 场景 / 症状 / 上下文 | 先看什么 | 选择路径 / 动作 | 为什么这么选 |
|---------------------|----------|-----------------|--------------|
| 资料量小、问题低频、回答很少复用 | 过去一个月是否真的反复问同类问题 | **继续用简单检索 / 文件上传** | 还没到需要维护中间知识层的规模 |
| 资料持续增加，同类综合问题反复出现 | 下次问同类问题时是否总得重新读多篇原料 | **建立持久 wiki**：先分层，再跑最小 ingest | 这里真正缺的是“让知识积累”，不是“再找得快一点” |
| 团队连目录职责、页面类型和命名都没定 | 现在能否清楚区分 raw、published、meta | **先写 schema，不急着自动化** | 没有规则骨架，LLM 只会变成散装摘要器 |
| 主要痛点是实时指标、结构化查询或强一致口径 | 回答是否高度依赖最新数据库 / API | **把 wiki 当解释层，不当事实源** | compiled wiki 不适合伪装成实时真值系统 |

### 1.2 默认路径
- **默认路径 / 首选**：先用一类资料跑通 `raw source -> published wiki -> index/log -> lint` 的最小闭环，再决定是否补搜索增强和更多自动化。
- **什么时候不要走默认路径**：如果你目前还说不清哪些文件属于原料、哪些属于 LLM 输出，就先停在分层设计，别急着 ingest。

## 按阶段把最小闭环搭起来
### 2.1 先把三层资产硬拆开
- **进入信号 / Entry Cue**：当前仓库里原始资料、总结笔记、过程日志和 LLM 输出混在一起。
- **动作 / Action**：
  1. 单独划出 raw sources 目录，要求原始材料只读不改。
  2. 单独划出 published wiki 目录，要求 LLM 维护的 Source / Topic / Guide 只在这里产生。
- **完成证据 / Exit Evidence**：你能对任意一个文件回答“它是原料、编译结果，还是维护规则”，而不是三者混杂。

### 2.2 再建立最小 ingest / query / lint 回路
- **进入信号 / Entry Cue**：目录边界已经清楚，但知识还没有真正开始积累。
- **动作 / Action**：
  1. 只选一篇 source 试点 ingest，要求它不止产出摘要，还要更新 index、log，并补至少一个既有 Topic / Guide 的知识挂点。
  2. 规定 query 的回写条件：凡是形成稳定比较、分析或操作顺序的回答，都要沉淀成页面，而不是只停留在聊天里。
- **完成证据 / Exit Evidence**：同一份原料能同时改变多个知识资产，而且后续 query 已经能优先复用这些资产。

### 2.3 最后再补搜索增强与健康巡检
- **进入信号 / Entry Cue**：wiki 页数开始增长，单靠目录浏览或 index 已经不够快，或者你开始发现页面陈旧和断链问题。
- **动作 / Action**：
  1. 加入本地搜索增强，例如 qmd 这类 CLI/MCP 搜索器，但只把它当加速器，不把它当 wiki 的替代品。
  2. 建立固定 lint / review 节奏，优先检查陈旧结论、孤立页、冲突和还没进入下游的 source。
- **完成证据 / Exit Evidence**：你既能更快找到页面，也能更早发现 wiki 正在坏掉的地方，而不是等到完全失控才回头清理。

### 2.4 升级条件 / 何时求助
- 如果一篇 source ingest 完成后，系统仍说不清“它改动了哪些既有知识”，先暂停自动化，回头重写 schema。
- 如果 lint 开始稳定报出陈旧结论、孤立页面和冲突，但没人接着修，说明你缺的不是工具，而是维护节奏与责任人。

## 每次迭代前后的检查清单
### 3.1 执行前 / Preflight
1. 先确认 raw source 不会被 LLM 直接改写。 —— 证据 / 完成标志：原始资料与 published wiki 分目录、分职责。
2. 先写清一轮 ingest 至少要更新哪些资产。 —— 证据 / 完成标志：本轮不只是“生成一页”，而是明确会碰 index、log 或既有主题页。

### 3.2 执行中 / Execution
1. 不要一次导入全部历史资料，只拿一类 source 先试点。 —— 证据 / 完成标志：试点范围足够小，能说清每次更新带来的结构变化。
2. 每次 query 都要判断结果是否值得回写。 —— 证据 / 完成标志：高价值比较、分析或手册不会留在聊天记录里蒸发。

### 3.3 执行后 / Done Check
1. 看这轮知识是否真的被编译进图谱，而不是多了一份孤立摘要。 —— 证据 / 完成标志：至少一个既有 Topic / Guide 被补强或新增导航关系。
2. 跑一次健康检查，看 wiki 是否更稳而不是更乱。 —— 证据 / 完成标志：你知道本轮新增了什么，也知道本轮还留下了哪些断链、空白或陈旧点。

## 最容易把这套模式做坏的地方
### 4.1 反模式 / 不要这样做
1. **把“有搜索”误当成“有持久 wiki”。** —— 为什么错：搜索解决的是找得快，不是知识有没有被编译和持续修缮。
2. **让 raw source 和 published wiki 混在一起。** —— 为什么错：这样一来你既丢了事实底座，也丢了知识层的边界。
3. **一开始就全量自动化 ingest 全仓。** —— 为什么错：schema 还没站稳时，自动化只会放大错误结构，而不是放大收益。

### 4.2 失败信号 / 异常现象
1. **同类问题下次还得重新读多篇原料。** —— 先排查：这轮 ingest 是否只是新增摘要，没有改动既有知识页。
2. **回答很精彩，但从不回写。** —— 先排查：query-writeback 规则是不是根本不存在，或者没人执行。
3. **wiki 页数增长很快，但没人知道哪些页过时、断链或互相矛盾。** —— 先排查：lint / review 是否只是名义存在，没有形成固定回路。

### 4.3 回退 / 恢复 / 升级条件
- 如果一轮试点后你仍然无法指出“知识到底积累到了哪里”，就回退到更小的试点，不要继续扩目录。
- 如果搜索增强已经加上，但主要痛点还是陈旧与冲突，说明该补的是 review / lint，而不是继续堆搜索基础设施。

## 可以直接复用的最小骨架
### 5.1 可复用模板 / Prompt / 命令
```text
请把这份新 source 当成“要编译进 wiki 的原料”，而不是一次性回答素材：
1. 先判断它会补强哪些既有页面，而不是只生成一篇摘要。
2. 明确区分 raw source、published wiki 和 meta/schema，不要改写原始材料。
3. 至少产出：
   - 1 篇 Source
   - 对 index / log 的更新
   - 1 个被补强的 Topic / Guide / Concept
4. 如果这份材料只会多出一页孤立摘要，请先说明为什么当前 schema 不足。
5. 如果 query 过程中形成了稳定比较、分析或操作顺序，把它回写成页面，不要留在对话里。
6. 最后输出：
   - 本轮改动了哪些页面
   - 为什么这些页面被改
   - 还留下了哪些空白或冲突
```

### 5.2 产物骨架 / 输出格式
```text
原始资料：
- 路径：
- 类型：

本轮编译结果：
- 新增页面：
- 被补强页面：
- index/log 更新：

本轮回写：
- 是否有 query 结果沉淀为新页面：

健康状态：
- 新增断链 / 冲突 / 空白：
- 下一轮最值得补什么：
```

## 术语统一与继续深挖
### 6.1 术语统一
| 术语 | 在本文里的含义 | 不要混淆 |
|------|----------------|----------|
| Raw Source | 原始证据层，只读不改 | 不是 LLM 整理后的 Source 页面 |
| Persistent Wiki | 被 LLM 持续更新、互链和修缮的知识层 | 不是单纯的文档目录或上传区 |
| Schema | 告诉 LLM 如何 ingest/query/lint 的规则骨架 | 不是一次性聊天里的长 prompt |
