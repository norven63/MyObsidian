# 单 Agent、Sub-Agent 与 Managed Agents 升级选择手册

> 这份手册不回答“哪种架构听起来更先进”，只回答一件事：**你现在到底该补单 Agent 基线、引入 Sub-Agent，还是把运行时托管出去**。

## 什么时候该拿出这份手册
### 0.1 适用对象
- 正在做 AI 编码、研究代理或业务自动化，准备从“能跑 demo”升级到“能稳定交付”的工程团队。
- 已经开始讨论 planner / worker / reviewer、托管 runtime、长会话和权限治理，但还没把升级理由说清的人。

### 0.2 触发信号
- 一个主 Agent 已经长期背负大日志、多分支探索、多个工具域或长上下文，团队直觉上想“上多 Agent 试试”。
- 团队已经不再只为 prompt 发愁，而开始为 sandbox、凭证、session 恢复、trace 和运行时维护开销发愁。

### 0.3 非目标 / 不适用
- 不用于挑模型、比 benchmark、讨论谁更聪明。
- 不用于低风险一次性脚本、短链路原型或尚未建立基本 Spec / Skills / Harness 的场景。

### 0.4 前置准备
- 关联 Topic：(depends) [[Topics/AI 编码工程化与 Agent 体系选型]]
- 先读 Source：(depends) [[Sources/2026-01-23-Building-multi-agent-systems]]
- 准备最近 2-3 条失败或返工样本，至少包含任务描述、关键输出、失败信号和一次实际修复代价。
- 准备一份当前运行时负担清单：是否已有 sandbox、session 恢复、权限边界、trace、凭证管理与人工恢复流程。

## 先判断你到底卡在哪一层
### 1.1 决策矩阵 / Decision Matrix
| 场景 / 症状 / 上下文 | 先看什么 | 选择路径 / 动作 | 为什么这么选 |
|---------------------|----------|-----------------|--------------|
| 任务经常返工，但主要是范围不清、验收模糊、项目规则总忘 | 最近失败样本里第一次偏航发生在哪一步 | **先补单 Agent 基线**：Spec / Skills / Harness，不要急着加 Agent 数量 | 这不是协调复杂度问题，多 Agent 只会把低层混乱复制给更多上下文 |
| 主 Agent 长期被高噪声上下文拖垮，某些子任务输入输出清楚、可以独立验收 | 子任务是否真的能独立产出摘要、diff、报告或验证结果 | **引入 Sub-Agent**，只拆高噪声、强隔离、可局部验收的那一段 | 多 Agent 的核心收益是隔离噪声和扩宽搜索面，不是“角色看起来更完整” |
| 主要痛点已经不是生成质量，而是长会话、权限、凭证、恢复、trace、运行时维护成本 | 团队是不是在重复造 sandbox、state、permission、audit plumbing | **评估 Managed Agents / 托管 runtime** | 当基础设施工时已经吞掉交付节奏时，买掉 runtime 负担往往比继续自建更划算 |
| 同时存在高噪声执行和重运行时治理需求 | 先确认当前是否已有可观测单 Agent / Sub-Agent 样本 | **先在一个任务流里跑通 Sub-Agent，再决定是否托管 runtime** | 不先跑通任务边界，直接上托管平台容易把错边界固化到更高抽象层 |

### 1.2 默认路径
- **默认路径 / 首选**：先证明单 Agent + 基础 Harness 已经到边界，再引入 Sub-Agent；只有当运行时治理本身成为主成本时，再认真上 Managed Agents。
- **什么时候不要走默认路径**：如果你当前的问题根本不是上下文和 runtime，而是任务契约、背景知识或验证闭环没立住，就不要跳层升级。

## 按阶段把升级顺序走对
### 2.1 先证明单 Agent 已经到边界
- **进入信号 / Entry Cue**：团队正在说“复杂任务越来越难做”，但说不清难点是在理解、执行还是治理。
- **动作 / Action**：
  1. 用最近 2-3 条失败样本，标出第一次偏航发生在 Spec、Skills、Harness 还是协调层。
  2. 先补一条最小单 Agent 基线：明确任务契约、补齐关键背景、把最必要的验证和权限边界接回执行流程。
- **完成证据 / Exit Evidence**：你能明确说出“单 Agent 仍然失败的原因不是缺背景或缺验证，而是被高噪声子任务或运行时治理拖垮”。

### 2.2 只对真正高噪声的环节引入 Sub-Agent
- **进入信号 / Entry Cue**：已经确认某一段子任务会注入大量无关上下文，或者需要并行探索多个独立方向。
- **动作 / Action**：
  1. 只拆输入输出清楚、可以独立验收的一段，例如资料检索、长日志排查、验证或黑盒检查。
  2. 为每个子代理写清产物格式和 success criteria，例如“只返回 100 字摘要”“只返回验证 verdict + 失败原因”。
- **完成证据 / Exit Evidence**：主 Agent 的输入明显更短、更聚焦，且协调者不需要频繁重写子代理产物。

### 2.3 当基础设施工时已经吞掉交付时，再上 Managed Agents
- **进入信号 / Entry Cue**：团队开始反复讨论 sandbox、session 恢复、权限治理、凭证隔离、trace 或长任务断连恢复。
- **动作 / Action**：
  1. 盘点当前 agent runtime 的真实维护成本，包括恢复、值班、权限、安全和可观测性。
  2. 选一个高价值长任务做托管试点，验证长会话、trace、受控权限和恢复语义是否真的替你减负。
- **完成证据 / Exit Evidence**：试点能稳定跨断连继续、trace 可追、恢复动作清楚，且团队确实少花了自建 runtime 的工时。

### 2.4 升级条件 / 何时求助
- 如果你连“当前失败首先发生在哪一层”都说不清，先回到 (depends) [[Topics/AI 编码工程化与 Agent 体系选型]] 做分层诊断，不要直接拍板。
- 如果 Sub-Agent 试点后 token 暴涨、主协调者更忙、产物冲突更多，说明拆分边界不清，应该立刻回滚到单 Agent 基线。

## 每次升级前都要过的检查清单
### 3.1 执行前 / Preflight
1. 写清这次要优化的单一工作流，不要拿“所有复杂任务”做抽象目标。 —— 证据 / 完成标志：有一条明确任务链和近 2-3 个真实失败样本。
2. 把当前最贵的问题写成一句话，例如“主上下文被检索垃圾拖垮”或“session 恢复和权限治理已经吞掉交付节奏”。 —— 证据 / 完成标志：问题描述能指向单一主瓶颈，而不是混成一团。

### 3.2 执行中 / Execution
1. 对 Sub-Agent 试点，只允许拆一个局部高噪声环节，并规定交付物格式。 —— 证据 / 完成标志：worker 的输入、输出和 success criteria 可单独复述。
2. 对 Managed runtime 试点，只选一个需要长会话或权限治理的任务，不要全线迁移。 —— 证据 / 完成标志：有单独的试点任务、对照基线和观测指标。

### 3.3 执行后 / Done Check
1. 复盘这次升级到底减少了什么成本：上下文长度、协调返工、恢复工时还是安全 / 治理负担。 —— 证据 / 完成标志：至少能给出一项明确下降的代价或一项明确提升的稳定性。
2. 明确哪些问题没有被这次升级解决。 —— 证据 / 完成标志：复盘里保留未解决项，而不是泛化成“整体感觉更先进了”。

## 最容易做错的升级姿势
### 4.1 反模式 / 不要这样做
1. **因为任务听起来复杂，就直接拆成 planner / worker / reviewer。** —— 为什么错：这会把契约不清和验证缺失复制给更多上下文，而不是解决问题。
2. **还没盘清 runtime 成本，就先上托管平台。** —— 为什么错：如果当前主要问题其实是 Spec、Skills 或 Harness 缺位，托管平台只会让错误跑得更快。
3. **让协调者顺手下场改细节。** —— 为什么错：一旦 coordinator 重新背起所有局部噪声，Sub-Agent 的上下文隔离收益会立刻消失。

### 4.2 失败信号 / 异常现象
1. **拆了 Sub-Agent 以后，主 Agent 仍然拿到大段原始日志或检索结果。** —— 先排查：worker 是否真的被要求只返回摘要 / verdict / 结构化工件。
2. **Managed runtime 讨论会上全在聊模型和 demo，没有人聊 session、权限、恢复和 trace。** —— 先排查：你们是不是在用平台替代架构判断，而不是解决真实运行时成本。
3. **升级后 token、时延和人工协调都更高，但没人能指出换来了什么确定收益。** —— 先排查：这次升级是不是只满足了“看起来更高级”，没有满足一个明确主瓶颈。

### 4.3 回退 / 恢复 / 升级条件
- 如果 Sub-Agent 试点没有明显减轻主上下文负担，就回退到单 Agent，并重写子任务边界，而不是继续加角色。
- 如果托管试点没有明显减少恢复、权限和治理负担，就不要为了“路线正确”继续扩大接入范围。

## 可以直接复用的判断 Prompt 与输出骨架
### 5.1 可复用模板 / Prompt / 命令
```text
请把当前工作流按以下顺序做一次升级判断，不要跳步：
1. 先判断失败首先发生在 Spec、Skills、Harness 还是协调 / runtime。
2. 如果不是协调 / runtime 问题，明确指出应先补哪一层，以及为什么现在不该上多 Agent 或托管 runtime。
3. 如果是协调问题，只判断是否命中以下三类 Sub-Agent 信号：
   - 上下文污染已经可观察
   - 任务可以并行探索
   - 工具域 / prompt / 专长已经明显冲突
4. 如果是 runtime 治理问题，只判断是否命中以下 Managed Agents 信号：
   - 长会话恢复成为刚需
   - 权限 / 凭证 / trace 成为主成本
   - 团队已经在重复造 sandbox / session / governance plumbing
5. 最终只输出：
   - 推荐路径
   - 不推荐路径
   - 需要先补的前置条件
   - 最小试点方案
```

### 5.2 产物骨架 / 输出格式
```text
当前主瓶颈：
- ...

推荐路径：
- 单 Agent 基线 / Sub-Agent / Managed Agents

为什么现在选它：
- ...

为什么现在不选另外两条路：
- ...

最小试点：
- 任务：
- 指标：
- 成功标准：
- 回退条件：
```

## 术语对齐与继续深挖
### 6.1 术语统一
| 术语 | 在本文里的含义 | 不要混淆 |
|------|----------------|----------|
| 单 Agent 基线 | 一个主 Agent 加必要的 Spec / Skills / Harness，而不是“纯 prompt 聊天” | 不是“完全没有工具、没有验证的单轮助手” |
| Sub-Agent | 为高噪声或独立任务面临时引入的隔离执行单元 | 不是角色名字越多越高级 |
| Managed Agents | 把 session、sandbox、治理、trace 等运行时对象托管化交付 | 不是单纯“Agent 在云上跑” |
