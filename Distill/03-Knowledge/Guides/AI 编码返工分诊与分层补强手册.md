# AI 编码返工分诊与分层补强手册

> 这份手册不替你证明“哪家模型更强”，只负责把一个更实际的问题拆清楚：**这次返工到底是 Spec、Skills、Harness 还是运行时承载出了问题，你现在最该补哪一层**。

## 什么时候该拿出这份手册
### 0.1 适用对象
- 已经在仓库内持续使用 AI 编码，但最近开始出现重复返工、重复解释背景、改坏后才发现的人和团队。
- 需要给 AI 编码问题做分诊，而不是每次都笼统归因成“模型又抽风了”的维护者。

### 0.2 触发信号
- 同一类任务反复失败，但每次失败表象不同，团队无法判断该先补规则、补知识、补验证，还是升级运行时。
- 任务能跑出一版结果，但方向常偏、背景常忘、错误常晚暴露，导致人审成本越来越高。

### 0.3 非目标 / 不适用
- 不用于比较 benchmark、挑模型榜单、讨论推理能力上限。
- 不用于低风险一次性脚本；这类任务通常只需要轻量契约和最基本验证。

### 0.4 前置准备
- 关联 Topic：(depends) [[Topics/AI 编码工程化与 Agent 体系选型]]
- 先读 Source：(depends) [[Sources/2026-04-02-AI-编码-渐进式-Spec-实战指南]]
- 准备最近 2-3 次真实返工样本，至少包括任务描述、首次错误暴露点、最终修复动作。
- 准备一份当前 AI 编码链路的最小事实：有没有 task spec、有没有可复用 skill、有没有测试 / evaluator / 权限边界。

## 先把返工定位到正确层级
### 1.1 决策矩阵 / Decision Matrix
| 场景 / 症状 / 上下文 | 先看什么 | 选择路径 / 动作 | 为什么这么选 |
|---------------------|----------|-----------------|--------------|
| 同一任务反复在“到底要不要改、改到哪算完成”上跑偏 | 第一次返工是不是发生在写代码前 | **先补 Spec**：范围、非目标、验收条件 | 这是契约缺失，不是模型不会写，也不是运行时不够强 |
| 需求方向大体对，但总忘仓库规则、命令顺序、业务术语、现有模块边界 | 这些背景信息是不是跨很多任务都在重复讲 | **先补 Skills**：把长期背景抽成复用知识包 | 这类失败是知识供给不稳，把信息继续塞进聊天上下文只会重复丢失 |
| 代码看起来像完成了，但总是到 lint / test / review / 回归时才暴露问题 | 第一条硬错误是不是总在执行后半段才出现 | **先补 Harness**：测试、评估器、权限、恢复、日志 | 问题不在“知不知道”，而在系统没有把错尽早暴露出来 |
| Spec、Skills、Harness 都已有样子，但长任务、多角色、多分支协作仍经常失控 | 是否已经出现上下文污染、角色冲突、长会话恢复或权限治理负担 | **再讨论框架 / runtime 升级** | 只有低层已经站住，协调复杂度才配升级为主问题 |

### 1.2 默认路径
- **默认路径 / 首选**：先判断失败首次发生在写代码前、写代码中还是写代码后，再按 `Spec -> Skills -> Harness -> Runtime` 的顺序补短板。
- **什么时候不要走默认路径**：如果你已经能明确证明单 Agent 在清晰契约、稳定知识和基础护栏下仍被长状态 / 多角色协调拖垮，就不要再在低层重复加规则。

## 按阶段把补强顺序走对
### 2.1 先把返工样本收成可诊断事实
- **进入信号 / Entry Cue**：团队正在说“AI 最近很不稳定”，但没有人拿得出一条完整失败链。
- **动作 / Action**：
  1. 只挑最近 2-3 个高频失败样本，记录任务目标、第一次偏航点、第一条硬错误和最后的修复代价。
  2. 把失败按时间顺序写成一行链条：`任务输入 -> 首次错误判断 -> 第一条验证失败 -> 人工补救动作`。
- **完成证据 / Exit Evidence**：你能指出每个样本第一次出错是在“理解任务”“缺背景”“执行没挡住”还是“协作承载失控”。

### 2.2 只补当前主瓶颈那一层
- **进入信号 / Entry Cue**：样本已经足够让你判断主失败层，而不是四层都像有点问题。
- **动作 / Action**：
  1. 如果失败首先发生在任务理解，补最小 Spec；如果首先发生在长期背景缺失，补 Skill；如果首先发生在执行后期，补 Harness。
  2. 只在一条高频工作流上试点，不要同时对所有任务、所有目录、所有 agent 能力做“大治理”。
- **完成证据 / Exit Evidence**：试点任务里最常见的那类返工明显减少，而且你能说清这次减少的是“理解偏差”“背景遗忘”还是“后期漏错”。

### 2.3 只有低层站稳后，再升级运行时与协作层
- **进入信号 / Entry Cue**：Spec、Skills、Harness 已经有最低可用形态，但上下文污染、长会话恢复、权限治理或角色协调仍明显是主成本。
- **动作 / Action**：
  1. 先用 (depends) [[Guides/单 Agent、Sub-Agent 与 Managed Agents 升级选择手册]] 判断当前究竟需要 Sub-Agent 还是托管 runtime。
  2. 只为一个已经证明存在协调瓶颈的工作流升级承载层，并保留回退条件。
- **完成证据 / Exit Evidence**：升级后减少的是协调返工、上下文噪声或运行时治理成本，而不是仅仅让系统“看起来更高级”。

### 2.4 升级条件 / 何时求助
- 如果你发现四层都有问题，但说不清哪一层最先触发返工，先暂停升级，回到样本采集阶段，不要直接上更重框架。
- 如果补了某一层后，返工形态没有明显变化，说明你补错层了，应回滚并重新做首次错误定位。

## 每次分诊前都要过的检查清单
### 3.1 执行前 / Preflight
1. 先选一条高频工作流，不要拿“AI 编码整体都不稳”这种抽象抱怨做入口。 —— 证据 / 完成标志：有一条具体任务链和近 2-3 个真实失败样本。
2. 把第一次偏航点写出来，而不是只记录最后哪条测试挂了。 —— 证据 / 完成标志：每个样本都有“首次错误发生在哪一步”的一句话描述。

### 3.2 执行中 / Execution
1. 这轮只补一个主瓶颈层，不允许同时把 Spec、Skills、Harness 全部重写。 —— 证据 / 完成标志：本轮改动能清楚归类到单一主层。
2. 每次补强都要配一个可观察结果，例如减少一次重复解释、提前一次失败暴露或减少一次人审返工。 —— 证据 / 完成标志：能指出本轮试点最想压下来的单一成本。

### 3.3 执行后 / Done Check
1. 复盘返工有没有被前移发现，而不是只看最终有没有勉强交付。 —— 证据 / 完成标志：能指出第一条错误是否比以前更早暴露。
2. 记录这次仍未解决的问题，不把阶段性改善包装成“体系已经稳定”。 —— 证据 / 完成标志：复盘里保留未解决项和下一轮假设。

## 最容易做错的补强姿势
### 4.1 反模式 / 不要这样做
1. **每次返工都先怪模型，然后直接换模型或加长 prompt。** —— 为什么错：如果失败根因在契约、知识供给或验证闭环，换模型只会改变错误外观，不会消灭错误。
2. **一看到问题就同时加 Spec、RAG、Skill、测试、子代理和平台。** —— 为什么错：这样会让你失去因果判断，最后根本不知道哪一层真的产生了收益。
3. **把运行时升级当成治理起点。** —— 为什么错：低层还没站稳时上重编排，通常只是把混乱封装到更高抽象里。

### 4.2 失败信号 / 异常现象
1. **补了大量规则，但 AI 仍然经常问“这次到底要做什么”。** —— 先排查：你补的是仓库通用规则，还是当前任务真正缺失的 Spec。
2. **做了 Skill / RAG，但人类还是每次要重新解释项目背景。** —— 先排查：沉淀的是不是高频长期知识，还是一堆与当前任务无关的材料。
3. **加了测试和人审关口，但错误仍总是很晚才暴露。** —— 先排查：Harness 是否只检查表面格式，而没有把关键交互、状态或权限边界接进去。

### 4.3 回退 / 恢复 / 升级条件
- 如果一次补强后，你无法指出它到底减少了哪类返工，就回退到更小试点，而不是继续加更多层。
- 如果已经能证明低层稳定，但多角色协作和长状态仍明显失控，再升级到 (depends) [[Guides/单 Agent、Sub-Agent 与 Managed Agents 升级选择手册]] 所覆盖的协作层判断。

## 可以直接复用的分诊 Prompt 与输出骨架
### 5.1 可复用模板 / Prompt / 命令
```text
请把当前 AI 编码返工样本按下面顺序分诊，不要跳步：
1. 先指出这条失败链第一次偏航发生在哪一步，而不是最后在哪里报错。
2. 判断主问题更像 Spec、Skills、Harness 还是 Runtime/协作层。
3. 如果是 Spec 问题，只输出这次任务最缺的范围、非目标、验收条件。
4. 如果是 Skills 问题，只输出应该沉淀成长期知识包的内容，不要把一次性任务描述混进去。
5. 如果是 Harness 问题，只输出最该提前接入的验证、评估、权限或恢复动作。
6. 只有当低层已经成立时，才讨论是否要升级到 Sub-Agent 或 Managed runtime。
7. 最终只输出：
   - 首次偏航点
   - 主瓶颈层
   - 这轮最小补强动作
   - 不该现在做的升级
   - 下一次复盘要看的证据
```

### 5.2 产物骨架 / 输出格式
```text
失败样本：
- ...

首次偏航点：
- ...

主瓶颈层：
- Spec / Skills / Harness / Runtime

这轮最小补强：
- ...

为什么现在不补其他层：
- ...

复盘证据：
- 这次会更早暴露什么错误
- 这次会减少哪种重复解释或返工
```

## 术语对齐与继续深挖
### 6.1 术语统一
| 术语 | 在本文里的含义 | 不要混淆 |
|------|----------------|----------|
| 首次偏航点 | 这条失败链第一次走错方向的时刻，而不是最后报错的位置 | 不是“最终哪条测试挂了” |
| 分层补强 | 只对当前主瓶颈层做最小必要修复 | 不是一次把所有治理措施全开 |
| Runtime 升级 | 为协调复杂度、长状态和运行时治理成本做的承载层升级 | 不是任何规则补充都叫 runtime 升级 |
