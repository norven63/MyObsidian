# Android 运行时性能分诊与证据顺序手册

> 这份手册不帮你背八股，只回答一件事：**当用户说“启动慢、点了没反应、滑动掉帧、内存一直涨”时，你现在第一份证据该看什么，下一步该往哪条链路下钻**。

## 什么时候该拿出这份手册
### 0.1 适用对象
- 已经在做 Android 性能排障、架构复盘或面试准备，但总把启动、输入、渲染和内存问题混在一起的人。
- 需要把一次模糊的“卡顿 / 不流畅 / 变慢”快速拆成具体链路，并给团队一个统一排查顺序的维护者。

### 0.2 触发信号
- 用户反馈里只剩“有点卡”“点了没反应”“返回怪”“越用越重”，但团队对第一份证据看什么没有共识。
- 同一个问题在评审会上被轮流解释成主线程忙、渲染卡、系统服务慢、内存泄漏，却没人先把它落到统一时间线上。

### 0.3 非目标 / 不适用
- 不用于替代深入的 Binder 驱动、GPU driver、ART 内核级分析。
- 不用于网络、后端、产品策略类问题；那类问题不该伪装成 Android 运行时性能问题。

### 0.4 前置准备
- 关联 Topic：(depends) [[Topics/Android 性能优化与运行时机制]]
- 先读 Source：(depends) [[Sources/2026-04-10-Android系统性能优化与底层机制深度解析]]
- 先准备一条稳定复现路径，至少包含用户动作、异常现象、第一次可观察到的坏结果。
- 如果当前问题还停留在“Framework 主线在哪一层”，先回到 (depends) [[Topics/Android Framework 进程、消息、窗口与渲染主线]]，不要直接跳进细项排障。

## 先把症状归位，再决定看哪份证据
### 1.1 决策矩阵 / Decision Matrix
| 场景 / 症状 / 上下文 | 先看什么 | 选择路径 / 动作 | 为什么这么选 |
|---------------------|----------|-----------------|--------------|
| 点击图标或通知后“页面就是慢出来” | 统一启动时间线 | **先看 Perfetto / 启动 trace**，分清是 task 语义、进程启动还是首帧准备 | 启动慢如果不先切时间段，很容易把导航语义问题和时间分布问题混在一起 |
| 点击后半秒没反应、滑动不跟手 | 输入分发与 Frame Timeline | **先分流输入链还是渲染链**，不要直接归因为主线程忙 | 用户感知的“没反应”可能晚在输入进入应用之前，也可能晚在渲染提交之后 |
| 线程 blocked，但本地业务函数看起来不重 | Binder wait 与 system_server 相关轨迹 | **先查 Binder / 系统服务链** | 同步 IPC 会把等待折叠回当前线程，不先看跨进程等待就容易误判成 app 自己慢 |
| 页面退出了，内存长期不降，越用越重 | heap dump 与 retained object | **先看 Dominator Tree / Shark**，证明谁让对象活着 | 内存问题先要证明保活关系，而不是只盯总量 |
| 症状跨多层、复现不稳、每个人解释都不一样 | 统一 marker + Perfetto | **先统一时间线**，再决定下钻哪个概念 | 没有同一份时间坐标，分散日志最容易把先后因果看反 |

### 1.2 默认路径
- **默认路径 / 首选**：先按症状把问题分到启动 / IPC / 输入 / 渲染 / 内存五条链之一，再决定第一份证据。
- **什么时候不要走默认路径**：如果你已经明确拿到单一主因，比如确定是 retained object 或 task 复用错误，就不要为了“完整流程”重新把所有链都看一遍。

## 按阶段把排障顺序走对
### 2.1 先写清“用户真正抱怨的是什么”
- **进入信号 / Entry Cue**：团队嘴上都在说“卡”，但没人能复述用户是“页面晚出来”“点了没反应”“滑动掉帧”还是“越用越重”。
- **动作 / Action**：
  1. 用一句话把症状写成用户语言，不要直接上技术归因。
  2. 给这次排障只选一个主症状，其他现象先当伴随噪声记录。
- **完成证据 / Exit Evidence**：你能把问题归成启动、输入、渲染、IPC 或内存中的一类主链，而不是“全都可能”。

### 2.2 先抓第一份能分流的证据
- **进入信号 / Entry Cue**：已经知道主症状，但还没决定从 trace、task、Binder、Frame 还是 heap dump 开始。
- **动作 / Action**：
  1. 启动 / 跨层混合问题优先抓 Perfetto；输入与掉帧优先抓 Input dispatch latency 和 Frame Timeline；内存问题优先抓 heap dump。
  2. 如果线程 blocked 却看不出本地重活，先把视角移到 [[Concepts/Binder IPC]] 和相关系统服务，不要急着继续放大本地火焰图。
- **完成证据 / Exit Evidence**：你已经拿到一份能回答“时间主要耗在哪”或“对象为什么没死”的证据，而不是只有截图和主观感觉。

### 2.3 再决定下一跳该补哪块专用知识
- **进入信号 / Entry Cue**：第一份证据已经拿到，但还需要继续下钻。
- **动作 / Action**：
  1. 如果问题晚在输入进入 App 之前，就继续走 [[Concepts/InputManagerService]]；如果事件已到但反馈还没到屏幕，就走 [[Concepts/RenderThread]]。
  2. 如果问题表现成“看起来是启动慢，其实是实例 / task 语义错”，就转去 [[Concepts/Activity 启动模式]]；如果问题表现成 retained object，立刻切到 [[Concepts/Dominator Tree]]。
- **完成证据 / Exit Evidence**：你能说清下一步为什么看这个概念，而不是凭经验跳到最熟的工具或 API。

### 2.4 升级条件 / 何时求助
- **进入信号 / Entry Cue**：已经看过第一份证据，但症状仍跨多层或根因落到了 framework 更深层。
- **动作 / Action**：
  1. 如果同一条时间线同时命中 Binder、输入、渲染和内存副作用，先保留统一 Perfetto，再分别拆子问题，不要试图一句话解释全部现象。
  2. 如果问题已经进入 Binder 驱动、GPU driver、厂商内核或 ART 深层实现，就停止在本手册继续硬推，转交更深专题或专项排查。
- **完成证据 / Exit Evidence**：你能明确说明“这已经超出运行时主线排障，需要更深层入口”，而不是继续在中层概念里兜圈子。

## 每次排障前都要过的检查清单
### 3.1 执行前 / Preflight
1. 写清这次只追一个主症状，不要把“启动慢 + 点击没反应 + 内存涨”一次全吃。 —— 证据 / 完成标志：有一个明确主症状和一条稳定复现路径。
2. 决定第一份证据是什么，并说清为什么不是别的证据。 —— 证据 / 完成标志：有一个明确的 trace / task / Binder / heap dump 起点。

### 3.2 执行中 / Execution
1. 先把“时间主要耗在哪”或“对象为什么活着”证明出来，再开始写根因结论。 —— 证据 / 完成标志：已有时间线切分或支配链结论。
2. 每次只增加一个下钻维度，不要同时并行看五类工具输出。 —— 证据 / 完成标志：当前分析只围绕一条主链延伸。

### 3.3 执行后 / Done Check
1. 复盘这次问题真正属于哪条链，而不是只写“修好了”。 —— 证据 / 完成标志：结论能落在启动 / IPC / 输入 / 渲染 / 内存中的一类。
2. 记录这次第一份关键证据是什么。 —— 证据 / 完成标志：下次面对同类症状时，团队能直接复用入口顺序。

## 最容易做错的排障姿势
### 4.1 反模式 / 不要这样做
1. **一听到“卡”，就先看主线程函数耗时。** —— 为什么错：同步 Binder、输入分发和 RenderThread 都可能让用户觉得卡，但不一定体现在本地重 CPU 上。
2. **还没统一时间线，就到处打日志和截 profiler。** —— 为什么错：跨层症状最怕先后顺序被看反，分散证据会放大噪声。
3. **内存问题只盯总量，不追支配链。** —— 为什么错：你会知道“高”，却解释不了“为什么还活着”，最后只能靠猜。

### 4.2 失败信号 / 异常现象
1. **评审会上每个人都能说出一个原因，但没有人能指出第一份证据。** —— 先排查：你们是不是还没把问题先归位到链路。
2. **trace 里看见 blocked，就直接断言是 app 自己慢。** —— 先排查：有没有同步 Binder 或等待系统服务的折叠时间。
3. **已经拿到 heap dump，却还是只在看 Histogram。** —— 先排查：是不是还没有切到 Dominator Tree 或支配节点。

### 4.3 回退 / 恢复 / 升级条件
- 如果当前证据无法继续分流，就回退到统一时间线，而不是继续在错的工具输出里硬找答案。
- 如果结论必须依赖厂商驱动、内核或 ART 深层实现，及时升级给更深专题，不要把本手册当万能诊断器。

## 可以直接复用的判断 Prompt 与输出骨架
### 5.1 可复用模板 / Prompt / 命令
```text
请先不要直接猜根因，按以下顺序输出：
1. 当前主症状属于启动 / IPC / 输入 / 渲染 / 内存中的哪一类？
2. 第一份最该看的证据是什么？为什么不是别的？
3. 这份证据主要要回答“谁在等谁”还是“谁让对象活着”？
4. 下一步最该补哪个概念：Perfetto、Binder IPC、InputManagerService、RenderThread、Dominator Tree 还是 Activity 启动模式？
5. 最终只输出：
   - 主链路
   - 第一份证据
   - 下一跳概念
   - 暂不展开的噪声
```

### 5.2 产物骨架 / 输出格式
```text
当前主症状：
- ...

主链路：
- 启动 / IPC / 输入 / 渲染 / 内存

第一份证据：
- ...

为什么先看它：
- ...

下一跳概念：
- ...

暂不展开的噪声：
- ...
```

## 术语对齐与继续深挖
### 6.1 术语统一
| 术语 | 在本文里的含义 | 不要混淆 |
|------|----------------|----------|
| 第一份证据 | 当前最值得先拿来分流问题的那份 trace / dump / 状态信息 | 不是“把所有工具都开一遍” |
| 主链路 | 当前问题最先落位的运行时链：启动、IPC、输入、渲染或内存 | 不是最终根因的全部描述 |
| Dominant wait | 时间主要耗在哪一段、谁在等谁 | 不是 CPU 最忙的线程 |
| Dominant owner | 谁让对象还活着、谁支配 retained set | 不是实例数最多的类 |
