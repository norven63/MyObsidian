# Harness Engineering 与 Agent 运行时设计

## 这篇主题在图谱中的位置
- 上级主题：[[Topics/知识图谱总纲|知识图谱总纲]]
- 本专题回答：为什么 Agent 一旦进入跨文件、跨工具、跨验证的长任务，决定成败的首要变量就从 prompt 修辞切换成 runtime 设计。
- 为什么值得单独学：因为运行时不是一个“附加层”，而是把任务契约、执行循环、工具接入、证据验收和角色隔离真正焊成系统行为的地方；不把这层看清，Skills、MCP、评估器、多 Agent 都会被混成一锅。
- 默认语境 / 术语 / 版本假设：默认讨论 **2025-2026 年主流 coding agent / knowledge worker agent** 的长任务场景；“运行时”指模型外层的 session、harness、sandbox、tool adapter、evaluator、sub-agent orchestration 等执行壳；“prompt”只指自然语言指令与上下文组织，不包含测试、日志、权限、恢复；“工具连接”默认包括 MCP、CLI 包装器和受控 API；“稳定接口”指比单代模型、单个容器、单个工具 SDK 更长寿的对象边界。
- 建议先备知识：先有 [[Concepts/Harness Engineering|Harness Engineering]]、[[Concepts/Managed Agents|Managed Agents]] 的直觉，再回看 [[Concepts/渐进式 Spec|渐进式 Spec]]、[[Concepts/Agent Skills|Agent Skills]]、[[Concepts/MCP|MCP]]、[[Concepts/Sub-Agent 编排|Sub-Agent 编排]]、[[Concepts/独立评估器|独立评估器]]。
- 相邻专题：[[Topics/AI 编码工程化与 Agent 体系选型|AI 编码工程化与 Agent 体系选型]] 负责说明“为什么 AI 编码必须分层”；本专题负责说明“这些层如何在运行时落地成可恢复、可验证、可扩展的执行系统”。

## 读完后你应该会回答的问题
1. 为什么长任务的稳定性提升，通常先来自 runtime 护栏，而不是 prompt 再润色一轮。
2. 为什么 session、harness、sandbox、evaluator 这些对象，比某个模型版本或某个具体工具接法更接近长期稳定接口。
3. 执行循环、工具连接、证据评估、角色隔离分别回答什么问题，为什么它们不能互相替代。
4. 在真实系统里，先补哪几个部件最能提升交付质量，什么时候才值得升级到托管运行时。
5. 当 Agent 任务出问题时，如何按层诊断：是契约没写清、循环没收口、工具没标准化、验收没独立，还是角色边界已经混乱。

## 先把全局图装进脑子
### 2.1 一屏总图
| 层 | 它负责什么 | 失效时最典型的症状 | 第一补救动作 |
| --- | --- | --- | --- |
| **契约层** | 定义任务目标、收口条件与不该做什么 | 改着改着偏题、没人知道何时算完成 | 先补 Spec 与显式验收 |
| **执行循环层** | 让任务按 plan → act → check → repair 推进 | 一次长跑到底、错了才回头、失败后整段重来 | 先把验证和局部修复接进 loop |
| **工具 / 证据层** | 标准化外部能力连接，并把“完成”变成证据 | wrapper 各写各的、作者自述完成但一测就穿帮 | 先统一 adapter 语义，再补 evaluator |
| **角色 / 托管层** | 控制噪声、隔离职责、托管稳定对象 | 主上下文被日志和分支拖垮、换模型就大面积返工 | 先拆 reviewer / worker，再把 session / sandbox 外置 |

### 2.2 问题地图
| 运行时问题 | 典型症状 | 对应层 | 核心对象 / 代表做法 |
| --- | --- | --- | --- |
| 任务目标靠聊天记忆维持 | 改着改着偏题、无法判断是否完成 | 契约层 | [[Concepts/渐进式 Spec]]、显式验收标准 |
| 执行过程没有稳定闭环 | 一次长跑到底、错了才回头、失败后整段重来 | 执行循环层 | plan → act → check → repair 的 Harness loop |
| 工具接入全靠散装胶水 | 同一能力各写各的 wrapper，权限和错误语义漂移 | 工具连接层 | [[Concepts/MCP]]、受控 CLI / API adapter |
| “看起来做完”无法变成证据 | 作者自述完成，真实运行一测就穿帮 | 评估层 | 测试、[[Concepts/独立评估器]]、执行式验收 |
| 主上下文被日志和分支拖垮 | 一个 Agent 同时背规划、执行、调试、评审，越做越乱 | 角色隔离层 | [[Concepts/Sub-Agent 编排]]、planner / worker / reviewer |
| 模型、容器、工具版本一换就全线返工 | 任务无法跨会话恢复，平台演进代价高 | 稳定接口 / 托管层 | [[Concepts/Managed Agents]]、session / harness / sandbox |

### 2.3 主线全景
理解 Agent 运行时，最好把它看成一个**把非确定性模型包进确定性工作壳**的问题。模型负责提出候选动作，但“动作能否执行、失败如何暴露、证据如何回流、角色如何分工、状态如何跨中断保留”都不是模型自带能力，而是运行时外加的系统能力。任务一旦变长，真正致命的通常不是某一句 prompt 写得不够漂亮，而是系统没有把错误前置、没有把状态外置、没有把证据独立化。

因此，这个主题的主线不是罗列术语，而是建立一个判断顺序：**先看契约是否清楚，再看循环是否闭环，再看工具是否标准化，再看结果是否被独立证明，最后看是否需要角色分离与托管接口。** 只有按这个顺序看，才不会把不同层的概念误当成同一类“高级能力”。

### 2.4 这条线先不展开什么
- 不展开模型训练、RAG 检索算法、向量库选型等“让模型知道更多”的问题；本专题关心的是“模型知道以后如何稳定做事”。
- 不展开前端工作流编排器、通用 BPM/工作流引擎的产品比较；这里只讨论面向 Agent 长任务的运行时分层。
- 不展开某个具体厂商的 SDK 细节、协议字段或 API 版本差异；我们只抓比实现更长寿的接口与机制。
- 不把“prompt engineering 小修小补”当成核心主线；它有价值，但不是长任务可靠性的第一杠杆。

## 把主线真正讲透
### 3.1 为什么长任务里 runtime 通常比 prompt polish 更重要
- **30 秒答案**：短任务里，prompt 质量主要影响“模型是否理解你”；长任务里，runtime 质量主要决定“模型即便一度理解错了，系统能不能及时发现、局部修复、继续推进”。一旦任务跨多文件、多工具、多轮验证，后者通常更关键。
- **展开解释**：prompt 解决的是一次前向生成的起点质量，但长任务失败更常发生在执行中段：上下文漂移、工具误用、局部回归、角色混淆、状态丢失、错误发现太晚。没有 runtime 时，系统只能把这些问题继续塞回同一个上下文，期待模型下次更小心；有 runtime 时，测试、日志、检查点、权限、评估器会把失败转成可见信号，让错误在代价更低的地方暴露。也因此，“再润一次 prompt”常常只能提升首轮命中率，而不能稳定提升整条任务链的完工率。
- **关键机制 / 决策链**：
  1. 先承认模型在长链路里一定会出现偏差，而不是把成功建立在“这次别出错”上。
  2. 再把 build、lint、test、规则检查、日志和恢复点接进执行过程，让错误在执行时而不是交付后暴露。
  3. 一旦错误暴露，优先允许局部重试、局部回滚、局部复查，而不是整段上下文从零重跑。
  4. 于是系统改进的主轴就从“怎样把 prompt 写得更像专家”转成“怎样让错误更早被 runtime 抓住”。
- **诊断 / 应用抓手**：
  - 如果你的 Agent 一失败就只能“再说一遍任务背景”，说明系统还在把稳定性交给 prompt，而不是 runtime。
  - 如果规则文档越写越长、但回归缺陷还是只能到最后才发现，说明缺的不是文案，而是执行闭环。
  - 给一条关键任务链先补最小可执行验证，通常比先重写主 prompt 更快见效。

### 3.2 稳定接口为什么比单代模型或单次容器更重要
- **30 秒答案**：因为模型、工具协议、容器、SDK 都会变，但任务状态、执行壳、受控环境、验收责任这些问题不会变。把会变的实现压在 session / harness / sandbox / evaluator 等稳定接口后面，系统才有可能长期演进。
- **展开解释**：所谓 stable interface，不是指“永远不改代码”，而是指上层工作流依赖的对象边界不要跟着每次模型升级一起震荡。session 负责可恢复的任务历史；harness 负责上下文装配、循环推进、日志和检查点；sandbox 负责执行环境与权限隔离；evaluator 负责给出独立证据。只要这些对象是清楚的，上层业务就能在不同模型、不同工具连接方式甚至不同部署形态之间切换。反过来，如果任务状态、凭证、日志、执行代码都混在一个短命进程里，那么任何底层变化都会把平台稳定性拖下水。
- **关键机制 / 决策链**：
  1. 识别哪些问题会比模型版本活得更久：恢复、隔离、审计、接棒、权限。
  2. 把这些问题抽象成比实现更长寿的运行时对象，而不是附着在某次 prompt 或某个容器里。
  3. 允许模型、工具 adapter、容器编排器在接口后面独立替换，只要对象契约不破。
  4. 当平台规模上升时，再把这些接口托管化，进入 [[Concepts/Managed Agents|Managed Agents]] 范式。
- **诊断 / 应用抓手**：
  - 每次换模型就要大改平台逻辑，通常说明你把“模型特性”错当成了“运行时接口”。
  - 任务中断后若无法知道做到哪一步、为什么失败、还能否继续，优先检查 session 与 harness 是否被真正外置。
  - 凭证、执行代码和审计日志如果混在同一环境里，说明 sandbox 边界还不够成熟。

### 3.3 执行循环、工具连接、评估、角色隔离为什么必须分清
- **30 秒答案**：它们解决的是四类不同问题：执行循环回答“任务怎样持续推进并收口”，工具连接回答“外部能力怎样稳定接进来”，评估回答“谁来证明结果真的对”，角色隔离回答“高噪声工作怎样不污染主判断”。混淆它们，就会把错误修在错误的层。
- **展开解释**：很多运行时设计失败，不是因为没有“高级能力”，而是因为把不同层混成同一种能力包。例如，[[Concepts/MCP|MCP]] 让工具接入标准化，但并不自动给你验收证据；[[Concepts/Agent Skills|Agent Skills]] 让高复用知识可加载，但并不替代执行循环；[[Concepts/独立评估器|独立评估器]] 负责证明结果，而不是帮你连接数据库；[[Concepts/Sub-Agent 编排|Sub-Agent 编排]] 负责隔离上下文噪声，而不是定义任务契约。只有先把“这层到底在回答什么问题”分清，运行时升级才不会东补西补。
- **关键机制 / 决策链**：
  1. 先问失败发生在哪一层：是不会连工具、不会推进任务、不会证明结果，还是主上下文已经被噪声压垮。
  2. 再给每层一个单一职责：loop 管推进，adapter 管连接，evaluator 管证据，orchestrator 管角色边界。
  3. 当一层变重时，只升级那一层，而不是统一堆到主 prompt 或单个大 agent 身上。
  4. 最终形成“契约清楚—循环稳定—工具标准—证据独立—角色分明”的运行时骨架。
- **诊断 / 应用抓手**：
  - 如果你说“我们已经有 MCP，所以应该不需要 evaluator 了”，说明把工具层和验收层混了。
  - 如果一个主代理既写计划、又调工具、又审自己、又做最终验收，说明角色隔离还没建立。
  - 如果系统到处是脚本却没有明确 loop 状态、重试语义和收口点，说明“会调用工具”还不等于“有运行时”。

### 3.4 一个真实可落地的运行时升级顺序
- **30 秒答案**：对大多数团队，最稳的升级顺序不是先上重平台，而是**先补契约与可执行验证，再补独立评估，再做角色隔离，最后才做托管接口与 sandbox 平台化**。先证明错误能被发现，再追求错误能被规模化控制。
- **展开解释**：很多团队一看到多 Agent、托管运行时就想一步到位，但如果连“什么算完成”都没写清，或者连最基本的验证都没有，再复杂的平台只会更快复制错误。务实的顺序应该是：先用 [[Concepts/渐进式 Spec|渐进式 Spec]] 把任务契约写清；再把 build / lint / test / 规则检查接进 Harness loop；接着引入 [[Concepts/独立评估器|独立评估器]]，让完成判断不再依赖作者自述；当主上下文开始被大日志、大 diff、多分支探索拖垮时，再引入 [[Concepts/Sub-Agent 编排|Sub-Agent 编排]]；最后，只有当系统真的需要跨会话恢复、多租户隔离、稳定对象生命周期时，才上 [[Concepts/Managed Agents|Managed Agents]] 与更完整的 sandbox 编排。
- **关键机制 / 决策链**：
  1. **先补“能发现错”**：契约 + 可执行验证是所有后续升级的前提。
  2. **再补“能证明对”**：独立评估器把“作者觉得完成”变成“系统给出证据”。
  3. **再补“能控噪声”**：子代理与角色分离在复杂度真的压垮单上下文时再上。
  4. **最后补“能长期托管”**：只有状态恢复、安全隔离、平台演进成为痛点时，托管接口才值得建设。
- **诊断 / 应用抓手**：
  - 如果团队现在最常见的问题是“返工太多”，先查验证和评估，不要先查多 Agent。
  - 如果团队现在最常见的问题是“主上下文越来越脏”，再考虑 planner / worker / reviewer 分层。
  - 如果团队现在最常见的问题是“任务跨会话容易失忆、权限隔离困难”，才说明托管运行时进入优先级。

### 3.5 完整案例推演
- **场景**：一个 coding agent 被要求做跨多文件改动：先改接口契约，再改服务端逻辑，最后补测试与文档。第一轮产出方向大致正确，但测试失败、日志噪声很大，主上下文开始混入大量失败命令和半成品 diff。
- **先看什么 / 第一证据**：先看 Spec 是否写清完成标准，再看第一条失败验证记录和当前 loop 状态；不要一上来就继续往主上下文里塞更多日志。
- **推演链 / Decision Path**：
  1. 如果连“这次到底交付什么、哪些文件不能碰”都说不清，先修 **契约层**，否则后续所有验证都缺锚点。
  2. 如果任务目标清楚，但系统没有把 build / lint / test 接进 loop，先补 **执行循环层**，让错误在执行中段暴露并支持局部修复。
  3. 如果错误来自工具调用语义漂移、权限不一致或“作者自述完成但证据不足”，先修 **工具 / 证据层**，统一 adapter 并引入独立评估。
  4. 只有当验证链已经稳定，主上下文仍被长日志、多分支研究和 reviewer 职责拖垮，才升级 **角色 / 托管层**，引入 Sub-Agent 或托管对象边界。
- **结论 / 迁移**：runtime 设计的关键不是“一次把所有高级能力都装进去”，而是先按层判断：现在缺的是契约、loop、tool/evidence，还是 noise isolation。分层诊断越早，系统越不容易被“继续改改 prompt”拖住。

## 把关键概念重新串成一张图
### 4.1 为什么这组概念必须一起看
这组概念必须一起看，因为它们分别回答 runtime 的不同层，但真正的交付失败几乎总是**跨层**发生：Spec 没写清，会让 evaluator 无法验收；Skill 没沉淀好，会让主上下文反复重讲背景；MCP 没立住，会让工具调用语义漂移；Harness 不闭环，会让前面所有契约都停留在纸面；Sub-Agent 与 evaluator 不分，会让“执行”和“证明”互相污染；Managed Agents 缺位，则会让 session / sandbox 这样的长期对象继续绑在短命实现上。单看某一个概念，很容易误以为它能“一招解决大多数问题”；把它们放在同一张图里，才会明白运行时本质上是**契约、循环、接口、证据、分工、托管**六件事的组合。

### 4.2 概念分工表
| 概念 | 在这条主线里负责什么 | 如果缺它会漏掉什么 |
| --- | --- | --- |
| [[Concepts/渐进式 Spec]] | 把单次任务的范围、验收门槛和流程重量写成入口契约 | 系统不知道“什么算完成”，后续测试与评估都失去锚点 |
| [[Concepts/Agent Skills]] | 沉淀高复用知识、规则和步骤骨架，减少重复解释成本 | 主上下文会越来越胖，稳定经验无法复用 |
| [[Concepts/MCP]] | 把工具和数据源接成统一能力面，稳定输入输出与错误语义 | 每个 Agent 各写一套 wrapper，工具层无法治理 |
| [[Concepts/Harness Engineering]] | 把 loop、验证、恢复、日志、审计和权限真正执行起来 | 契约和工具存在，但任务链仍靠运气推进 |
| [[Concepts/独立评估器]] | 把“做完了”升级成“被证明做完了”的证据环 | 完成判断继续依赖作者自述，缺陷暴露太晚 |
| [[Concepts/Sub-Agent 编排]] | 隔离高噪声执行、分离 planner / worker / reviewer 职责 | 主上下文承担过多分支和日志，决策质量不断下降 |
| [[Concepts/Managed Agents]] | 把 session、harness、sandbox 等对象托管成稳定接口 | 平台一换模型或执行环境就大面积返工，长任务难以接棒 |

### 4.3 关键连接
- [[Concepts/渐进式 Spec|渐进式 Spec]] ↔ [[Concepts/Harness Engineering|Harness Engineering]]：Spec 决定门禁应该长什么样，Harness 决定门禁是否真的在执行流里生效。
- [[Concepts/Agent Skills|Agent Skills]] ↔ [[Concepts/MCP|MCP]]：Skill 解释“为什么/何时这样做”，MCP 提供“实际可以怎么连工具”，一个是知识层，一个是能力接口层。
- [[Concepts/Harness Engineering|Harness Engineering]] ↔ [[Concepts/独立评估器|独立评估器]]：Harness 负责把评估接进主流程，评估器负责给出可反驳证据；没有其中任意一个，“验证”都会退化。
- [[Concepts/Harness Engineering|Harness Engineering]] ↔ [[Concepts/Sub-Agent 编排|Sub-Agent 编排]]：Harness 给子代理提供受控执行壳，子代理帮 Harness 降低主上下文噪声与角色混淆。
- [[Concepts/Managed Agents|Managed Agents]] ↔ [[Concepts/Harness Engineering|Harness Engineering]]：Managed Agents 不是替代 Harness，而是把 Harness 的稳定对象进一步产品化、平台化。
- [[Concepts/独立评估器|独立评估器]] ↔ [[Concepts/Sub-Agent 编排|Sub-Agent 编排]]：reviewer 往往就是一种专职子代理；两者共同解决“执行者不该自证完成”的问题。

## 容易混淆的地方与边界
### 5.1 常见误解
- **误解 1：** 长任务质量不够好，先把主 prompt 再润一轮总会有效。更准确的说法是：prompt 优化提升的是起步理解，runtime 设计提升的是整条任务链的可恢复性、可验证性和可治理性。
- **误解 2：** MCP、Skills、Sub-Agent、Evaluator 都是同一种“高级 Agent 能力”，选哪个都差不多。实际上它们分属工具接口、知识沉淀、角色隔离、证据验收四个不同层级。
- **误解 3：** 只要有 CI / 测试，就不需要独立评估器。CI 擅长跑既有断言，评估器补的是探索式验收、边界探测和“作者不能自证”的那一层。
- **误解 4：** 只要做托管运行时，前面那些轻量改进都可以跳过。没有契约、验证和评估的基础，托管只会把混乱规模化。

### 5.2 适用边界
- 当任务跨文件、跨工具、跨验证步骤，且失败成本明显高于“再跑一遍”时，runtime 设计就应成为第一关注点。
- 当同一外部能力被多个 Agent / 工作流复用时，应优先思考 [[Concepts/MCP|MCP]] 这类稳定工具连接层，而不是继续散装集成。
- 当主上下文开始被大 diff、长日志、分支探索和失败历史拖垮时，应考虑 [[Concepts/Sub-Agent 编排|Sub-Agent 编排]] 与 reviewer 分层。
- 当系统需要跨会话恢复、多租户隔离、稳定审计与权限边界时，[[Concepts/Managed Agents|Managed Agents]] 才进入高优先级。

### 5.3 不适用边界
- 低风险、一次性、局部影响很小的任务，不值得一开始就建设完整运行时；轻量规则加最小验证通常更划算。
- 如果团队连最基本的验收口径都没有写清，先上多 Agent、MCP Server 或托管平台，往往只会把模糊性复制到更多组件里。
- 如果当前主要问题只是单轮理解偏差，而任务本身很短、几乎没有执行链路，那么先改 prompt、示例和上下文组织仍然可能是更便宜的解法。
- 如果外部能力几乎不复用，且只是少量局部脚本调用，过早协议化为 MCP 往往会徒增维护成本。

### 5.4 高频追问
- **应该先补测试还是先上子代理？** —— 大多数情况下先补测试 / 可执行验证。因为没有“能发现错”的闭环，再多子代理也只是在并行地产生不可验证结果。
- **独立评估器一定要换模型吗？** —— 不一定。真正关键的是角色分离、上下文分离和证据分离；同模型但不同上下文、不同输入证据也能形成有效评估。
- **MCP 能不能替代 Skill？** —— 不能。MCP 解决“怎么连能力”，Skill 解决“为什么这样做、这类事长期如何做”；一个是协议，一个是知识骨架。
- **什么时候才值得做 Managed Agents？** —— 当状态恢复、安全隔离、接口稳定性、平台演进成本已经成为系统瓶颈，而不仅仅是“偶尔有任务失败”时。
- **为什么说 reviewer 也是 runtime 的一部分？** —— 因为它决定完成判断的证据来源；没有独立验收，运行时就缺了收口机制。

## 复盘框架与自测
### 6.1 记忆框架
- **六字主线：约 → 环 → 接 → 证 → 分 → 托。**
  - **约**：先有契约，知道这次任务到底要交付什么。
  - **环**：再有执行循环，保证任务能持续推进、检查、修复。
  - **接**：再把工具连接标准化，避免能力面各写各的。
  - **证**：再让结果有证据，而不是只有作者自述。
  - **分**：噪声上来以后再做角色隔离，而不是默认多 Agent。
  - **托**：最后才把稳定对象托管化，解决跨会话恢复与平台演进。
- **一句话口诀**：先让系统知道“做什么”，再让系统知道“怎么稳地做”，最后才让系统知道“如何长期托管地做”。
- **抓手提示**：看到“返工多”先想证与环；看到“上下文脏”先想分；看到“模型一换就痛”先想托。

### 6.2 自测问题
1. 为什么说长任务的首要风险通常不是“模型不够聪明”，而是“runtime 没有把错误前置”？
2. session、harness、sandbox、evaluator 这四类对象里，哪几个最直接体现“稳定接口比具体实现更长寿”？
3. 如果一个团队已经把数据库访问都做成统一 MCP，但任务仍频繁“看起来完成、实际上没完成”，最该补哪一层？为什么？
4. 什么信号说明你该考虑 Sub-Agent 编排，而不是继续把所有动作塞进主上下文？
5. 为什么托管运行时不应该是大多数团队的第一步，而是第四步或第五步？
- **一句话复盘**：Harness Engineering 的核心，不是把 prompt 写得更像专家，而是把 Agent 放进一个能定义目标、推进任务、稳定连工具、给出证据、隔离噪声并长期演进的运行时壳里。

- 如果你总想一步到位做托管平台，先反问自己：当前链路是否已经有清晰契约、可执行验证和独立验收；没有的话，平台化大概率会过早。
