# AI 编码工程化与 Agent 体系选型

## 这篇主题在图谱中的位置
- 上级主题：[[Topics/知识图谱总纲|知识图谱总纲]]
- 本专题回答：为什么 AI 编码质量主要由分层系统设计决定，而不是由模型强弱单点决定；以及团队该按什么顺序升级能力。
- 为什么值得单独学：因为真实团队最常见的失败不是“模型太笨”，而是把任务契约、知识供给、执行护栏、框架承载四类问题混成一个问题，导致一上来就换模型或换框架。
- 默认语境 / 术语 / 版本假设：默认讨论 2025-2026 年主流仓库内 AI 编码场景，团队已经具备 Git、基础测试或人工 review 流程。文中的 **Spec** 指任务级契约与验收边界，**Skills** 指跨任务复用的知识 / 流程包（可包含 RAG、命令约定、项目规则），**Harness** 指执行、验证、权限、恢复、审计外壳，**框架选型** 指工作流 / runtime / multi-agent 底座的选择；本专题不讨论模型训练、纯聊天问答或一次性 demo 展示。
- 建议先备知识：理解 [[Concepts/AI Agent|AI Agent]] 的最小闭环，以及软件工程里“测试、CI、代码评审、回归风险”的基本含义即可。
- 相邻专题：[[Topics/Harness Engineering 与 Agent 运行时设计|Harness Engineering 与 Agent 运行时设计]] 继续回答“第三层如何真正落地到运行时”；本专题更聚焦为什么必须分层，以及团队该先补哪一层。
- 建议阅读顺序：先用本专题建立四层判断，再沿 [[Concepts/渐进式 Spec|渐进式 Spec]]、[[Concepts/Agent Skills|Agent Skills]]、[[Concepts/Harness Engineering|Harness Engineering]]、[[Concepts/Agent 框架选型|Agent 框架选型]] 逐层下钻。

## 读完后你应该会回答的问题
1. 为什么“模型更强”能提高上限，却依然不能替代 Spec、Skills、Harness 和框架选择？
2. 四层拆分分别解决哪一类复杂度，为什么它们彼此互补而不能互相替代？
3. 一个真实团队应先补哪一层，才能最快减少返工，而不是先堆多 Agent 或重平台？
4. 什么时候轻工作流 / 单 Agent 就够，什么时候框架选型才真正变成主问题？

## 先把全局图装进脑子
### 2.1 一屏总图
| 层 | 它真正决定什么 | 失效时最常见的表象 | 第一补救动作 |
| --- | --- | --- | --- |
| **Spec** | 这次任务到底要交付什么、什么算完成 | 需求一变就返工、不同人理解完全不同 | 先补范围、非目标、验收条件 |
| **Skills** | 模型做事前需要的长期背景与项目规则 | 每次都要重讲仓库结构、命令约定、业务语境 | 把重复背景抽成可复用知识 / 流程包 |
| **Harness** | 执行过程如何被验证、约束、恢复与审计 | 代码“看起来对”，但经常改坏、越权、漏回归 | 把测试、评估、权限、日志、恢复放到模型外层 |
| **框架选型** | 系统能否承载长状态、多步骤协调与角色分工 | 单任务还能跑，多步骤一上来就乱 | 先确认前三层稳定，再为协调复杂度选承载层 |

### 2.2 问题地图
| 你看到的现象 | 更可能缺的层 | 容易被误判成 | 第一补救动作 |
| --- | --- | --- | --- |
| 需求一变就重做、同一任务不同人理解完全不同 | **Spec** | 模型不稳定、prompt 不够长 | 先写清范围、非目标、验收条件 |
| 每次都要重讲仓库结构、业务规则、命令约定 | **Skills** | 上下文窗口不够、模型记忆差 | 把重复背景抽成可复用知识 / 流程包 |
| 代码“看起来对”，但经常改坏、越权、漏回归 | **Harness** | 模型编码能力差 | 把测试、评估、权限、日志、恢复放到模型外层 |
| 单任务还能跑，多步骤、多角色、长状态一上来就乱 | **框架选型** | 只要换个热门框架就能解决 | 先确认前三层已稳定，再为协调复杂度选承载层 |

### 2.3 主线全景
AI 编码质量可以粗略理解成：**模型能力 × 任务契约 × 知识供给 × 执行护栏 × 承载匹配度**。模型能力当然重要，但它只是其中一个因子。只要契约不清、背景知识供给不稳、执行缺乏验证与权限边界，再强的模型也会在长链路任务里稳定地产生高置信错误，然后被团队误判为“随机抽风”。

因此，更可靠的视角不是继续追问“哪个模型最强”，而是先判断问题卡在四层中的哪一层：任务到底有没有被写成可验收契约；跨任务重复背景有没有被稳定供给；执行过程有没有测试、评估、恢复和边界；当前任务形态到底需不需要更重的工作流 / runtime / 多 Agent。四层一旦拆开，诊断和投资顺序就会清晰很多。

### 2.4 这条线先不展开什么
- 不展开基座模型训练、微调、benchmark 细节；这里关心的是“如何把现有模型接进稳定交付系统”。
- 不展开企业级多租户平台、成本核算、组织权限体系；这里只讨论团队级 AI 编码工程化选型。
- 不把一次性脚本、低风险玩具项目强行升级成完整 Agent 平台；本专题强调的是“何时值得分层治理”。

## 把主线真正讲透
### 3.1 为什么 AI 编码质量不是“模型越强越稳”
- **30 秒答案**：强模型提高的是局部生成和推理上限；长期交付质量则由 Spec、Skills、Harness、框架匹配共同决定。缺哪一层，强模型就只是在更快地放大那一层的缺陷。
- **展开解释**：从聊天走向仓库内交付后，模型不再只负责“写一段像样代码”，还要持续理解范围、遵守项目约定、调用工具、接受验证反馈，并在失败时知道何时回退。这里真正脆弱的，不是某一轮 token 预测本身，而是整个系统有没有把“目标、知识、执行、承载”拆成可治理对象。模型更强只能让好系统更好，却不能让缺层系统自动补齐缺口。
- **关键机制 / 决策链**：
  1. 如果 **Spec** 不清，模型从第一步就在猜“这次到底要改什么、什么算完成”。
  2. 如果 **Skills** 缺位，模型即使看懂了任务，也会在项目约定、领域事实、命令流程上反复猜测。
  3. 如果 **Harness** 缺位，错误不会在执行过程中被及时抓住，而会延迟到人工 review、回归测试甚至线上事故才暴露。
  4. 只有当前三层都基本稳定后，**框架选型** 才决定系统能否承载长状态、并行分解和复杂协作。
- **诊断 / 应用抓手**：
  - 返工主要表现为“理解偏差”时，先查 Spec，而不是先换模型。
  - 返工主要表现为“总忘项目规则 / 业务事实”时，先查 Skills / RAG 的沉淀方式。
  - 返工主要表现为“改坏了但没人及时发现”时，先查 Harness 的验证与边界。
  - 只有当问题表现为“上下文爆炸、协调失控、长任务状态难维护”时，才把框架当主问题。

### 3.2 把四层拆清楚：Spec、Skills、Harness、框架选型分别回答什么
- **30 秒答案**：四层分别回答四个不同问题：**Spec** 解决“做什么”；**Skills** 解决“要知道什么”；**Harness** 解决“怎么稳做并证明做对”；**框架选型** 解决“用什么承载这些能力”。
- **展开解释**：Spec 是任务级契约，决定范围、非目标、依赖、验收条件；它针对“这一次任务”。Skills 是可复用知识与流程包，解决“跨很多任务都要重复解释的东西”；RAG、操作手册、仓库约定都更接近这一层。Harness 是模型外层执行系统，把测试、评估、权限、日志、恢复、审计接起来，让系统不靠“这次刚好答对”。框架选型则是承载层问题：当任务已经天然需要长状态、复杂工作流、子代理分工或托管运行时，你才需要认真比较不同底座，而不是拿框架替代前面三层。
- **关键机制 / 决策链**：
  1. 先判断失败发生在“生成前”还是“生成后”：生成前多半是 Spec / Skills，生成后多半是 Harness。
  2. 再判断问题是“这次任务特有”还是“跨任务反复出现”：前者写进 Spec，后者抽成 Skill。
  3. 只有当单 Agent 在清晰契约、稳定知识和基础护栏下，仍被长状态 / 多分支 / 多角色协作拖垮时，才升级框架。
  4. 投资顺序应该从低抽象、高杠杆的层开始，逐步走向高抽象、高维护成本的层。
- **诊断 / 应用抓手**：
  - 问自己：“如果换一个模型，这个问题大概率还会出现吗？”如果会，说明更像系统层问题。
  - 问自己：“这条规则只对本次任务生效，还是以后也会反复用到？”前者写 Spec，后者做 Skill。
  - 问自己：“没有测试 / 评估器时，我其实知道系统成功率吗？”如果不知道，Harness 还没建好。

### 3.3 真实团队先加什么：一个最小可行升级顺序
- **30 秒答案**：多数团队的最优先顺序是 **先 Spec，再 Skills，再 Harness，最后才框架化**。先把一条高价值任务链做稳，比先搭完整平台更划算。
- **展开解释**：务实落地不是一开始就做“企业级 AI 编码平台”，而是先挑一条高频、高返工、高价值的任务链，例如常见 bugfix、局部功能增量、配置改动或测试补齐。先把这条链路的输入、输出、非目标和验收标准写成 Spec；再把反复出现的背景、命令、规则、代码风格抽成 Skills；随后把测试、lint、评估器、权限边界、失败恢复接入 Harness。只有当这条链路已经证明“前三层确实有收益”，且出现无法用轻量编排解决的协调问题时，才值得认真上更重框架。
- **关键机制 / 决策链**：
  1. 先选一条最值得治理的任务链，而不是想“一次治理所有任务”。
  2. 用 Spec 固定“完成标准、禁止触碰范围、必须通过的检查”，先把返工从模糊问题变成显性问题。
  3. 把每周都要重复解释的仓库背景、领域术语、命令流程沉淀为 1-2 个核心 Skills，而不是一口气造一个技能森林。
  4. 把最常漏掉的验证做进 Harness：必跑命令、差异审查、评估器、恢复策略、权限边界。
  5. 如果这时仍存在长状态、多线程研究或多角色协作瓶颈，再进入框架选型与 Sub-Agent 讨论。
- **诊断 / 应用抓手**：
  - 如果团队还说不清“最常见的前三类任务是什么”，先别做平台。
  - 如果同样的项目规则一周要重复解释三次以上，就值得抽成 Skill。
  - 如果问题总是在人工 review 最后一步才被发现，就先补 Harness，而不是继续堆 prompt。
  - 如果一个任务天然包含多条独立研究线，才考虑把它拆给 Sub-Agent。

### 3.4 什么时候该升级框架，什么时候不要过度工程
- **30 秒答案**：框架升级只在“协调复杂度已经成为主瓶颈”时才成立；如果真正缺的是 Spec、Skills 或 Harness，换框架通常只是把混乱包装得更高级。
- **展开解释**：框架真正擅长的，是承载长状态、复杂工作流、多角色并行、沙箱与会话管理、人工 checkpoint、跨工具编排等问题。它不擅长自动替你澄清任务、自动抽取长期知识、自动补上验证闭环。所以一个团队如果还没有稳定任务样本、没有最低限度的技能沉淀、没有可执行检查，却想先搭多 Agent 平台，结果往往是平台代码增长得很快，交付质量却没有同步提升——因为它只是把低层问题转移到了更难调试的高层抽象里。
- **关键机制 / 决策链**：
  1. 先确认前三层不是当前主瓶颈：任务是否已写清、知识是否已复用、验证是否已外置。
  2. 再确认确实存在不可消除的协调需求：长状态、并行分解、多角色审批、长时间任务恢复、复杂工具会话管理。
  3. 选择“刚好够用”的承载方式：轻脚本 / CLI 工作流 → 轻量 orchestrator → 托管 runtime → 更重多 Agent 平台。
  4. 用可观测指标评估收益：交付周期是否缩短、回滚率是否下降、人审负担是否减少，而不是只看 demo 是否更炫。
- **诊断 / 应用抓手**：
  - 如果你说不清“这个新框架明确解决了哪个当前瓶颈”，那它大概率还不该上。
  - 优先在单一 workflow 上试点，而不是全仓库同时框架化。
  - 如果胶水代码、YAML、路由规则增长得比稳定收益还快，就是明显的过度工程信号。

### 3.5 完整案例推演
- **场景**：团队想让 AI 接手一条“新增接口 + 改前端表单 + 补测试”的常见需求。模型每次都能很快写出一版“看起来差不多”的代码，但结果时而偏题、时而忘仓库约定、时而直到人工 review 才暴露回归。
- **先看什么 / 第一证据**：先把最近 2-3 次失败任务的 Spec、实际 diff、第一条失败验证记录放在一起看，判断失败首先发生在“任务理解、背景供给、执行验证”还是“长状态协调”。
- **推演链 / Decision Path**：
  1. 如果每次返工都围绕“到底要不要改这块、什么算完成”打转，先补 **Spec**，把范围、非目标、验收写成明确契约。
  2. 如果需求理解基本一致，但模型总忘仓库命令、领域术语、已有模块边界，先补 **Skills**，把这些跨任务背景从聊天里移出来。
  3. 如果代码主体方向对了，但问题总在 lint / test / diff review / 权限越界时才暴露，先补 **Harness**，把验证、评估、恢复和边界放进执行闭环。
  4. 只有当前三层都已稳定，任务仍因为长状态、多角色、多分支协作而失控，才把 **框架选型** 升成主问题。
- **结论 / 迁移**：真实团队最容易犯的错，不是“模型不够强”，而是把四层问题混成一句“AI 不稳定”。先按四层定位，才能决定该补 contract、knowledge、runtime 还是 orchestration。

## 把关键概念重新串成一张图
### 4.1 为什么这组概念必须一起看
这组概念其实描述的是同一套控制系统里的不同角色。只学 [[Concepts/AI Agent|AI Agent]]，容易把问题抽象成“智能体够不够聪明”；只学 [[Concepts/Agent 框架选型|Agent 框架选型]]，容易把所有故障都理解成 orchestration 问题；只学 [[Concepts/Harness Engineering|Harness Engineering]]，又可能忽略验证本身需要建立在明确任务契约上。把它们放在一起看，才会发现：Agent 是执行单元，Spec 决定任务边界，Skills 稳定供给背景，Harness 把执行纳入可验证闭环，框架选型决定这些能力如何承载，Sub-Agent 只是其中一类协调手段，而不是默认答案。

### 4.2 概念分工表
| 概念 | 在这条主线里负责什么 | 如果缺它会漏掉什么 |
|------|----------------------|--------------------|
| [[Concepts/AI Agent]] | 定义最小的“感知—决策—行动—反馈”单元，提醒我们讨论的是可执行系统而非单次对话 | 容易把脚本、prompt 模板、平台、代理混成一团，讨论失焦 |
| [[Concepts/渐进式 Spec]] | 把本次任务写成契约，限定范围、依赖、非目标与验收标准 | 返工会被误报成模型随机性，团队无法分清到底是需求问题还是执行问题 |
| [[Concepts/Agent Skills]] | 沉淀跨任务复用的背景知识、命令流程、仓库约定与工作手册 | 每次会话都要重新解释上下文，效果不可复现，也无法团队共享 |
| [[Concepts/Harness Engineering]] | 把测试、评估、权限、恢复、日志与审计放在模型外层 | 产出看似完成，实际缺少验证闭环，失败无法早暴露、无法止损 |
| [[Concepts/Agent 框架选型]] | 让任务形态与承载复杂度匹配，决定何时单 Agent、何时工作流、何时托管 runtime | 容易在轻问题上上重框架，或在重问题上继续靠脆弱脚本硬撑 |
| [[Concepts/Sub-Agent 编排]] | 为独立研究线、角色分工和上下文隔离提供升级手段 | 主代理上下文被高噪声任务污染，复杂任务难以分治 |

### 4.3 关键连接
- [[Concepts/渐进式 Spec|渐进式 Spec]] ↔ [[Concepts/Agent Skills|Agent Skills]]：Spec 解决“这次任务要什么”，Skills 解决“很多任务都反复要什么”；不分边界，就会把所有信息都塞进主 prompt。
- [[Concepts/Agent Skills|Agent Skills]] ↔ [[Concepts/Harness Engineering|Harness Engineering]]：Skill 让模型更懂背景，但 Harness 才让系统能证明它做对了；“知道怎么做”不等于“被验证为做对”。
- [[Concepts/Harness Engineering|Harness Engineering]] ↔ [[Concepts/Agent 框架选型|Agent 框架选型]]：验证、权限、恢复要求越复杂，对 runtime 承载能力的要求就越高；框架不是抽象喜好，而是护栏需求的结果。
- [[Concepts/Agent 框架选型|Agent 框架选型]] ↔ [[Concepts/Sub-Agent 编排|Sub-Agent 编排]]：Sub-Agent 只是框架层处理上下文隔离与并行分解的一种回答，只有在单 Agent 真的扛不住时才值得引入。

## 容易混淆的地方与边界
### 5.1 常见误解
- **误解 1：模型更强，工程化问题自然会消失。** 现实里更常见的是：强模型把错误做得更像“看起来合理”，但 Spec、Skills、Harness 缺位的问题仍然存在。
- **误解 2：写一段超长 system prompt，就等于同时拥有 Spec 和 Skills。** Prompt 可以承载信息，但不能替代任务契约与可复用知识资产的分层管理。
- **误解 3：先搭多 Agent / 重平台，后面再慢慢补规范。** 这通常是在把低层混乱固化到更高抽象层，维护成本会上升得更快。
- **误解 4：只要有测试，就不需要 Spec 或 Skills。** 测试告诉你结果对不对，但它不替你定义“这次到底该做什么”，也不替你持续供给项目背景。

### 5.2 适用边界
- 适合仓库内长期维护、多人协作、跨会话复用、失败成本中高的 AI 编码场景。
- 适合那些已经出现重复返工、重复讲背景、验证滞后、协调失控迹象的团队。
- 适合做“先补哪一层”的治理决策，而不是只做工具或框架介绍。

### 5.3 不适用边界
- 不适合一次性 throwaway 脚本、低风险玩具项目；此时轻量规则和少量人工 review 通常更省。
- 不适合团队还没有稳定任务样本、也没有统一验收口径的阶段；此时先把人工流程跑顺，比先平台化更重要。
- 不适合把所有前沿难题都归结为工程问题；如果核心瓶颈是模型根本不会、领域知识根本不存在，先补知识源或换任务策略更现实。

### 5.4 高频追问
- **我团队只有 1-2 个人，也需要分四层吗？** —— 需要的是轻量分层，不是重平台。最少也要把任务契约和执行验证从“全靠聊天记忆”里拿出来。
- **什么时候写 Spec，什么时候做 Skill？** —— 只对本次任务生效的内容写进 Spec；跨多次任务反复出现的规则、命令、背景与流程，沉淀成 Skill。
- **RAG 属于哪一层？** —— 更接近 Skills / 知识供给层。它解决“模型需要稳定拿到什么背景”，不是执行护栏本身。
- **多 Agent 一定比单 Agent 更强吗？** —— 不一定。只有当任务天然存在上下文隔离、并行研究、角色分工需求时，多 Agent 才有净收益；否则只是增加编排成本。

## 复盘框架与自测
### 6.1 记忆框架
- **四层四问**：Spec 问“做什么算完成”；Skills 问“要知道哪些长期背景”；Harness 问“怎么稳做、怎么验、怎么止损”；框架选型问“用什么承载这些能力”。
- **症状反查口诀**：返工像“理解错”先查 Spec；返工像“背景忘”先查 Skills；返工像“漏错 / 越权”先查 Harness；返工像“协调乱 / 上下文爆”再查框架。
- **建设顺序口诀**：先固定任务契约，再沉淀可复用知识，再补执行护栏，最后才上更重编排。
- **过度工程警报**：还没有稳定任务样本，就急着做多 Agent 平台，通常是在把混乱工程化。

### 6.2 自测问题
1. 如果一个任务范围已经写得很清楚，但模型总忘仓库约定和命令顺序，你应先补哪一层，为什么？
2. 如果 AI 产出经常要到人工 review 最后一步才发现改坏了东西，你应优先补 Skills、Harness 还是框架？为什么？
3. 给定一条真实工作流，你如何判断应该继续用单 Agent + 轻工作流，还是开始考虑 Sub-Agent / 更重 runtime？
4. 说出一个“先补 Spec 比先换模型更有效”的场景，再说出一个“先补 Harness 比先换框架更有效”的场景。
5. 如果团队已经有清晰 Spec、稳定 Skills、基础 Harness，但长状态与并行协作仍明显失控，这时框架选型为什么才变成主问题？
- **一句话复盘**：AI 编码工程化不是追一个最强模型，而是把任务契约、知识供给、执行护栏和承载方式分层治理，并按收益顺序逐步升级。
