# LLM 持久 Wiki 与知识库编译范式

## 这篇主题在图谱中的位置
- 上级主题：(relates) [[Topics/知识图谱总纲|知识图谱总纲]]
- 本专题回答：为什么“让 LLM 增量维护一套持久 wiki”不是把 RAG 再包装一层，而是把知识库从检索层升级为编译层的工作方式。
- 为什么值得单独学：因为一旦知识开始被持续累积、交叉修缮和回写，问题的主轴就不再是“怎么把文档喂给模型”，而是“怎么让知识成为一个可维护、可巡检、可复用的中间工件系统”。
- 默认语境 / 术语 / 版本假设：默认讨论 2025-2026 年面向研究、团队知识库和个人长期学习的 Markdown/Obsidian 场景。文中的 **raw source** 指不可变原始材料，**persistent wiki** 指由 LLM 增量维护的结构化知识层，**schema** 指约束 ingest/query/lint 的规则文件，**compiled** 指知识先被写入稳定页面，再服务后续提问。
- 建议先备知识：先有 (depends) [[Concepts/Agent Skills|Agent Skills]] 和 (depends) [[Concepts/Harness Engineering|Harness Engineering]] 的直觉，再读这篇会更容易理解“知识库为什么也需要运行时”。
- 相邻专题：(relates) [[Topics/Harness Engineering 与 Agent 运行时设计|Harness Engineering 与 Agent 运行时设计]] 解释运行时为什么重要；本专题解释这套运行时如何落到“知识库本身的持续编译”上。

## 读完后你应该会回答的问题
1. 为什么 retrieval-only 的知识库很容易让 LLM 每次都重新发现知识，而不是让知识真正积累。
2. 为什么 persistent wiki 需要 raw sources、compiled wiki、schema/runtime 三层分工，而不是只要一个“文档目录”。
3. ingest、query、lint 三种操作如何构成知识库的复利回路，而不是三件互不相干的小功能。
4. 为什么 schema、索引、日志和独立巡检，比“再加一点搜索”更决定这套系统能不能长期成立。
5. 什么时候这套模式真正值得上，什么时候继续用简单检索或数据库查询反而更便宜。

## 先把全局图装进脑子
### 2.1 一屏总图 / One-Screen Model
| 层 | 它负责什么 | 如果缺它会怎样 | 第一补救动作 |
| --- | --- | --- | --- |
| **Raw Sources** | 保存原始证据，作为不可变事实底座 | 原料被 LLM 改写，后续无法追溯真实出处 | 先把 raw 与 published 目录分开 |
| **Compiled Wiki** | 把摘要、实体、主题、比较、分析写成持久页面并互相链接 | 每次提问都要重新从原料拼知识，无法累积 synthesis | 先要求 ingest 不止写摘要，还要更新既有页面 |
| **Schema / Runtime** | 约束 ingest、query、lint 的规则与维护顺序 | Agent 会退化成一次性摘要器，页面结构与维护节奏无法稳定 | 先写清目录、命名、索引、log 和健康检查入口 |
| **Health / Review Loop** | 检查断链、陈旧说法、冲突与知识空白 | wiki 越长越乱，最后又回到“有很多页面但没人敢信” | 先补 lint / review / query writeback 的闭环 |

### 2.2 问题地图 / Domain Map
| 知识库模式 | 默认动作 | 最大优点 | 最大代价 | 什么时候想到它 |
| --- | --- | --- | --- | --- |
| 纯 query-time retrieval | 每次提问时从原文检索片段再拼答案 | 上手快，不必维护中间层 | 结论不积累，跨文档 synthesis 每次重做 | 资料量小、问题低频、场景短平快 |
| 持久 wiki / 编译式知识库 | 新 source 到来时先改 wiki，再用 wiki 回答问题 | 知识会积累，冲突可显式记录，结果能持续复用 | 需要维护 schema、index、log 与健康检查 | 资料持续增长、问题反复出现、希望复利沉淀 |
| 实时结构化数据系统 | 查询数据库、API 或报表作为事实源 | 新鲜度高、口径稳定 | 不擅长承载长篇 synthesis 和解释性互链 | 关注强实时指标、交易状态、业务真值 |

### 2.3 主线全景
这条主线最重要的变化，是把“问答系统”改成“知识编译系统”。在 retrieval-only 模式里，LLM 每次收到问题，都要重新在原始材料里找片段、拼关系、形成临时答案；答案很可能不错，但它默认不会反过来改变知识库本身。结果就是：同类问题下次再来一次，还是得重做一遍。

持久 wiki 则把真正值钱的部分前移了。新 source 进来时，系统先不是回答问题，而是把知识写进结构化页面、补 cross-link、修订既有总结、标出冲突、更新 index 和 log。于是后续 query 面向的，优先是已经被编译过的知识层，而不是散乱原文。这样积累的不是“回答过多少次”，而是“wiki 本身已经长成什么样”。

### 2.4 关键条件 / 例外 / 易错分叉
- **只有当资料和问题会持续累积时，这套模式才值得**：如果只是偶尔问一次、资料也很少，继续用简单检索通常更省心。
- **raw source 与 compiled wiki 必须硬分层**：如果原始材料和 LLM 维护页面混在一起，系统很快就会失去可追溯性。
- **没有 lint / review 的 persistent wiki 很容易再次退化成静态摘要堆**：页面数量增加本身不等于知识积累，真正决定质量的是持续修缮能力。
- **实时事实源不能被 wiki 伪装替代**：数据库、实时指标和交易状态仍应从结构化源头取值，wiki 更适合作为解释层与 synthesis 层。

### 2.5 这条线先不展开什么
- 不展开向量检索、embedding、reranker 的底层算法细节；本专题关心的是“为什么先有编译层，再考虑搜索层”。
- 不展开具体 Obsidian 插件安装、快捷键绑定和浏览器剪藏教程；这些是实现细节，不是模式本身。
- 不把大型企业级 KM、权限组织架构和知识治理制度展开成另一条专题；这里只讲“LLM 维护的持久 wiki”最小成立条件。

## 把主线真正讲透
### 3.1 为什么持久 Wiki 不是换皮的 RAG
- **30 秒答案**：RAG 的默认动作是“提问时找资料”，持久 wiki 的默认动作是“资料来了先改知识库”。前者重在临时检索，后者重在持续编译。
- **展开解释**：两者都可能用到检索，但判断重点完全不同。RAG 关心的是单次查询时如何找到最相关片段；持久 wiki 关心的是每次新材料到来后，哪些既有页面应被修订、哪些对照应被补上、哪些冲突应被显式标注。也因此，它真正对抗的不是“检索不准”，而是“知识每次都要重新发现，没有复利”。
- **关键机制 / 决策链**：
  1. 原始材料保持不可变，只承担事实底座角色。
  2. 每次 ingest 都要求 LLM 更新现有页面，而不是只追加一篇孤立摘要。
  3. 后续 query 尽量优先读已经组织好的 wiki 页面，而不是每次回原文做全量拼装。
  4. 高价值 query 再回写成新页面，让分析本身也进入知识层。
- **关键条件 / 例外 / 易错分叉**：如果 ingest 从不更新旧页面、query 结果也从不回写，那么这套系统再像 wiki，也只是“归档了很多摘要”，还不算真正的编译式知识库。
- **诊断 / 应用抓手**：
  - 如果你发现同类综合问题每次都要重新读 5 篇 source，说明现在更像 retrieval，而不是编译。
  - 如果新材料进来以后，系统说不清“哪些既有页面该被修订”，说明还没有真正建立持久知识层。

### 3.2 为什么 schema / runtime 比“再加一点搜索”更决定长期价值
- **30 秒答案**：因为搜索只能帮你更快找到内容，schema / runtime 才决定内容是否被稳定维护、如何进入图谱、以及什么时候该被修缮。
- **展开解释**：文章把第三层单独命名为 `schema`，不是偶然。没有 schema 时，Agent 只会像一个泛化摘要器；有了 schema，Agent 才知道目录结构、页面职责、ingest 顺序、query 回写原则、lint 口径、index 和 log 的维护方式。也正因为如此，持久 wiki 的关键不是“多一个搜索框”，而是“有没有一套长期稳定的工作规则，让知识可以持续进入和持续被修理”。
- **关键机制 / 决策链**：
  1. 先把高频稳定规则沉淀成类似 Skill 的 schema，而不是继续塞在会话 prompt 里。
  2. 再把 ingest / query / lint 组织成 Harness loop，让知识库维护不是一次性动作。
  3. 引入 index、log、review 和 health-check，确保页面增加时质量不会跟着塌。
  4. 搜索引擎在规模升高后再补，服务的是“找得更快”，不是“维护得更稳”。
- **关键条件 / 例外 / 易错分叉**：schema 不能只是写一份没人看的大说明；如果它不能被运行时稳定执行，知识库依然会退化成一次性人工整理。
- **诊断 / 应用抓手**：
  - 如果每次 ingest 结果都很不一样，先看 schema 是否真的定义了页面职责和更新顺序。
  - 如果大家只在讨论搜索体验，却没人讨论 index、log 和 health-check，说明系统还停留在“找资料”而不是“养知识”阶段。

### 3.3 ingest、query、lint 三个动作为什么会形成复利
- **30 秒答案**：因为它们分别负责“把新知识写进去”“把高价值回答沉淀回来”“把系统里正在变坏的地方找出来”，三者连起来才会真正累积。
- **展开解释**：仅有 ingest，知识会进入系统，但不一定长成适合使用的结构；仅有 query，回答很可能只停留在聊天记录；仅有 lint，又只会不断报问题而没有新知识进入。复利来自这三者形成闭环：ingest 让新 source 改变 wiki，query 把高价值分析回写成页面，lint 则把断链、陈旧说法、孤立页和知识空白持续暴露出来，让下一轮 ingest / query 有方向。
- **关键机制 / 决策链**：
  1. 用 ingest 把 raw source 变成结构化页面更新。
  2. 用 query 让“问出来的好问题”反过来变成新资产，而不是只留在会话里。
  3. 用 lint / review 检查冲突、孤岛、过时判断和可继续深挖的缺口。
  4. 于是知识库每轮都不只是“多一页”，而是“结构更稳、导航更好、结论更近事实”。
- **关键条件 / 例外 / 易错分叉**：如果 query 从不回写、lint 也只做字符串层面的机械检查，系统会慢慢失去“复利”，重新退回到人肉维护与聊天蒸发。
- **诊断 / 应用抓手**：
  - 如果一份很有价值的对比回答下次还得再问一遍，说明 query-writeback 还没建立。
  - 如果 lint 只能告诉你“有几个坏链接”，却说不清哪些页面陈旧、哪些结论冲突，说明评估层还太浅。

### 3.4 完整案例推演 / Worked Case
- **场景**：你在 Obsidian 里长期积累 AI Agent、Harness 和运行时材料。起初每次问“多 Agent 什么时候值得上”都要重新翻文章、重新总结，回答不错，但总是重复劳动。
- **先看什么 / 第一证据**：先看原料是否真的会持续增加，以及旧回答是否经常被新材料推翻或补强；如果是，这就不是一次性问答问题，而是知识编译问题。
- **推演链 / Decision Path**：
  1. 先把文章、笔记、报告放进 raw sources，禁止直接改写原件。
  2. 再让 LLM ingest 单篇材料，同时更新 Source、Topic、Guide、index 和 log，而不是只写一篇摘要。
  3. 当你问“单 Agent、Sub-Agent、Managed Agents 怎么选”时，优先让系统读已经成型的 Topic / Guide，而不是重新检索全部原文。
  4. 每周再跑一次 lint / review，把孤立页、陈旧判断和未消化 source 暴露出来，决定下一批应该补什么。
- **结论 / 迁移**：这套方法真正省下来的，不是某一轮问答的 token，而是下一次遇到同类问题时，不必再从原始材料重造知识。

## 容易混淆的地方与边界
### 5.1 常见误解
- 误以为“只要有 RAG 和搜索框，就已经拥有了持久 wiki”。真正关键的是知识是否会被编译进页面，而不是每次问答都重新找片段。
- 误以为 persistent wiki 可以替代所有事实源。它更像解释层和 synthesis 层，不是实时数据库或权威真值库。
- 误以为只要页面越来越多，知识就在累积。没有 review、lint、回写和导航，页面增长也可能只是噪声增长。

### 5.2 适用边界
- 适合资料会长期增长、问题会反复复用、而且你希望知识结构本身越来越稳的场景。
- 适合需要把对比、总结、争议点和操作手册真正沉淀回仓库，而不是只停留在聊天历史里的团队或个人。

### 5.3 不适用边界
- 不适合低频、短平快、资料量很小的场景；此时 simple retrieval 往往已经够用。
- 不适合强实时口径、交易数据和控制面状态管理；这些仍应该由数据库、API 或报表承担。

### 5.4 高频追问
- **为什么不直接把 query 结果当最终答案就好？** —— 因为高价值 query 往往会重复出现，不回写就意味着下次还要重新构造。
- **这是不是要求一开始就做很重的平台？** —— 不是。最小成立版本只需要先分层、跑单篇 ingest、维护 index/log，再补 lint。

## 复盘框架与自测
### 6.1 记忆框架 / Memory Frame
- **源 -> 编 -> 用 -> 巡**：先有原料，再编译成 wiki，然后用 wiki 回答问题，最后持续巡检。
- **一句话口诀**：不是每次提问都重新发现知识，而是让知识先变成可维护的页面。

### 6.2 自测问题
1. 为什么说 persistent wiki 的默认动作是“先改知识库”，而不是“先回答问题”？
2. 在这套模式里，index、log 和 lint 分别解决的是什么问题？
3. 如果 query 结果很精彩，但从不回写，它会怎样削弱整个系统的复利？
- **一句话复盘**：LLM 持久 wiki 的核心，不是让模型更会找资料，而是让知识先被编译、再被使用、并且能持续被修缮。

### 7.3 缺口提醒
- 如果你现在还在把 raw sources 和 LLM 写出的页面混在一起，说明这套模式的第一层都还没建立。
- 如果你能说清 persistent wiki 的理念，却说不清“query 结果何时该回写”，说明复利回路还没真正成立。
- 下一步最值得补的，不是再多看一个搜索工具，而是先验证一篇 source 是否真的能改动多个既有页面，并被后续 query 复用。
